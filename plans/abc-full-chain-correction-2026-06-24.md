## 0. 范围、前提与原则

- **范围**：不是只修引擎 bug，而是**逐层矫正**「领域调研 → 方案 → 需求 → 数据/成本模型设计 → 实现 → 验证」整条链，使每一层都由上一层推导而来、可追溯、可验证。
- **前提（关键）**：产品未上线 → **数据库 schema 可自由变更，不背迁移包袱**。因此设计矫正应**一次性建对**，而不是用 `ensureColumn` 兼容旧库的补丁式叠加。
- **领域层免动**：诊断确认领域调研（`pathology-workflow-and-bom-guide.md`）质量最高、且诚实。**矫正以领域文档为蓝本反推需求与数据模型，不推翻重来。**
- **核心原则**：①有真动因的地方做 ABC，没有的地方做**诚实且披露**的吸收法，**永不静默平均分**；②每个成本数字都能解释「哪个作业按哪个动因以哪个费率消耗」；③成本必须**完全吸收对账**（Σ分摊 = Σ成本池）；④准确性用**黄金用例测试**钉死（≤5%）；⑤**采纳优先（最高否决权）**——技术正确只是必要条件，任何设计若让用户因"太麻烦/太复杂/不是他要的"而绕开，即判失败（见 §2.5）。

## 2. 矫正后的成本模型规格（B 方案）

> 这是全计划的"靶子"。L1-L5 都服务于把系统改造成符合本节的模型。

### 2.1 成本对象层级（来自领域，正式固化）

| 对象 | 角色 | 主动因 |
|---|---|---|
| 病例 case | **收费对象**（阶梯/封顶按病例聚合） | case_count |
| 蜡块 block | 标本处理/切片的成本动因 | block_count |
| **切片 slide** | **核心成本对象**（成本最小可归因单位） | slide_count |

### 2.2 成本类别与处理方式（B 方案核心）

| 类别 | 可追溯性 | 处理 | 现状 |
|---|---|---|---|
| 材料（专用试剂/抗体/耗材） | **可干净追溯** | BOM 用量 × FEFO 批次加权价，逐切片 | ✅ 已正确 |
| 质控品 QC | 半追溯 | 按"覆盖样本数"批分摊（`calculateQCCost`） | 🟡 已有，需纳入统一引擎 |
| 人工（分步） | **可追溯到作业** | 每个人工步骤映射到作业中心；中心成本=Σ(标准分钟×费率)；按中心动因分摊 | ❌ 来源无中心映射 |
| 设备折旧 | **可追溯到作业** | 设备/设备模板映射到作业中心；按机器分钟或切片数动因分摊 | ❌ 来源无中心映射 |
| 间接费（房租/水电/管理/信息系统） | **天然只能近似**（领域 §5.4 已承认） | **B 方案：单一披露基准**——全期间接费按一个公开声明的基准（推荐：各中心直接成本占比）分到各中心，UI 标注"间接分摊估算" | ❌ 当前平均分且不披露 |

### 2.3 归集机制（替换平均分摊）

**期间归集（每月一次）：**
1. 每个作业中心声明自己的 `cost_driver_type`（已有字段）。
2. 每个成本来源声明归属的作业中心（**L2 新增映射**）：人工步骤→中心、设备→中心、间接成本中心→中心（或按基准分配）。
3. 对每个中心：
   - `pool_cost = Σ(本中心人工) + Σ(本中心设备) + 本中心分得的间接（按单一披露基准）`
   - `driver_quantity = 该中心动因的当期实际总量`（如染色中心=Σslide_count，处理中心=Σblock_count；**不再是全局样本量**）
   - `driver_rate = pool_cost / driver_quantity`
4. **完全吸收校验**：Σ_中心 pool_cost 必须等于（本期人工+设备+间接总额）。差额→异常台账。

**逐切片分摊（出库时/重算时）：**
- 对 BOM 关联的每个中心：`activity_cost = driver_rate × 本BOM单位动因量 × 数量`
- `切片总成本 = 材料成本 + Σ_中心 activity_cost`
- **下游对账**：Σ_全部成本对象 activity_cost ≈ Σ_中心 pool_cost（完全吸收，测试断言）。

### 2.4 标准成本 vs 实际成本（消除"两套引擎"困惑）

当前 `calculateProjectCost`（五要素直算）与 `calculateSlideCostWithFee`（材料+池）冲突。**矫正不是强行合并，而是明确定位**（这是正规成本会计的标准做法）：

| 引擎 | 定位 | 用途 | 数据源 |
|---|---|---|---|
| 标准成本（BOM `standard_*`） | **计划值** | 定价、毛利规划、预算 | BOM 标准工时/设备/间接标准 |
| 实际成本（ABC 切片成本） | **核算值** | 月度核算、关账、利润 | 期间成本池 + 动因 |
| 差异 = 标准 − 实际 | **特性** | 差异分析、流程优化 | 二者之差 |

→ 矫正动作：①前端/接口**明确标注**某数字是"标准/计划"还是"实际/核算"，杜绝同一标签下两个数打架；②`calculateProjectCost` 改为"标准成本"专用，实际项目成本改为切片实际成本聚合；③差异分析对比二者（已有 `variance-analysis` 入口，接真实口径）。

### 2.5 采纳优先：避免"有用但没人用"（一等约束，最高否决权）

> 技术正确（§2.1-2.4 + CHAIN-01~10）是**必要条件，不是充分条件**。本产品的头号失败模式不是算错，而是用户嫌麻烦/复杂/不对路而绕开——`.claude/research/abc-market-research.md` 已明确警示深度 ABC"用户接受度低"，根因是数据采集负担、配置复杂、输出不是管理者关心的。**L0-L5 的每个任务都必须先过这一关。**

| 采纳风险（反模式） | 本产品的具体表现 | 设计强制要求（必须满足才算合格） |
|---|---|---|
| **录入负担**：要用户手填一堆数据 | 每月手录人工工时/设备时长/间接费/各种映射 | **靠系统已有数据自动推导**（出库/BOM/批次价/FEFO）；成本是**出库的副产品**，不是月末苦差；映射"**设一次复用**"，手填只作调整例外 |
| **配置复杂**：需会计专家才能跑 | 作业中心、动因、来源→中心映射、收费映射全靠用户从零配 | **内置病理领域模板 + 合理默认**（领域文档已有步骤→动因表），**开箱即用**；新建实验室一键载入模板 |
| **输出错位**：给的不是用户要的 | 首屏堆"每张切片消耗多少资源"，而管理者只想知道"赚不赚钱" | **首屏=收费 vs 成本 / 是否盈利**（收费锚点）；ABC 作业级明细放进**下钻**；**按角色渐进披露**（仓管不被迫走成本界面，财务才进核算工作台） |
| **流程摩擦**：新增强制步骤 | 出库时强制填全套 ABC 字段否则不让出库 | 成本采集**骑在既有动作上**，缺配置时允许"**先继续、后补算**"，异常只在真缺时浮现（库存动作与成本动作解耦，已是既有正确设计，须保持） |
| **信任流失**：算不清就回 Excel | 切片成本无法解释、两个页面对同一项目给两个数 | 可解释 + 可对账 + 口径标注（§2.2-2.4）**同时也是采纳要求**；间接费明确标"分摊估算"，不假装精确 |

**落地为流程要求**：L0-L5 每个任务在技术验收外，附一条**采纳自检**——"用户会不会因为太麻烦/太复杂/不是他要的而绕开它？"答不了"不会"的任务不算完成。对应 §4 的 **ADOPT-01~05 采纳验收关**。

## 4. 验收标准（全链路成立的判据）

| 编号 | 场景 | 通过标准 |
|---|---|---|
| CHAIN-01 领域→需求 | 每条 ABC 需求可回溯领域依据 | 需求文档有领域出处 |
| CHAIN-02 需求可验证 | ≤5% 在 PRD/AC/章程三处一致且有可执行用例 | AC-COST-ACCURACY 存在 |
| CHAIN-03 设计补链 | 任一成本来源可查归属作业中心 | 映射表/字段存在且非空 |

| **CHAIN-05 准确性** | 黄金用例切片成本 ≤5% 偏差 | L4-2 绿 |
| **CHAIN-06 完全吸收** | Σ分摊到切片 = Σ成本池（±误差容限） | L4-2 对账断言绿 |
| CHAIN-07 可解释 | 任一切片成本可还原逐中心动因/费率/出处 | 下钻可见；算不清即异常 |
| CHAIN-08 标准/实际不打架 | 同一项目不出现两个无标注的"成本" | UI/接口明确标注 |
| CHAIN-09 间接诚实 | 间接费用单一披露基准且 UI 标注估算 | 不再隐藏平均分 |
| CHAIN-10 关账可信 | 关账门禁含完全吸收/准确性检查，非仅状态 | close 端点增校验 |

|---|---|---|

## 5. 实施次序与里程碑

| 里程碑 | 内容 | 工期 | 出口 |
|---|---|---|---|
| **M1 把靶子立起来** | L0 + L1 + L4-1/L4-2(先红) | ~3 天 | 黄金用例红着等引擎；需求可验证 |
| **M2 补链（设计）** | L2 全部 | ~3 天 | 来源→中心映射建成，schema 一次到位 |
| **M3 重建引擎** | L3 全部 | ~5 天 | 删平均分，真实动因归集，可解释 |
| **M4 钉死准确性** | L4-2 转绿 + L4-3~L4-6 | ~3 天 | ≤5% + 完全吸收 CI 红线；拟真对比通过 |
| **M5 收口** | L5 全部 | ~3 天 | 前端真实下钻 + 口径标注；可对外称"精确" |

总计约 **15-17 个工作日**到"可信首版"。M1-M4 是核心（不可压缩），M5 可与 M4 部分并行。

## 6. 风险与待决策点

| 项 | 说明 | 建议 |
|---|---|---|
| **采纳风险（头号）** | 算得对但用户嫌麻烦/复杂/不对路而绕开 → 产品失败（市场调研已警示 ABC 接受度低） | 全程贯彻 §2.5 与 ADOPT-01~05；**自动推导优先、开箱即用、首屏给盈利、不挡道、可解释**；每个任务过采纳自检 |
| 间接费单一基准选哪个 | `by_direct_cost` / `by_driver_volume` / `by_slide_equivalent` | 推荐 `by_direct_cost`（实务最常见、可解释）；做成可配置，默认它 |
| 设备/人工→中心映射的 seed 工作量 | 需病理业务知识填映射 | 以领域文档 §5 的步骤-动因表为蓝本，admin 可后续调整 |
| 标准成本版本化与 DEC-009 张力 | DEC-009 曾定"历史版本不保留" | 本计划 L2-5 用版本快照承接核算引用，建议更新 DEC-009 |
| schema 大改影响既有测试/E2E | 未上线可接受，但需回归 | M2 后整体跑一次回归，修连带 |
| "≤5%"对间接占比高的项目可能挑战 | 间接是近似分摊 | 黄金集需覆盖高/低间接占比两类，必要时调容限或加披露 |
