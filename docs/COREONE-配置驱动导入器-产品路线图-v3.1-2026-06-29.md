# COREONE 配置驱动导入器产品路线图 v3.1

日期：2026-06-29

## 0. 本版定位

v3.1 是在 v2 合并终稿基础上的执行版。它不推翻 v2 的方向，而是把 v2 的 Epic 列表改成更容易交付和验收的纵向路线。

v3.1 继承 v2 已锁定的 4 项决策：

1. 病例身份采用全链路复合键 `(partner_id, case_no)`。
2. Phase 0 先修正确性子集：跨院串账、配置读/回滚归一、NGS 缺成本/售价质量标记。
3. 联合运营/共建按逐院 `special.joint` 配置决定是否进院级 P&L 主表。
4. 已关账月份只出调整单，未关账月份可一键重算。

v3.1 新增 5 个重点：

1. Phase 1 从“多个 Epic 并行”改成“一个最小可关账月”的纵向切片。
2. 数据层先建立统一规范行 `statement_normalized_lines`，再派生病例收入、聚合收入、OUT 台账和 P&L 快照。
3. 月结工作台以“能不能关账”为主状态，而不是导入任务列表。
4. 质量标记从告警表升级为“阻断等级 + 责任人 + 处理动作”的闭环。
5. 增加指标树，路线图按月结效率和可信度衡量，而不是只看功能交付。

一句话判断：COREONE 的目标不是做一个 Excel 导入器，而是做病理实验室的月结 sub-ledger，承接收入、外送、成本、调整、关账和审计。

## 1. 北极星和首版成功标准

北极星：财务能在一个月结周期内完成所有医院结算单导入、分类确认、异常处理、院级盈亏生成与审计留痕，不再依赖线下 Excel 当主账。

首版成功标准：

- 真实 7 类模板家族均有明确处理路径：入库、登记、转 OUT 台账、待处理或调整。
- 每月、每院、每文件都有可解释状态。
- 合木佳黄金值 `13,152` 不回退。
- 东安科目汇总、赣州纯外送、平泉宽表远程会诊能跑通最小可关账月。
- 所有金额都能拆解为 IN 劳务收入、OUT 外送/合作结算、调整、未匹配、成本未就绪或人工确认。
- 缺成本、缺规则、缺 LIS、缺价格不再按 `0` 静默入账。
- 已关账月份不被静默改写，只能生成调整单。

## 2. 产品原则

- 不强行把所有结算单塞进病例级模型。逐病例、科目汇总、宽表列金额、纯外送、联合运营并存。
- 原始文件不可变，规范化理解可重算，派生账本可重建，人工覆盖必须留痕。
- OUT 不是实验室劳务收入，但它是月结事实，必须进入财务闭环。
- `0` 只能代表明确业务值；未知、缺失、未就绪要通过质量标记暴露。
- 配置驱动不是让财务填配置表，而是让系统通过样表、候选规则、影响预估和一键确认减少配置成本。
- 月结工作台的核心问题不是“导入了几个文件”，而是“这个医院这个月还差什么不能关账”。

## 3. 用户角色和关键任务

### 3.1 财务月结负责人

关键任务：

- 收集当月所有医院结算单。
- 确认应收、扣率、外送分摊和调整。
- 判断每家医院是否可关账。
- 输出院级盈亏、差异包和审计记录。

核心诉求：少返工、可对账、可解释。

### 3.2 运营/实施人员

关键任务：

- 新医院建档。
- 识别模板和列语义。
- 维护规则和默认服务目录。
- 处理待人工归类、LIS 未匹配、成本缺失等异常。

核心诉求：样表驱动配置，规则变更前知道影响范围。

### 3.3 管理层/经营分析人员

关键任务：

- 查看院级收入、成本、毛利、OUT 占比和风险。
- 追溯报表数字来源。
- 区分真实毛利、成本未就绪和外送移出。

核心诉求：数据可信，口径稳定。

## 4. 版本变化：v2 到 v3.1

| 模块 | v2 | v3.1 |
| --- | --- | --- |
| 交付方式 | Phase 1 聚合 A/B/C/D/G 等多个 Epic | Phase 1 拆成 1A 最小可关账月、1B 模板扩展 |
| 数据模型 | 直接列出多张事实表 | 先统一规范行，再派生多类账本 |
| 月结工作台 | 文件状态总控 | 以关账状态机为主 |
| 质量标记 | `statement_quality_flags` | 阻断等级、责任人、处理动作、是否阻断入账/关账 |
| 指标 | 成功标准为主 | 增加月结完成时长、文件闭环率、金额解释率等指标 |
| 验收方式 | 3 样本验收包 | 3 样本跑通一个最小可关账月 |

## 5. 目标指标

### 5.1 主指标

月结完成时长：从当月首个文件上传到该月份关账的自然日。

Phase 1A 目标：3 个验收样本在一个演示月内完成闭环，形成可导出的最小月结包。

Phase 1B 目标：7 类模板均有初步处理路径，文件闭环率达到 80% 以上。

Phase 2 目标：规则变更能给出影响范围，已关账月份调整单可追溯。

### 5.2 过程指标

- 文件闭环率：进入 `posted`、`closed` 或 `adjusted` 状态的文件数 / 已上传文件数。
- 金额解释率：可归因到 IN、OUT、调整、未匹配、成本待补的金额 / 总金额。
- 自动分类率：无需新增人工规则即可分类的金额 / 总金额。
- 人工触达率：每万元结算金额需要人工处理的次数。
- 成本就绪率：已入账收入中成本已接通金额 / 已入账收入金额。
- 规则影响可解释率：规则变更后可列出影响批次和金额的变更数 / 总规则变更数。

### 5.3 防护指标

- 黄金值回归：合木佳 `13,152` 不回退。
- 误入账率：缺成本、缺价、缺 LIS 被当作正常完成的数量为 0。
- 已关账静默改写次数为 0。
- 跨院同号串账回归用例必须通过。

## 6. 产品架构

v3.1 采用两层数据架构：先规范化对账单金额行，再派生财务账本。

### 6.1 第一层：不可变导入与统一规范行

这一层不直接代表最终 P&L，而是代表系统对原始对账单的结构化理解。

核心对象：

- `statement_import_batches`：导入批次，记录医院、月份、模板、文件哈希、解析器版本、配置版本、上传人。
- `statement_raw_rows`：原始行快照，保留 sheet、row、原始单元格内容和解析上下文。
- `statement_normalized_lines`：规范化金额行，是 v3.1 的核心新增层。

`statement_normalized_lines` 建议字段：

| 字段 | 说明 |
| --- | --- |
| `batch_id` | 对应导入批次 |
| `partner_id` | 医院/合作方 |
| `service_month` | 服务月份 |
| `case_no` | 病例号，可为空 |
| `source_sheet` | 来源 sheet |
| `source_row` | 来源行 |
| `source_column` | 来源列，宽表必须保留 |
| `source_label` | 来源列业务标签，例如远程会诊结算 |
| `template_family` | 模板家族 |
| `line_grain` | `case` / `aggregate` / `out` / `joint` / `adjustment` |
| `business_line` | `IN` / `OUT` / `UNKNOWN` |
| `service_code` | 服务项目或规则映射代码 |
| `amount_role` | `gross` / `settlement` / `discount` / `cost` / `adjustment` |
| `amount` | 金额 |
| `classification_status` | 自动分类、待人工、已确认、已覆盖 |
| `rule_version` | 规则版本 |
| `quality_state` | 正常、警告、阻断 |
| `raw_payload` | 必要的原始上下文 |

这样做的原因：

- 宽表列语义、科目汇总、纯外送都能落在同一中间层。
- 新模板可以先落规范行，不必一开始决定最终账本表。
- 规则重算可以重建派生账本，不需要改原始批次。
- 质量标记可以绑定到具体规范行，而不是只在批次层泛泛提示。

### 6.2 第二层：派生账本与快照

这一层面向财务使用、P&L、导出和关账。

建议对象：

- `case_revenue_ledger`：病例级收入账本，可兼容现有 `case_revenue` / `case_revenue_lines`。
- `partner_month_revenue_ledger`：非逐病例聚合收入账本，承接东安科目汇总等。
- `out_settlement_ledger`：OUT 外送/合作结算台账，承接赣州纯外送、NGS、FISH、HPV-E6E7、远程、共建。
- `joint_operation_ledger`：联合运营/共建专项账本，按 `special.joint` 决定展示和 P&L 归属。
- `quality_flags`：质量标记和处理闭环。
- `pnl_snapshots`：院级 P&L 快照，支持未关账重算和已关账留档。
- `adjustment_entries`：已关账月份调整单。

派生原则：

- 原始批次和规范行不可变。
- 未关账月份可以重算并生成新快照。
- 已关账月份只能新增调整单。
- 派生账本必须可追溯到 `statement_normalized_lines` 和原始文件哈希。

## 7. 月结状态机

月结工作台以关账推进为主，而不是只展示导入任务。

### 7.1 批次状态

| 状态 | 含义 | 下一步 |
| --- | --- | --- |
| `uploaded` | 文件已上传 | 解析 |
| `parsed` | 已解析为 raw rows / normalized lines | 分类 |
| `classification_required` | 存在待归类行 | 人工确认或建规则 |
| `review_required` | 可入账但需复核 | 财务复核 |
| `posted` | 已派生到账本 | 等成本或关账 |
| `cost_pending` | 入账完成但成本未就绪 | 补成本或标记待补 |
| `ready_to_close` | 不存在阻断项 | 月份/医院关账 |
| `closed` | 已关账 | 只能调整 |
| `adjusted` | 已生成调整单 | 追踪调整 |

### 7.2 医院月份状态

| 状态 | 含义 |
| --- | --- |
| `not_started` | 该医院该月无文件 |
| `in_progress` | 已有文件但未完成 |
| `blocked` | 存在阻断项 |
| `reviewing` | 已入账但待复核 |
| `ready_to_close` | 可关账 |
| `closed` | 已关账 |
| `needs_adjustment` | 规则变更或补数后需调整 |

### 7.3 状态机验收

- 财务能在月份视图看到每家医院为什么不能关账。
- 每个阻断状态都能跳到具体批次、具体行、具体处理动作。
- 批次状态变更必须有操作人、时间和原因。

## 8. 质量标记闭环

质量标记不是提示文案，而是月结工作项。

`quality_flags` 建议字段：

| 字段 | 说明 |
| --- | --- |
| `flag_type` | 缺成本、缺价、缺 LIS、缺规则、疑似重复、人工覆盖、重算影响 |
| `severity` | `blocking` / `warning` / `info` |
| `owner_role` | 财务、实施、成本、系统管理员 |
| `resolution_action` | 补 LIS、补成本、建规则、人工确认、忽略并说明 |
| `blocks_posting` | 是否阻断入账 |
| `blocks_closing` | 是否阻断关账 |
| `related_line_id` | 关联规范行 |
| `related_batch_id` | 关联批次 |
| `resolved_by` | 处理人 |
| `resolved_at` | 处理时间 |
| `resolution_note` | 处理说明 |

默认阻断建议：

- 缺规则：阻断入账和关账。
- 缺成本：可允许收入入账，但阻断毛利关账或在 P&L 中单列。
- 缺 LIS：可允许收入暂入账，但阻断最终关账，除非财务选择带原因放行。
- 缺价格：阻断入账。
- 纯 OUT 无病例号：不阻断，进入 OUT 台账。
- 联合运营：按 `special.joint` 配置决定账本和展示口径。

## 9. 路线图

### Phase 0：可信度止血

周期建议：1-2 周。

Outcome：当前已存在的收入、LIS、ABC、P&L 链路不再污染数据。

范围：

- 全链路复合键 `(partner_id, case_no)`。
- LIS 导入、人工覆盖、ABC 回填、P&L join、成本 rollup 全部带 partner 维度。
- 配置读路径、回滚路径做 best-effort normalize 和校验。
- NGS 缺成本/售价不再按 0 静默入账，输出质量标记。
- 黄金值 `13,152` 回归保护。

验收：

- 跨院同号回归用例通过。
- `buildPartnerPnl`、`loadCasePnlsWithCost`、`backfillAbcPartnerIds` 覆盖 partner 维度。
- 历史 `discount.def=90` 读出后按 0.9 口径处理。
- 缺成本/缺价 NGS 不进入正常毛利。

非目标：

- 不重做完整月结工作台。
- 不覆盖全部 7 模板。
- 不做 maker-checker。

### Phase 1A：最小可关账月

周期建议：2-3 周。

Outcome：用 3 个真实样本跑通一个最小月结闭环，证明系统可以承接非逐病例、纯 OUT 和宽表列语义。

验收样本：

- 东安科目汇总。
- 赣州纯外送。
- 平泉宽表远程会诊。

范围：

- 建 `statement_import_batches`、`statement_raw_rows`、`statement_normalized_lines` 基础结构。
- 月结工作台基础版：月份、医院、批次状态、金额拆解、阻断原因。
- 东安：科目汇总进入 `partner_month_revenue_ledger`。
- 赣州：纯 OUT 进入 `out_settlement_ledger`，`labRevenue = 0`，`outSettle = 全额`。
- 平泉：宽表金额保留 `sourceColumn/sourceLabel`，支持按列确认规则。
- 基础 `quality_flags`：缺规则、缺 LIS、缺成本、缺价。
- 最小月结差异包：IN、OUT、未归类、成本待补、人工确认。

验收：

- 3 个样本在同一演示月内从上传、解析、分类、入账/登记到月结汇总全链路跑通。
- 每笔金额能追溯到原始文件、来源行、来源列和规则版本。
- 工作台能显示每个批次是否阻断入账、是否阻断关账。
- 可导出最小月结包。

非目标：

- 不要求 7 模板全量完成。
- 不要求公司级 OUT 系统接口。
- 不要求复杂审批流。

### Phase 1B：模板扩展与确认体验

周期建议：2-4 周。

Outcome：真实 7 模板均有处理路径，首版月结可以覆盖主要医院样本。

范围：

- 温州混合服务费：IN/OUT 混合分类，HPV-E6E7/FISH/NGS/远程目录补齐。
- 宁波宽表诊断费：复用宽表列级语义。
- 石门联合运营：按 `special.joint` 进入联合运营账本或单列 P&L。
- 模板化预览组件：病例明细、宽表、科目汇总、纯 OUT、联合运营。
- confirm 增加原因、影响摘要、审计记录。
- 默认服务目录支持全局/逐院启停和覆盖。

验收：

- 7 类模板均可进入明确状态：已入账、OUT 台账、联合运营、待规则或待补数。
- 待人工归类页面提供候选规则、影响金额和下一步动作。
- 人工确认必须可追溯到操作人、原因和前后差异。

### Phase 2：治理、重算与关账制度

周期建议：3-6 周。

Outcome：规则变更和补数不会破坏已关账数据，财务敢持续维护配置。

范围：

- 规则影响预估：列出受影响医院、月份、批次、金额和 P&L。
- 未关账月份一键重算。
- 已关账月份生成调整单。
- `pnl_snapshots` 留存每次关账快照。
- LIS 匹配恢复：候选匹配、手动绑定、忽略原因、补匹配。
- 黄金值产品化：前端维护财务期望实收和差异阈值。
- maker-checker：大额规则变更、OUT 改 IN、未识别确认入账需要复核。

验收：

- 修改 HPV-E6E7 OUT 规则前能看到影响范围。
- 已关账月不被改写，只生成调整单。
- 未关账月重算后保留前后快照。
- LIS 未匹配能导出和回填。

### Phase 3：规模化运营和外部协同

周期建议：后续迭代。

Outcome：从单月可关账扩大到多院、多文件、多人协同的稳定月结运营。

范围：

- 批量文件自动识别医院、月份和模板。
- 样本库：按医院、模板、月份保留脱敏样本和验收结果。
- NGS/外送前端导入入口。
- 公司级 OUT 台账接口或导出。
- 完整度评分和月结风险评分。
- 任务分派和通知。
- 季度模板库复盘：低价值规则清理、常见项目沉淀为全局目录。

验收：

- 60 份文件级别的月结可在工作台统一推进。
- OUT 台账可以交接到公司级财务或采购流程。
- 每家医院有数据完整性评分和未完成原因。

## 11. PRD 切片建议

### PRD-0：可信度止血

目标：修复当前会污染数据的正确性问题。

交付物：

- 复合病例键迁移方案。
- 配置读/回滚归一。
- NGS 缺成本/售价质量标记。
- 回归测试包。

### PRD-1A：最小可关账月

目标：3 样本跑通一个演示月的端到端月结闭环。

## 13. 风险与控制

| 风险 | 表现 | 控制 |
| --- | --- | --- |
| Phase 1A 范围膨胀 | 试图一次覆盖 7 模板和完整审批 | 明确 1A 只验收 3 样本和最小月结包 |
| 规范行过度设计 | 字段太多导致工程迟迟不能落地 | 先保留必需字段，raw_payload 承接长尾 |
| 质量标记无人处理 | flags 变成告警池 | 强制 owner_role、resolution_action、blocks_closing |
| 工作台退化为文件列表 | 财务看不到为什么不能关账 | 医院月份状态必须作为主视图 |
| OUT 口径混入实验室毛利 | P&L 失真 | OUT 单列台账，默认不进实验室工序毛利 |
| 已关账被重算覆盖 | 报表不可审计 | 快照 + 调整单，禁止静默改写 |

## 14. 待确认事项

5. 规范行字段命名是否与现有表兼容。建议工程设计时保留映射层，不强行一次改完所有旧表。

## 16. 结论

v2 已经完成方向收束和关键口径决策；v3.1 的重点是让工程和产品能更快验证真实月结闭环。

建议按“Phase 0 可信度止血 -> Phase 1A 最小可关账月 -> Phase 1B 模板扩展 -> Phase 2 治理关账 -> Phase 3 规模化运营”推进。这样既不牺牲 v2 的完整性，也避免 Phase 1 变成多个 Epic 同时展开但无法验收的功能堆叠。
