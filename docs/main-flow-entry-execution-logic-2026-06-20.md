# 主流程导航入口执行逻辑

> 创建日期: 2026-06-20
> 配套文档: `docs/main-flow-entry-goal-workflow-2026-06-20.md`
> 入口队列: `docs/main-flow-entry-list-2026-06-20.md`

## 一、执行模式

这是一个顺序执行工作流:

1. 从入口清单选择入口。
2. 锁定当前入口。
3. 全面检索当前入口链路。
4. 质疑并评估当前入口的产品建模。
5. 现场建立并维护问题清单。
6. 修复当前入口的高频主流程问题和关键建模问题。
7. 验证页面、后端副作用、数据回看和事实链影响。
8. 收口当前入口。
9. 自动切换到下一个入口。

执行者不得在入口未收口时跳到另一个导航入口做修复。相邻入口问题只能记录为线索。

## 二、启动步骤

每次目标模式启动或恢复时，先执行:

```bash
git status --short --branch
git log --oneline --decorate -n 40
```

然后读取:

1. `docs/main-flow-entry-goal-workflow-2026-06-20.md`
2. `docs/main-flow-entry-execution-logic-2026-06-20.md`
3. `docs/main-flow-entry-list-2026-06-20.md`

如果存在未提交改动，先判断是否属于当前入口或历史遗留。无关改动不得回退、覆盖或顺手整理。

## 三、入口选择算法

按以下顺序选择当前入口:

1. 如果入口清单中有 `in_progress`，继续这个入口。
2. 否则选择排序最靠前的 `pending`。
3. 如果某入口状态是 `blocked`，只有阻塞条件已消失才恢复，否则跳过到下一个 `pending`。
4. 如果所有入口均为 `completed`，目标完成。

选择入口后，必须在入口清单中把状态更新为 `in_progress`，并创建或继续对应工作记录。

## 五、全面检索要求

围绕当前导航入口，从入口反向检索完整链路。至少覆盖:

1. 前端路由、页面、组件、hook、状态管理、API 调用。
2. 后端路由、service、数据表、事务、审计日志。
3. 类型定义、接口契约、错误处理、权限校验。
4. 已有单测、集成测试、页面测试、E2E 记录。
5. 交互规范、FRS、测试场景、历史审计报告中与该入口有关的线索。
6. 该入口对 ABC、库存、BOM、出库、成本、关账或审计事实链的影响。
7. 当前入口的分类、字段、关系、状态流转是否能表达真实业务，而不是只检查代码是否能运行。

建议命令:

```bash
rg -n "<入口标签|路由|接口名|表名|业务关键词>" 前端代码/src 后端代码/server/src docs V1.1设计稿
find 前端代码/src/pages 后端代码/server/src/routes -maxdepth 2 -type f | sort
```

检索完成后，先把入口链路写入工作记录，再开始改代码。

## 六、产品建模评估

执行者不能默认当前产品设计正确。每个入口必须先完成一轮产品建模评估，并写入工作记录。

必须质疑的问题包括:

1. 分类是否准确: 现有分类是否能被真实用户理解和长期维护，是否会把不同业务口径混在一起。
2. 字段是否够用: 是否缺少支撑主流程、追溯、成本分析或审计的关键字段。
3. 关系是否正确: 主数据和过程数据之间是否存在错误绑定、弱绑定或无法追溯。
4. 状态流转是否合理: 启用、停用、删除、取消、审核、关账、重算是否保护历史事实。
5. 用户工作量是否合理: 是否要求用户重复录入、线下修正、人工核对或记住系统外规则。
6. 成本分析是否支撑: 该入口产出的数据是否能支撑可信、可重算、可审计的成本结果。

建模问题不是“建议项”。如果它会导致后续业务事实错误、成本分析失真、用户长期手工修正，必须进入现场问题清单。

## 七、问题分级

问题清单只记录本入口现场发现的问题，不提前填充旧审计缺口。

| 优先级 | 判定标准 | 当前入口处理要求 |
|:---:|:---|:---|
| P0 | 主流程无法完成，或会写错库存、成本、审计关键事实 | 必须修复并验证 |
| P1 | 主流程能完成但结果不可信、不可回看、需要人工修正，或产品建模会导致事实链/成本分析失真 | 必须修复并验证 |
| P2 | 高频体验或效率问题，会增加日常操作成本 | 影响主流程则修复，否则说明暂缓 |
| P3 | 低频边界、细节优化、与当前入口主流程关系弱 | 默认记录，不展开 |

如果旧审计报告里有问题，但当前入口现场复核后不影响主流程，应记录为 P3 或历史线索，不要自动升级。

## 八、修复顺序

按业务闭环修复，而不是按文件类型修复:

1. 先确认分类、字段、关系、状态流转是否能承载真实业务。
2. 再让用户可以完成主动作。
3. 再保证后端副作用正确。
4. 再保证结果可回看、可追踪。
5. 再补齐错误提示、禁用状态、确认弹窗。
6. 最后补测试和回归。

涉及 bug 修复或新能力时，优先补能证明业务结果的测试。测试不要只断言按钮存在，要断言真实副作用或数据回看。

## 九、验证要求

每个入口至少选择与风险匹配的验证组合:

- 页面/弹窗真实操作或 Playwright 等价验证。
- API 请求和响应验证。
- 数据库或后端状态验证。
- 列表、详情、报表、日志或下游入口回看。
- 单测、集成测试、构建或针对性回归。
- 如果影响库存、BOM、出库、成本或 ABC 输入，补相关回归。
- 对分类、字段、关系、状态流转等建模调整，必须说明它如何降低后续手工修正或成本分析失真。

不能只因为测试通过就判定入口完成；也不能只因为页面按钮存在就判定入口完成。

## 十、自动切换规则

当前入口满足完成标准后:

1. 更新工作记录的收口报告。
2. 更新入口清单中该入口状态为 `completed`。
3. 填写完成日期、工作记录路径、验证摘要。
4. 重新读取入口清单。
5. 选择下一个 `pending` 入口继续执行。

如果上下文或时间不足，必须至少确保入口清单和工作记录能让下一次恢复时继续，不要只在最终回复中口头说明。

## 十一、阻塞规则

只有以下情况可以把入口标记为 `blocked`:

1. 缺少必须由用户决定的产品口径，且无法做保守假设。
2. 当前入口需要外部服务、账号、数据或权限，且本地无法替代验证。
3. 继续执行会覆盖用户未确认的重要改动。

标记 `blocked` 时必须写明:

- 阻塞原因。
- 已完成的检索和修复。
- 需要用户提供或决定什么。
- 恢复后应从哪一步继续。

普通难度高、文件多、测试慢，都不是阻塞理由。

## 十二、收尾输出

每次回复用户时，优先说明:

1. 当前入口是什么。
2. 本轮发现并处理了什么 P0/P1/P2。
3. 用什么验证证明入口推进了。
4. 是否已自动切换到下一个入口。
5. 如果没有切换，剩余阻塞或未完成项是什么。
