# COREONE 阶段3功能设计依据（Design Rationale）

> 版本: v1.0
> 日期: 2026-05-29
> 范围: 替代物料管理 + BOM Panel 分组 + 成本核算准确性
> 依据: 行业调研（CAP/ISO 15189/病理科实践）+ 两轮需求确认 + 代码走读

---

## 一、Panel 分组成本统计

### 1.1 为什么需要分组

**临床现实**：免疫组化检测中，Panel 检测是常规工作流：

| Panel 名称 | 包含抗体 | 临床意义 |
|-----------|---------|---------|
| MMR 四联检 | MLH1 / PMS2 / MSH2 / MSH6 |  Lynch 综合征筛查 |
| 乳腺癌四联 | ER / PR / HER2 / Ki-67 |  乳腺癌分子分型 |
| 淋巴瘤初筛 | CD20 / CD3 / CD45 / BCL2 |  淋巴瘤 B/T 细胞分型 |

国家收费编码 **270500002**（免疫组织化学染色诊断）采用"基础项目费 + 按抗体数量加收"模式。科室管理者需要知道：
- 整个 Panel 的总成本（评估定价合理性）
- Panel 内各子项的独立成本（识别高成本子项，优化采购）
- 通用试剂（如 DAB、缓冲液）的分摊占比

**不采用"每个抗体建一个项目"的原因**：
- 临床开单时 Panel 是一个整体，拆分后增加操作负担
- 拆分后无法自然归集通用试剂成本
- 与国家收费编码的 Panel 计费模式不匹配

### 1.2 分组在 BOM 内的实现方式

**决策**: 在 `bom_items` 表增加 `group_name` 字段，同一 BOM 内按 group_name 聚合成组。

**为什么不做"BOM 嵌套"或"子 BOM"**：
- 嵌套 BOM 增加理解成本和实现复杂度
- 用户当前诉求是"成本分组统计"，不是"BOM 复用"
- `group_name` 是最小改动，不影响现有所有逻辑

**分组示例**（MMR 四联检 BOM）：

```
BOM: MMR 四联检
├── group: "MLH1"
│   └── 物料: MLH1 抗体 (Dako)  — 用量: 1滴
├── group: "PMS2"
│   └── 物料: PMS2 抗体 (Dako)  — 用量: 1滴
├── group: "MSH2"
│   └── 物料: MSH2 抗体 (Dako)  — 用量: 1滴
├── group: "MSH6"
│   └── 物料: MSH6 抗体 (Dako)  — 用量: 1滴
└── group: "通用试剂" (或不填，作为未分组)
    ├── DAB 显色液  — 用量: 2ml
    ├── PBS 缓冲液  — 用量: 5ml
    └── 苏木素复染   — 用量: 1ml
```

**报表价值**：
- 可下钻查看"MMR 四联检 → MLH1 组成本 vs 通用试剂组成本"
- 识别成本异常：某组成本突然升高 → 检查该抗体是否换了高价品牌

## 二、替代物料管理（修正后方案）

### 2.1 第一轮方案的偏差与修正

**第一轮假设**（已证伪）：
- 假设: 不同品牌抗体可以"自动替代"，只需选有库存的品牌
- 问题: 忽略了不同品牌的克隆号、稀释比、染色方案差异
- 风险: 自动替代可能导致染色结果不一致，影响病理判读

**第二轮调研确认的行业事实**：
1. Dako vs 迈新 vs 中杉金桥的同一靶点抗体，克隆号可能不同（如 Ki-67: MIB-1 vs SP6）
2. 不同克隆号的染色强度和判读阈值有差异，病理科医生对品牌有明确偏好
3. CAP/ISO 15189 要求新品牌/新批次使用前需做平行染色比对验证
4. 但验证是实验室**质量体系/SOP**层面的要求，不是库存系统必须硬拦截的

### 2.2 修正后的设计方案："已准入品牌池"

**核心原则**：系统不做"任意替代"，只做"池内选择"。

**概念定义**：
- **品牌池**: 针对 BOM 中某一物料行，实验室已确认可互换使用的多个品牌集合
- **准入**: 由技术员/病理医生在 BOM 配置时手动指定哪些品牌纳入池
- **出库选择**: 在池内按 FEFO + 库存充足原则自动选择

**与 ERP 替代物料的区别**：

| 维度 | ERP 替代物料 | 病理试剂品牌池 |
|------|-------------|---------------|
| 替代依据 | 规格相同即可 | 需实验验证确认 |
| 替代范围 | 全局物料级 | BOM 内限定（同一 Panel 可能允许不同品牌组合） |
| 用户决策 | 系统自动 | 技术员预配置，系统自动执行 |
| 风险等级 | 低（工业品） | 高（医疗诊断） |

**为什么不做"主物料 + 替代物料"的二元层级**：
- "主/替"暗示了优先级，但真实场景中优先级可能变化（如某月 Dako 促销，可能优先用 Dako）
- 二元层级无法表达"三个品牌等效"的情况
- 品牌池是平级关系，更灵活

### 2.3 BOM 内品牌池的实现方式

**数据模型**：
- 同一 `group_name` 下的多个 `bom_items` 记录构成一个品牌池
- 每个 `bom_items` 记录对应一个具体的物料（品牌）
- 不增加新表，复用现有 `bom_items` 结构

**示例**（Ki-67 品牌池）：

```
BOM: 乳腺癌四联检
├── group: "Ki-67"
│   ├── bom_item_1: materialId=Dako-Ki-67-MIB1,  usage=1滴
│   └── bom_item_2: materialId=迈新-Ki-67-SP6,   usage=1滴
├── group: "ER"
│   └── bom_item_3: materialId=Dako-ER-SP1,      usage=1滴
└── ...
```

**出库逻辑**：
1. 计算该组总需求（如 10 个样本 × 1滴 = 10滴）
2. 遍历品牌池内各物料的批次库存（按 FEFO）
3. 先尝试物料 A 的批次分配，不足则继续用物料 B 的批次
4. 记录实际消耗的物料和批次，用于成本核算

**用户可见性**：
- BOM 详情页显示品牌池内各品牌及当前库存状态
- 出库记录显示实际使用了哪个品牌（成本追溯）
- 成本报表按实际消耗品牌统计（不是按 BOM 预设品牌）

### 2.4 批次验证状态（信息展示层）

**调研结论**：
- CAP ANP.22760: 新批次需平行染色比对
- CAP ANP.22780: 供应商变更需 2 阳 + 2 阴确认病例
- ISO 15189: 验收验证记录需包含批号、效期

**设计决策**：
- 批次增加 `verified` 字段（INTEGER, 0=未验证, 1=已验证, 默认1兼容旧数据）
- **不拦截出库**: 验证是实验室 SOP 要求，紧急情况下不应被系统阻断
- **信息展示**: 在库存列表、批次详情中标注验证状态，提醒仓库管理员
- **报表区分**: 成本报表可筛选"仅看已验证批次成本"（如有需要）

**为什么不强制拦截**：
- 病理科存在紧急检测场景，系统不应成为临床障碍
- 验证记录通常由技师在纸质或 LIS 中完成，库存系统不掌握完整验证信息
- 强制拦截会导致用户绕过系统（手工记账），反而失去追溯能力

## 三、成本核算准确性

### 3.1 为什么用 FEFO（先过期先用）

**行业惯例**：试剂管理不同于普通商品，有效期是硬约束。过期试剂必须报废，不能用。
- 先用快过期的，减少报废损失
- 符合《医疗器械监督管理条例》对试剂有效期的管理要求
- 与医院 SPD 供应链管理模式一致

### 3.2 为什么用加权平均成本（不是标准成本）

**两种方法的对比**：

| 方法 | 计算方式 | 优点 | 缺点 |
|------|---------|------|------|
| **标准成本** | 预设统一单价 | 简单、便于预算 | 不反映实际采购价格波动 |
| **加权平均（实际成本）** | 按实际消耗批次加权 | 准确反映真实成本 | 计算复杂、需解释 |

**选择加权平均的理由**：
1. 试剂价格波动大（集采、供应商促销、进口汇率影响）
2. 用户核心诉求是"成本核算准确"，不是"预算控制"
3. 同一物料不同批次价格差异可达 20-50%（尤其进口抗体）

**用户理解成本的问题**：
- 加权平均计算对用户不直观，需要系统展示计算过程
- 建议阶段4增加"成本明细展示"功能（已列入 task_plan.md 阶段4.3）

### 3.3 为什么 BOM 理论成本要动态计算

**现状问题**：v1.0 中 BOM 理论成本在创建时计算并静态存储：
```typescript
const totalCost = materials.reduce((sum, m) => sum + (m.price || 0) * m.usagePerSample, 0)
```

**问题**：
- 后续采购价格变化，理论成本不更新
- 对账时"理论 vs 实际"差异包含价格波动因素，无法区分管理因素

**动态计算方案**：
- BOM 详情页实时计算理论成本：`Σ(usagePerSample × 当前批次加权平均价)`
- 不存储静态值，每次查看时计算
- 对账差异 = 实际出库成本 - 实时理论成本（更准确）

## 四、功能 vs 数据层面的区分

本次阶段3的修改，需要明确区分**数据层变更**和**功能层变更**：

| 层面 | 变更内容 | 用户感知 |
|------|---------|---------|
| **数据层** | `bom_items.group_name`、`batches.verified` | 无直接感知 |
| **功能层** | BOM 编辑时增加"分组"输入框 | 有感知 |
| **功能层** | BOM 编辑时同一组可添加多个品牌 | 有感知 |
| **功能层** | BOM 详情按 group 分组展示 | 有感知 |
| **功能层** | 新增"项目分组成本"报表 Tab | 有感知 |
| **功能层** | 成本报表增加计算过程展示 | 有感知（阶段4） |

**PRD 需要补充的是功能层变更**，而非仅数据层字段定义。

## 六、风险与缓解决策

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 品牌池配置增加技术员操作负担 | 中 | 默认单品牌（不配置池=现有行为），需要时才扩展 |
| 分组成本报表数据为空（旧 BOM 无 group） | 低 | 旧数据 group_name 为空，报表中显示为"未分组" |
| 多品牌出库后成本追溯复杂 | 中 | 出库记录保存实际使用的物料 ID 和批次，可追溯 |
| 用户误解"品牌池"为"任意替代" | 高 | 前端增加说明文案："仅限已验证品牌" |
