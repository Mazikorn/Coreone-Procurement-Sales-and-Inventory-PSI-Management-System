## 1. 产品目的基线（评估的标尺）

依据 `docs/01_Project_Charter.md`、`docs/02_PRD.md`、`docs/ABC成本核算产品化诊断与重构建议.md`：

- **系统定位**：病理免疫组化特染实验室的进销存 + 单张切片成本控制系统（PSI），单实验室 B 端部署。
- **两大核心价值**：
  1. 耗材**全链路追踪**（采购→入库→批次→出库→使用，可追溯到批次和操作人）。
  2. **单张切片精确成本核算**——章程 G2 原文：*"出库时自动计算材料成本、设备折旧、人工工时、间接成本分摊"*。
- **成功指标 S4**：单张切片成本核算准确，**偏差 ≤5%**（与手工核算对比）。
- **战略路线（已正确定调）**：`以收费为锚点，以 ABC 为引擎`——市场调研支持混合模式，最终面向用户输出"收费 vs 成本""项目是否盈利""成本为何波动"。

> 这条标尺很关键：产品自己承诺的是**四要素、按动因分摊、可解释、≤5% 准确**。下面所有评估都对照这条标尺。

## 2. 七视角评估

### 视角一：产品 / 战略 — ✅ 方向正确

- 定位清晰、差异化明确（进销存天然拥有精确材料出库 + BOM 用量数据，这是做切片成本的真实优势）。
- "收费锚点 + ABC 引擎"的混合路线判断成熟，避免了"纯理想化 ABC"的落地陷阱。
- 非目标边界（不做多租户 / 不做完整财务总账 / 不做 LIS 实时对接 / 不做审批流）与单实验室定位匹配，**范围克制得当**。
- **结论：方向无需修改。** 问题在执行深度，不在战略。

### 视角二：架构 / 后端 — 🟡 骨架真实，内核空心

- **真实且优秀的部分**：成本期间状态机（open→collecting→calculated→closed）、关账门禁（关账前校验未处理收费映射异常 / error 级异常 / 待补算出库，见 `abc-v1.1.ts` close 端点）、`cost_exceptions` 异常台账（resolve/ignore/retry）、`cost_runs` 重算任务、关账后调整单、审计日志——这套**核算治理骨架是真实资产**。
- **数据模型完整**：`abc_periods`、`cost_runs`、`cost_exceptions`、`bom_fee_mappings`、`case_charge_groups`、`outbound_abc_details`、`abc_cost_adjustments` 等，支撑闭环的表都在。
- **致命缺陷**：成本引擎核心算法不成立（见 §3 详解）。
- **技术债**：① 存在两套口径不一的成本引擎（见 §3.3）；② `abc-v1.1.ts` 单文件 3381 行，可维护性差（非阻断）；③ SQLite 并发写受限（已知约束，对 5-20 用户规模可接受）。
- **结论：骨架可留，内核须重建。**

### 视角三：成本正确性 / 财务可信 — ❌ 核心不满足（本评估的头号结论）

详见 §3。一句话：**所谓 ABC 实际退化为"总间接费 ÷ 总样本量"的单一吸收率，人工/设备/间接未按动因追溯，两套引擎口径冲突，且没有任何测试证明 S4 的 ≤5% 准确度。** 产品承诺的"精确""可解释"在引擎层面落空。

### 视角四：前端 / UX — ✅ 超前，但建在沙上

- 18 个成本页面（成本看板 / 切片成本 / 盈利分析 / 收费对照 / 成本池 / 作业中心 / 成本动因 / 异常中心 / 审计 / 调整单 / 模型验证…），真实接 40+ 后端 API，路由 + 侧边栏 + 角色权限齐全。
- B2B UX 优化（业务单据号深链回跳、确认弹窗点名下游影响、文案对齐真实能力）成熟，确实在消除"退回线下核对"的摩擦。
- **问题**：前端忠实地把一个**不可信引擎**的输出渲染成漂亮的"切片成本下钻"，用户看到的分解**不是真实的动因分解**——这正是诊断文档警告的"看起来像成本分析"陷阱。
- **缺口**：缺面向决策的工作台（哪个 BOM 成本异常、哪些动因配置不完整、成本-收费偏离度）。
- **结论：前端不是瓶颈；它在等一个可信的引擎。**

### 视角五：测试 / 质量 — ❌ 在最关键处证据最薄弱

- 既有审计自陈：空测试文件、宽松断言、**ABC 计算器单元测试仍失败**、病理工作流 fixture 缺失致 24 测试 skip、前端 3 个 `useInventoryPage.test.ts` 失败、lint 未绿。
- 关键矛盾：**关账门禁校验的是"有没有未处理异常 / 状态对不对"，而不是"成本数字对不对"。** 系统可以一路 open→closed 关账成功，却从未有人验证那个切片成本是不是真的。**正是这种测试缺位，让 §3 的平均分摊 bug 一直存活。**
- **结论：必须补"成本准确性"的黄金用例测试，否则 S4 永远无法证明。**

### 视角六：安全 / 审计 — 🟡 试点够用，宣称"替代台账"不够

- 历史 P0（operator 取自请求体 = 审计可伪造）已修复。
- 剩余 P2/P3：外部存储真实 WORM/留存锁未强制（声明级）、签名密钥托管/轮换未落地、统一审计写接口覆盖矩阵未补全（LOG-031-004）。
- **结论：内部试点可接受；要宣传"完全替代线下台账"需补这些。** 非头号矛盾。

### 视角七：治理 / 基线一致性 — 🟡 改善中

- 历史 P0-1（主目录注册了路由但缺失 `abc-v1.1.ts` 等核心文件，完整实现只在 worktree）已由 DEC-013 决策迁移到独立分支 `codex/abc-productization-phase0-1` 解决——**方向正确**。
- 张力：DEC-009"BOM 历史版本不保留"与 ABC 诊断"标准成本应版本化锁定历史"存在未完全调和的设计张力（已用追溯重算 + 审计快照部分缓解）。
- "6 角色"表述已过时（实际新增 manager）。
- **结论：治理在收敛，非阻断。**

## 3. 头号发现详解：ABC 成本引擎并非作业成本法（代码核验）

### 3.1 切片成本怎么算的

`后端代码/server/src/utils/cost-calculator.ts` 的 `calculateSlideCostWithFee`（462-524 行），出库时触发：

```
切片总成本 = 材料成本 + Σ_作业中心( driver_rate × 本切片动因量 )
```

- **材料成本（真实 ✅）**：按 BOM 物料用量 × FEFO 批次加权价（526-551 行）。这部分正确。
- **作业成本**：完全依赖各作业中心成本池的 `driver_rate`（477-481 行从 `abc_cost_pools` 取）。
- 注意：此函数**不直接调用** `calculateLaborCost` / `calculateEquipmentCost`——这本身对 ABC 是**正确**的，因为人工/设备本应通过成本池的动因费率进入。**所以问题不在这里，而在成本池怎么被填的。**

### 3.2 成本池怎么填的——退化的根源

`后端代码/server/src/routes/abc-v1.1.ts` 的 `autoCollectCostPools`（457-483 行，即前端"自动归集"按钮的后端）：

```js
const driverQty     = Math.max(1, sourceTotals.sampleCount)          // 全月总样本量
const directShare   = (laborTotal + equipmentTotal) / centers.length  // ← 按成本中心个数平均分
const indirectShare = indirectTotal / centers.length                  // ← 平均分

for (const center of centers) {
  upsertCostPool({ activityCenterId: center.id,
    directCost: directShare,      // 每个中心拿到一样的钱
    indirectCost: indirectShare,  // 每个中心拿到一样的钱
    driverQuantity: driverQty })  // 每个中心都用同一个全局样本量
}
```

**每个作业中心被分到完全相同的成本和完全相同的动因量。** 于是 driver_rate 对每个中心都一样，切片成本退化为：

```
切片作业成本 ≈ (本 BOM 关联的中心数) × (总间接费 / 中心数 / 总样本量) × 本切片样本数
```

如果一个 BOM 关联了全部 N 个中心，进一步坍缩为：

```
切片作业成本 ≈ 总间接费 × (本切片样本数 / 总样本量)
```

**这就是一个单一的全厂吸收率（总开销 ÷ 总产量），不是 ABC。** 而且比干净的吸收率更糟：

1. **结果取决于"BOM 碰巧关联了几个中心"** —— BOM-A 关联 2 个中心、BOM-B 关联 5 个中心，吸收的开销不同，却没有任何动因依据，纯粹是"关联数 × 平均分"的副作用。
2. **动因量对所有中心都用"全月样本量"** —— 哪怕某中心的动因是"蜡块数 / 机器分钟数"，分母仍是样本数，费率口径错。
3. **作业中心、成本动因、BOM-作业关联三张表结构都在，但不产生任何成本差异化** —— 而"按动因差异化成本"正是 ABC 存在的全部意义。

### 3.3 还有第二套口径冲突的引擎

同文件 `calculateProjectCost`（360-378 行）走的是**另一套**算法：材料 + 人工 + 设备 + 质控 + 间接，**五要素全部直接从标准值计算**。这套用于项目/传统成本报表。

→ **同一个项目/切片，"切片 ABC 引擎"和"项目成本引擎"会算出不同的数。** 诊断文档早已点名"传统成本分析和 ABC 报表口径不统一，必须统一成本入口函数"，但**至今未统一**。财务一旦发现两个页面对同一项目给出两个成本，整个系统的可信度崩塌。

### 3.4 公平起见的澄清

- 成本池**也可以**手工录入（`POST /cost-pools`）或 `sync`。理论上财务可以手动给每个中心填正确的钱——但 ① 即便手填也没有"动因追溯"的支持；② 产品力推的"自动归集"按钮恰恰是退化的那条路；③ 手填任意数字也不构成"可信核算"。**缺口成立。**
- 闭环治理（期间/异常/关账/审计）是真的、好的。问题精确地落在**"成本数字本身怎么来"**这一层。

### 3.5 对产品目的的直接冲击

| 产品承诺 | 现状 |
|---|---|
| G2：四要素按动因分摊到单张切片 | ❌ 人工/设备/间接被平均分摊，非动因驱动 |
| S4：切片成本偏差 ≤5% | ❌ 无任何测试证明；平均分摊下偏差不可控 |
| "成本透明化 / 可解释" | ❌ 切片成本无法被真实解释为"某作业按某动因以某费率消耗" |
| "可信财务核算结果" | ❌ 两套引擎口径冲突 + 平均分摊，财务无法采信 |

## 4. 修改方案

> 原则：**不推倒重来，不再加页面。** 闭环骨架、前端、数据模型都保留；只重建"成本数字怎么来"的正确性内核，并用验收测试钉死准确性。本方案**增强**而非替代既有 Phase 0-4 计划——既有计划把"成本池有费用来源和动因量"当勾选项，却没规定**归集必须按动因、不能平均分**，这正是要补的洞。

### 4.1 核心修复（P0，缺一不可）

**修复 ① 真实的成本池归集（引擎心脏）**
- 每个作业中心声明自己的**动因类型**和**成本归属来源**（哪些人工步骤 / 哪些设备 / 哪部分间接费属于本中心）。
- 人工：`standard_labor_times` 的步骤 → 映射到作业中心，按中心汇总人工成本。
- 设备：`equipment_usage` / `bom_equipment_templates` → 设备映射到中心，按中心汇总折旧。
- 间接：`indirect_cost_allocations` → 按**声明的分摊基准**分配到中心（不是平均）。
- 每个中心的 `driverQuantity` = **该中心自己动因**的当期实际总量（该作业处理的切片/蜡块/机器分钟），不是全局样本量。
- `driver_rate = 中心成本 / 中心动因量`。
- **删除 `autoCollectCostPools` 的平均分摊逻辑。**

**修复 ② 统一成本入口函数**
- `calculateProjectCost` 与 `calculateSlideCostWithFee` 合流到**单一成本事实源**：项目成本 = 切片/病例成本的聚合，而非独立从标准值重算。消除口径漂移。

**修复 ③ 准确性验收测试（证明 S4）**
- 黄金数据集：构造一个已知的实验室月份（已知人工分钟/设备用量/间接费/样本数/BOM），**手工算出每张切片应有的成本**，断言引擎输出 ≤5% 偏差，**且** Σ(分摊到切片的成本) 与 Σ(成本池成本) 对账（完全吸收，不漏不重）。
- 这条测试若早存在，§3 的平均分摊 bug 不可能存活。接入 CI。

**修复 ④ 可解释性 = 硬约束**
- 切片成本下钻必须展示每个作业的：动因类型、消耗的动因量、动因费率、费率出处（哪些来源成本、哪个期间）。
- **任何无法这样解释的数字 → 进 `cost_exceptions`，不许静默显示。**

### 4.2 一个需要拍板的方向：引擎深度

这一步**改变整个实施计划**，且属于会计政策/产品范围决策，需要你定：

| 方案 | 含义 | 代价 | 适配度 |
|---|---|---|---|
| **A. 真·全 ABC** | 多动因类型、每中心独立动因、完整动因追溯 | 最重 | 字面满足"精确成本核算"，但实施复杂、用户接受度低（市场调研已警示） |
| **B. 诚实的"动因优先 + 单一披露基准"（推荐）** | 干净动因处做真追溯（材料 ✅、设备分钟、人工分钟），残余间接费用**一个公开声明的基准**分摊；全程可解释 + 可对账 + 有验收测试；产品主打"收费 vs 成本 / 盈利" | 中 | **最贴合你们自己定的"收费锚点 + ABC 引擎"路线，最快到达"可信+可解释"** |
| **C. 最小诚实修复** | 保留现结构，但 ① 平均分→声明基准 ② 修每中心动因量 ③ 加对账测试 ④ UI 文案不再过度宣称"精确 ABC" | 最轻 | 仅止血，不提升产品力 |

**推荐 B**：在动因干净的维度做真实追溯（材料已真实，设备分钟、人工分钟都是干净动因），对真正无法干净归因的残余间接费用一个**公开披露的单一基准**——并把这一切做到可解释、可对账、有测试。即"**有真动因的地方做 ABC，没有的地方做诚实的吸收法，永不静默**"。这最贴合你们的战略定调，也最快让财务敢采信。

### 4.4 建议次序（2-3 周可达"可信首版"）

1. **第 1 步（半天）**：写成本事实源说明 + 黄金用例（先红）——把 S4 的目标固化成可执行断言。
2. **第 2 步（3-5 天）**：重建成本池归集（修复①），删平均分摊，每中心真实动因量与费率。
3. **第 3 步（2-3 天）**：统一两套引擎（修复②），项目成本 = 切片聚合。
4. **第 4 步（1-2 天）**：切片下钻可解释 + 无法解释即异常（修复④）。
5. **第 5 步（1 天）**：黄金用例转绿 + 对账断言 + 接 CI（修复③闭环）。
6. **第 6 步**：跑一遍真实/拟真月度数据，与手工核算对比，确认 ≤5%，再对外宣称"精确"。

---

## 5. 给产品经理的最终判断

- **方向对，骨架真，但核心算错了。** 现在最不该做的是"再加成本页面"或"再写一版看板"——前端已经超前。
- **最该做的是让每个切片成本数字"算得对、说得清、测得过"**：重建成本池归集（去平均分摊）、统一成本入口、补黄金用例验收测试。
- 在引擎可信之前，**任何"精确成本核算 / 成本透明化"的对外宣称都应暂缓**——否则财务第一次抽查发现两个页面对同一项目给两个数、或切片成本无法解释，信任会一次性崩掉。
- 安全/审计/治理的 P2/P3（WORM、密钥轮换、审计覆盖矩阵）是"宣称替代线下台账"的门槛，**排在成本引擎之后**。
