# 配置驱动对账单导入器产品方案

## 1. 产品定位

COREONE 的对账单导入器不应只定位为“把单张 Excel 解析成病例收入”的工具，而应升级为“财务月结工作台 + 院级盈亏可信事实层”。

目标用户是第三方病理实验室的财务、运营和成本核算人员。每月他们需要处理来自 19 家以上医院、约 60 份、多种格式的结算单，并把结算收入、外送/合作分成、LIS 病例、ABC 成本和院级盈亏连接起来。

当前代码已经证明常规病例明细型结算单可以跑通，但真实月结还缺少四类关键能力：

- 非逐病例模板也能入库：科目汇总、纯外送、联合运营等不能停留在预览层。
- 宽表列语义能被识别和复用：同一行内不同金额列代表不同业务边界，必须保留来源列上下文。
- 财务有月结总控台：所有文件、识别结果、入库状态、异常、成本就绪状态需要按月份闭环。
- 规则变更可追溯、可重算：分类规则、扣率、外送判定变更后，要知道影响了哪些历史月份和报表。

## 2. 北极星目标与成功标准

北极星目标：财务可以在一个月结周期内完成所有医院结算单导入、分类确认、异常处理、院级盈亏生成和审计留痕，不再依赖线下 Excel 作为主账。

首版成功标准：

- 覆盖真实样本中的 7 类模板家族，每类至少有明确的“入库/登记/转外送台账”路径。
- 每个月、每家医院、每份文件都有状态：未导入、待分类、待复核、已入库、成本未就绪、已出盈亏。
- 常规病例明细模板继续保持可解释的病例级收入，合木佳黄金用例 `13,152` 不回退。
- 纯外送、远程会诊、联合运营、科目汇总类结算单不再被“无法逐 case 落库”阻断。
- 所有差异都能解释为：院内劳务收入、外送/合作结算、未匹配 LIS、成本未就绪、人工调整或规则变更影响。
- 规则变更后，系统能展示影响范围，并生成重算结果或调整单，而不是静默改写历史。

## 3. 产品原则

- 不强行把所有结算单塞进病例级模型。真实结算单既有逐病例明细，也有科目汇总、远程会诊、联合运营、纯外送台账。
- 原始文件不可变，解析结果可重算，人工判断可覆盖但必须留痕。
- `OUT` 不是实验室劳务收入，但它仍然是月结事实，必须进入财务闭环。
- “未知、未就绪、缺配置”不能按 `0` 处理。`0` 只能代表明确业务值，缺失要以质量标记暴露。
- 配置驱动不是让用户填配置表，而是通过样表、推荐规则、一键确认和影响预估降低配置成本。

## 4. 用户与核心场景

### 4.1 财务月结负责人

每月收集所有医院对账单，确认应收、扣率、外送分摊和院级盈亏。核心诉求是少返工、可对账、可解释。

### 4.2 运营/实施人员

负责新医院上线、模板识别、规则维护和异常处理。核心诉求是用样本训练规则，并知道规则是否会影响历史报表。

### 4.3 管理层/院级经营分析人员

查看医院维度收入、成本、毛利和外送占比。核心诉求是数据可信，并能追溯到结算单来源。

## 5. 产品架构

### 5.1 月结工作台

以“月份 + 医院”为主线展示所有导入文件和状态。

关键视图：

- 月份总览：医院数、文件数、已入库金额、待处理金额、成本未就绪金额、外送金额。
- 医院进度：每家医院的文件清单、模板类型、识别状态、分类状态、入库状态、P&L 状态。
- 异常队列：待人工归类、LIS 未匹配、成本缺失、扣率冲突、重复病例、规则版本冲突。
- 出口：院级盈亏包、差异解释表、外送台账、审计日志。

### 5.2 对账单事实层

把“导入文件”拆成多个事实层，避免所有模板都依赖 `case_no`。

建议事实对象：

- `statement_import_batches`：文件批次、医院、月份、模板、原始文件哈希、解析器版本、配置版本。
- `statement_case_revenue`：逐病例收入事实，承接现有 `case_revenue` / `case_revenue_lines` 能力。
- `statement_aggregate_revenue`：非逐病例收入事实，承接科目汇总、纯外送、远程会诊金额、联合运营汇总。
- `statement_out_ledger`：外送/合作结算台账，记录 NGS、FISH、HPV-E6E7、远程会诊、联合运营等不计入实验室劳务收入的金额。
- `statement_quality_flags`：缺成本、缺规则、缺 LIS、疑似重复、人工覆盖、历史规则重算等质量标记。

### 5.3 模板库与规则中心

模板库不只是解析脚本集合，而是“样本 + 列语义 + 分类规则 + 入库策略”的组合。

每个模板家族至少记录：

- 适用医院、文件特征、表头/工作表识别规则。
- 行级字段映射和列级语义映射。
- IN/OUT 分类建议及置信度。
- 入库策略：病例级、科目汇总、纯外送台账、联合运营事实表。
- 预览组件类型和验收口径。

### 5.4 版本、重算与审计

规则、扣率、模板映射、服务目录都要版本化。

关键能力：

- 发布前展示影响范围：受影响医院、月份、批次、金额、报表。
- 支持重算派生事实，不覆盖原始批次。
- 对已结账月份生成调整单或重算快照。
- 人工确认必须记录原因、操作人、时间、前后差异和审批状态。

## 6. 关键流程

### 6.1 新医院上线

1. 上传 1-3 份历史样表。
2. 系统自动识别模板家族、表头、金额列、病例号列、科目列。
3. 对无法确定的列给出候选含义，例如“远程会诊结算”“免组结算金额”“HPV-E6E7 外送”。
4. 实施人员确认列语义、IN/OUT 分类、扣率和入库策略。
5. 系统生成模板规则草稿，并用历史样表回放校验。
6. 通过后发布为该医院的有效规则版本。

### 6.2 月度批量导入

1. 财务选择月份并批量上传所有医院文件。
2. 系统按医院、模板、月份自动分组。
3. 每份文件进入预览和质量检查。
4. 可自动入库的文件进入待复核或直接入库。
5. 有歧义的文件进入异常队列，按金额影响排序。
6. 月结工作台汇总已入库、待处理、外送、成本未就绪和差异金额。

### 6.3 预览与分类

不同模板使用不同预览组件：

- 病例明细型：按病例、项目、数量、单价、扣率、IN/OUT、LIS 匹配展示。
- 宽表型：先把金额列展开为“来源列 + 金额 + 业务语义”，再归类。
- 科目汇总型：按科目/分类聚合展示，映射到 IN、OUT、调整或忽略。
- 纯外送型：作为 `labRevenue = 0`、`outSettle = 全额` 的外送事实入库。
- 联合运营型：进入联合运营事实，不与普通病例劳务收入混算。

### 6.4 确认与入库

确认动作必须从“确认布尔值”升级为“带原因的业务决定”。

确认时展示：

- 本次入库金额、IN 金额、OUT 金额、未匹配金额、成本缺失金额。
- 与系统推荐的差异。
- 影响的院级盈亏月份。
- 是否需要二次审批。
- 确认原因和附件。

### 6.5 规则变更与历史重算

用户修改扣率、列语义、服务分类、OUT 规则时：

1. 系统先做影响预估。
2. 展示受影响批次、金额和报表。
3. 用户选择只影响未来、重算未结账月份、或生成已结账月份调整单。
4. 重算结果形成新快照，保留旧结果。

## 7. 模板覆盖路线

| 模板家族 | 典型样本 | 当前问题 | 产品策略 | 优先级 |
| --- | --- | --- | --- | --- |
| 常规病例明细 | 合木佳 | 已可跑通，但要守住黄金值和跨院病例身份 | 保持病例级入库，补齐复合病例身份和回归用例 | P0 |
| 混合服务费 | 温州 | IN/OUT 混合，HPV 等外送项目目录不足 | 扩展服务目录，按项目归类进入病例收入或外送台账 | P0/P1 |
| 宽表远程会诊 | 平泉 | 金额语义藏在列名，行项目为空 | 展开来源列，记录 `sourceColumn/sourceLabel`，支持列级规则 | P1 |
| 宽表诊断费 | 宁波 | 类似宽表语义问题 | 复用列级语义引擎和规则推荐 | P1 |
| 纯外送明细 | 赣州 | 无病例号，现有病例级入库会失败 | 作为外送月度事实入库，`labRevenue = 0` | P1 |
| 科目汇总 | 东安 | 科目汇总能预览但不能进入院级盈亏 | 建聚合事实表，科目映射到 IN/OUT/调整 | P1 |
| 联合运营 | 石门 | 联合运营 P&L 不是普通病例收入 | 建联合运营事实和单独展示口径 | P2 |

## 8. 分阶段实施计划

### Phase 0：修复阻断月结可信度的问题

周期建议：1-2 周。

- 明确并修复跨医院重复 `case_no` 身份问题，至少在 LIS、ABC、P&L 链路不再单键依赖病例号。
- 修复新医院首次分类时 `expectedVersion = 0` 与后端种子版本冲突导致的 409。
- NGS/外送确认时，缺成本或缺价格不能以 `0` 静默入账，要生成质量标记。
- 历史配置、回滚配置也必须走归一化校验，不能只在 PUT 时生效。
- 保持合木佳黄金用例收入 `13,152` 不回退。

### Phase 1：真实月结可用 MVP

周期建议：2-4 周。

- 上线月结工作台，按月份和医院追踪所有文件状态。
- 建 `statement_aggregate_revenue` 和 `statement_out_ledger`，让科目汇总、纯外送、远程会诊进入财务闭环。
- 宽表解析保留来源列语义，支持列级分类规则和一键确认。
- 扩展默认外送目录，至少覆盖 NGS、FISH、HPV-E6E7、远程会诊、联合运营。
- 预览页按模板类型展示，不再用一段难读说明承载科目汇总/联合运营逻辑。
- 确认动作增加原因、影响范围和审计记录。

### Phase 2：治理、重算和可审计

周期建议：3-6 周。

- 规则变更前展示影响范围，支持未结账月份重算和已结账月份调整单。
- 前端提供黄金期望值和差异解释，不依赖人工记忆测试口径。
- LIS 未匹配提供恢复路径：候选匹配、手动绑定、忽略原因、后续补匹配。
- 配置和导入引入 maker-checker，区分录入、复核、发布。
- 生成月结导出包：院级盈亏、外送台账、未匹配清单、调整单、审计日志。

### Phase 3：规模化运营

周期建议：后续迭代。

- 批量文件自动识别和模板推荐。
- 样本库沉淀：按医院、月份、模板家族保留脱敏样本和验收结果。
- NGS/外送入口前端化，并与公司级外送台账或采购系统定义接口。
- 按医院输出数据完整性评分和月结风险评分。
- 接入通知与任务分派，让异常处理责任到人。

## 9. MVP 功能清单

P0 必须做：

- 月结工作台基础版：月份、医院、文件、状态、金额汇总。
- 常规病例明细回归保护：合木佳黄金用例和跨院病例身份策略。
- 新医院首次分类 409 修复。
- 外送确认缺成本/缺价格质量标记。
- 配置归一化覆盖历史和回滚链路。

P1 进入可用 MVP：

- 聚合事实层：支持科目汇总和纯外送入库。
- 宽表列级语义识别：`sourceColumn/sourceLabel` 入库并可配置。
- 默认外送服务目录补齐 HPV-E6E7、FISH、NGS、远程会诊。
- 模板化预览：病例明细、宽表、科目汇总、纯外送、联合运营分开展示。
- 确认原因、审计记录、差异摘要。

P2 增强信任：

- 规则影响预估与重算。
- LIS 匹配恢复。
- maker-checker。
- 月结导出包。
- 联合运营独立事实和展示。

## 10. 用户故事

- 作为财务，我希望上传一个月份的所有医院文件后，能看到哪些文件已入库、哪些还缺规则、哪些只影响外送台账，这样我能控制月结进度。
- 作为财务，我希望赣州这类纯外送单据能登记为外送结算，而不是因为没有病例号失败，这样我不需要在线下 Excel 维护第二套账。
- 作为实施人员，我希望平泉宽表里的“远程会诊结算”列被识别成独立业务语义，并且下月自动套用规则，这样我不用每月重复人工归类。
- 作为财务主管，我希望修改 HPV-E6E7 的 OUT 规则前看到影响哪些历史月份和金额，这样我能决定重算还是出调整单。
- 作为审计人员，我希望看到每次人工确认的原因、操作者、原始文件和规则版本，这样报表数字能追溯。

## 11. 验收标准

功能验收：

- 同一月份可同时导入病例明细、宽表、科目汇总、纯外送和联合运营样本，并在月结工作台看到统一状态。
- 赣州纯外送样本可形成外送事实，`labRevenue = 0`，`outSettle` 等于结算总额，且不阻断院级月结。
- 东安科目汇总样本可按科目映射 IN/OUT，并进入聚合事实层。
- 平泉/宁波宽表金额行可追溯到原始来源列，列级规则下月可复用。
- 人工确认必须填写原因，并在导入批次审计日志中可查。
- 缺成本、缺价格、缺 LIS 匹配均以质量标记呈现，不计作正常完成。

数据验收：

- 合木佳黄金用例保持收入 `13,152`。
- 每个导入批次都能追溯原始文件哈希、模板版本、规则版本和操作记录。
- 月结汇总金额可以拆解为 IN 劳务收入、OUT 外送/合作结算、调整、未匹配、成本未就绪。
- 规则变更后不会静默改写已结账月份。

体验验收：

- 用户看到“待人工归类”时，页面必须给出候选规则、影响金额和下一步操作。
- 科目汇总、纯外送、联合运营不再只显示说明文字，而是有对应的结构化预览表。
- 月结负责人无需进入多个隐藏入口，就能完成导入、异常处理、确认和导出。

## 12. 风险与待决策

- 复合病例身份策略需要产品和数据共同确认：是统一使用 `partner_id + case_no`，还是引入全局病例主键。
- OUT 台账的最终归属需要明确：是否只在 COREONE 内展示，还是要对接公司级采购/外送系统。
- 联合运营金额是否进入院级 P&L 主表，还是作为单独经营口径，需要财务定规则。
- 已结账月份的重算策略需要制度支持：允许重算、只出调整单，还是两者并存。
- maker-checker 权限边界需要定义：哪些金额、哪些配置变更必须复核。

## 13. 建议的下一步

1. 先把 Phase 0 作为阻断项清单合入工程排期，确保现有病例级链路可信。
2. 同时设计 `statement_aggregate_revenue`、`statement_out_ledger` 和月结工作台信息架构。
3. 选 3 个样本作为 Phase 1 验收包：东安科目汇总、赣州纯外送、平泉宽表远程会诊。
4. 用这 3 个样本打通“预览 -> 分类 -> 入库/登记 -> 月结汇总 -> 导出”的端到端流程。
5. 在 Phase 2 前补齐规则影响预估和审计模型，避免上线后历史数据不可控。
