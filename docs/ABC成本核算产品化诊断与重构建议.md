# ABC 成本核算产品化诊断与重构建议

> 日期：2026-06-15
> 主题：将现有 ABC 成本核算方案重构为可落地、可审计、可持续迭代的产品能力
> 结论：现有方案具备 ABC 概念和部分页面/接口雏形，但尚未满足“基于 ABC 成本法产品化”的要求

---

## 1. 一句话结论

当前 COREONE 的成本核算方向应该继续采用“收费锚点 + ABC 引擎 + 月度核算闭环”，但不能把现有 ABC 页面和局部实现视为产品完成。

现有方案的主要问题不是缺页面，而是缺一条可靠的成本核算业务闭环：成本对象、成本池归集、收费映射、病例级聚合、出库沉淀、异常补算、月末关账、审计导出都还没有形成稳定产品链路。

---

## 2. 诊断依据

### 2.1 早期成本分析仍是传统报表

`docs/FRS/FRS-14-成本分析.md` 对成本分析的定位是“多维度成本分析报表”，支持按项目、物料、供应商聚合，且明确是“只读报表，基于出库/入库记录聚合计算”。

这套能力可以支持耗材成本监控，但不等于 ABC 成本法产品化。它缺少作业中心、成本动因、病例/切片成本对象、成本池期间归集、分摊确认等 ABC 关键机制。

### 2.2 市场调研支持混合模式，而不是纯 ABC 理想化

`.claude/research/abc-market-research.md` 的核心判断是：主流 LIS/ERP 更倾向收费驱动，深度 ABC 的产品占比极少，难点包括数据采集困难、实施复杂度高、用户接受度低。

该调研同时指出 COREONE 的差异化优势是进销存天然拥有精确材料出库和 BOM 用量数据。因此更合理的产品路线不是“纯 ABC”，而是：

```text
以收费为锚点，以 ABC 为引擎
```

也就是说，ABC 应作为成本计算和解释引擎，最终面向用户输出的仍应是“收费 vs 成本”“项目是否盈利”“成本为何波动”“哪里可以优化”。

### 2.3 现有方案内部存在互相矛盾的结论

`plans/abc-final-review.md` 和 `plans/abc-final-verification-v3.md` 倾向认为方案已经覆盖完整；但 `plans/abc-product-purpose-verification.md` 明确指出当时方案不能实现“成本透明化”，原因包括：

| 问题 | 产品影响 |
|---|---|
| 核心业务链路断裂 | 用户看不到可信成本数据 |
| BOM-收费标准映射未验证 | 收费金额无法可靠计算 |
| 阶梯定价聚合未解决 | 盈利分析会出现错误 |
| 出库流程缺少 ABC 成本写入 | 成本数据无法沉淀 |
| 核心计算函数缺少测试 | 计算正确性不可证明 |

后续 worktree 中补了一部分实现，但这些产品化问题只被部分修复，没有形成完整核算闭环。

2026-06-16 复核：以上是本轮产品化实施前的历史判断。当前 ABC 专用分支已经补齐收费映射、病例聚合、出库成本写入、导出、核心回归和 E2E，旧判断不再代表当前实现状态。

### 2.4 当前代码状态存在实现不一致

当前主目录 `后端代码/server/src/app.ts` 已 import `./routes/abc-v1.1.js`，但当前主目录未发现对应的 `后端代码/server/src/routes/abc-v1.1.ts` 和 `后端代码/server/src/utils/cost-calculator.ts`。完整实现存在于 `.worktrees/master-aligned-integration-2026-06-15`。

这说明当前仓库状态存在“文档、页面、接口封装、实际后端文件不一致”的风险。产品化工作开始前，必须先统一目标分支和实现来源。

### 2.5 worktree 实现仍偏“数据看板雏形”

在 `.worktrees/master-aligned-integration-2026-06-15` 中，ABC 后端已经有 `outbound_abc_details` 写入和 `/abc/dashboard` 等接口，但当时仍有明显缺口：

| 代码现象 | 产品判断 |
|---|---|
| 出库时 ABC 计算失败只记录 console，出库继续成功 | 库存流程合理，但成本核算必须生成异常台账并支持补算 |
| Dashboard 的 `costChange`、`feeChange`、`profitChange` 固定为 0 | 环比指标未真实实现 |
| `costByActivity` 和 `alerts` 返回空数组 | 作业成本结构和异常提醒没有落地 |
| `/abc/export` 返回 `{ url: null }` | 导出是空壳 |
| `/abc/batch-trace/:batchId` 返回空列表 | 批次追溯未落地 |
| `/abc/variance-analysis` 返回空结构 | 差异分析未落地 |
| `cost-pools/sync/auto-collect/recalculate` 只返回 OK | 成本池归集和重算流程未实现 |

2026-06-16 复核状态：

| 历史缺口 | 当前状态 |
|---|---:|
| ABC 失败只写 console | 已写入 `cost_exceptions`，并支持 resolve/ignore/retry |
| Dashboard 环比固定为 0 | 已按当前月和上月 ABC 快照计算 |
| `costByActivity` / `alerts` 为空 | 已接入成本池和开放异常 |
| `/abc/export` 空壳 | 已返回 CSV content、汇总、明细和审计日志 |
| `/abc/batch-trace/:batchId` 空壳 | 已返回批次、消耗、ABC 快照和库存流水 |
| `/abc/variance-analysis` 空壳 | 已返回项目/月/BOM 维度差异 |
| 成本池动作只返回 OK | 已接入同步来源、自动归集、动因费率和重算任务 |

## 3. 产品化目标

### 3.1 产品定位

COREONE 的 ABC 成本核算不应定位为“医院全成本核算系统”，而应定位为：

```text
病理科耗材与作业成本精细化核算工具
```

它的强项是把进销存数据、BOM 标准用量、出库批次成本、收费标准和作业成本结合起来，让科室管理者和财务人员回答三个问题：

| 用户问题 | 产品答案 |
|---|---|
| 这个项目赚钱吗？ | 收费、材料成本、作业成本、利润率 |
| 钱花在哪里？ | 材料、作业中心、人工/设备/间接费用拆解 |
| 数据可信吗？ | 成本期间、核算批次、异常台账、重算记录、导出报表 |

### 3.2 目标闭环

目标产品链路应是：

```text
基础配置
→ 出库/业务发生
→ 成本实时预估与沉淀
→ 月度成本池归集
→ 费用分摊与重算
→ 财务确认/关账
→ 报表分析/导出
→ 异常修正/审计追踪
```

这条链路必须能解释“某一张切片/某一类项目/某个月的成本是怎么来的”，并能在数据变化后按规则重算。

## 4. 核心产品缺口

### 4.1 缺口一：成本对象没有被产品化

现有材料同时提到了项目、BOM、出库单、切片、病例、月份，但没有形成统一的成本对象层级。

建议定义如下层级：

| 层级 | 用途 | 是否 MVP 必须 |
|---|---|---|
| 成本期间 | 月度归集、关账、报表口径 | 是 |
| 出库单成本 | 实时材料成本和 ABC 初算 | 是 |
| BOM 成本 | 标准切片成本、收费映射 | 是 |
| 项目成本 | 管理报表和盈利分析 | 是 |
| 病例成本 | 阶梯收费、临床真实收费口径 | 是 |
| 单张切片成本 | 成本透明化和作业拆解 | 是 |
| 批次成本 | 追溯材料来源和价格波动 | P1 |

### 4.2 缺口二：收费映射按 BOM 过于简单

现实收费往往不是“一个 BOM 对一个收费编码”。可能出现：

| 场景 | 例子 | 风险 |
|---|---|---|
| 一个病例包含多个 BOM | HE + IHC + 特染 | 阶梯/封顶应按病例聚合 |
| 一个 BOM 对多个收费项目 | 标本处理费 + 染色费 + 诊断费 | 单字段 fee_standard_id 不够 |
| 收费规则有阶梯/封顶 | 前 N 张、超出加收、封顶 | 按 BOM 单独算会错 |
| 收费和成本对象不一致 | 收费按病例，成本按切片 | 盈利分析口径错位 |

建议新增“收费映射规则”概念，而不是仅在 BOM 上挂一个 `fee_standard_id`。

### 4.3 缺口三：成本池没有归集、审核、重算流程

ABC 的核心不是表里有 `abc_cost_pools`，而是每个月能回答：

1. 这个作业中心本月归集了哪些费用？
2. 每类费用来源是什么？人工、设备、间接、手工调整？
3. 分摊动因总量是多少？
4. 动因费率由谁确认？
5. 费率变化后历史出库成本是否重算？
6. 月份关账后是否禁止随意改动？

目前 `sync/auto-collect/recalculate` 还没有真实业务动作，因此不能支撑产品化核算。

### 4.4 缺口四：异常没有产品入口

出库时 ABC 计算失败不应只写 console。对仓库操作来说，出库可以继续；对成本核算来说，必须记录异常：

| 异常 | 应进入异常台账 |
|---|---|
| BOM 未配置作业关联 | 是 |
| BOM 未配置收费映射 | 是 |
| 成本池无当月费率 | 是 |
| 阶梯收费无法匹配病例 | 是 |
| 出库成本与重算成本不一致 | 是 |
| 缺少批次价格或材料价格 | 是 |

异常台账应支持过滤、处理、补算、忽略、导出和审计。

### 4.5 缺口五：缺少月末关账

成本核算不是无限实时变化的看板。财务需要期间口径：

| 状态 | 含义 |
|---|---|
| open | 当月业务仍在发生，可实时更新 |
| collecting | 开始归集成本池，检查缺口 |
| calculated | 已完成本月分摊和重算 |
| reviewed | 财务/主任已复核 |
| closed | 已关账，除调整单外不可改 |
| adjusted | 关账后有调整，形成补充记录 |

没有关账，导出的报表无法成为可信核算结果。

### 4.6 缺口六：前端呈现未达到“成本透明化”

现有页面有看板、切片成本、盈利分析等入口，但部分能力仍是显示层：

| 页面 | 现状问题 |
|---|---|
| 成本看板 | 指标有，环比/作业结构/预警不完整 |
| 切片成本 | 展开仅粗拆物料/作业，未展示真实作业中心明细 |
| 盈利分析 | 以表格为主，缺病例/BOM/项目三维切换闭环 |
| 导出按钮 | 多处按钮没有真实导出动作 |
| 模型验证 | 前端手算，不能替代后端核算引擎 |

## 5. 推荐目标架构

### 5.1 产品模块

| 模块 | 目标 |
|---|---|
| 核算期间管理 | 建立月度核算状态、关账、调整入口 |
| 成本池归集 | 按作业中心归集人工、设备、间接和手工费用 |
| 收费映射规则 | 支持 BOM/项目/病例级收费映射和阶梯规则 |
| 出库成本沉淀 | 出库时生成成本快照和异常记录 |
| 病例收费聚合 | 按病例聚合多个 BOM 的收费，处理阶梯/封顶 |
| 成本重算引擎 | 成本池变化后可重算未关账期间 |
| 异常处理中心 | 缺映射、缺费率、计算失败、价格缺失等处理 |
| 分析报表 | 看板、切片、项目、病例、作业中心、差异、导出 |
| 审计追踪 | 配置变化、重算、关账、导出、异常处理均留痕 |

### 5.2 概念数据对象

| 对象 | 说明 |
|---|---|
| `cost_periods` | 核算期间，记录月份和状态 |
| `cost_runs` | 一次核算/重算任务，记录范围、状态、操作者 |
| `cost_run_items` | 核算结果明细，关联出库/BOM/项目/病例 |
| `cost_exceptions` | 成本计算异常台账 |
| `cost_pool_entries` | 成本池费用来源明细 |
| `activity_driver_rates` | 每个作业中心每月动因费率快照 |
| `fee_mapping_rules` | 收费映射规则，支持一对多和条件匹配 |
| `case_charge_groups` | 病例级收费聚合结果 |
| `cost_snapshots` | 出库/切片/BOM/病例成本快照 |
| `abc_cost_adjustments` | 关账后调整单；与季度间接成本调整 `cost_adjustments` 区分 |
| `abc_audit_logs` | 关键操作审计记录 |

说明：以上是产品对象，不要求一次性全部落表，但 MVP 必须至少覆盖核算期间、异常、收费映射、成本快照和重算任务。

## 6. MVP 范围建议

### 6.1 MVP 不做什么

MVP 暂不追求完整医院全成本核算，也不优先做预测、复杂质量成本、人员效率排行榜、供应商成本优化建议。

这些能力可以保留在后续版本，避免当前 ABC 产品继续膨胀。

### 6.2 MVP 必须做什么

| 优先级 | 能力 | 验收口径 |
|---|---|---|
| P0 | 统一当前实现来源 | 主目录能构建，ABC 路由和计算引擎文件存在 |
| P0 | 核算期间 | 可创建/查看月度期间，状态可流转 |
| P0 | BOM 收费映射规则 | 支持一个 BOM 对多个收费项，支持规则预览 |
| P0 | 病例级收费聚合 | 同一病例多个 BOM 能统一计算阶梯/封顶 |
| P0 | 出库成本快照 | BOM 出库后必有成功成本快照或异常记录 |
| P0 | 成本异常台账 | 缺映射/缺费率/计算失败可被发现和补算 |
| P0 | 成本池归集 | 当月作业中心费用可录入、查看、生成费率 |
| P0 | 重算任务 | 费率或映射变化后能重算未关账期间 |
| P0 | 成本看板可信化 | 环比、作业结构、异常数来自真实数据 |
| P0 | 导出 | 导出当前期间成本明细和汇总表 |

### 6.3 MVP 页面清单

| 页面 | 用户 | 核心动作 |
|---|---|---|
| 成本核算工作台 | 财务/主任 | 选择月份、查看状态、启动核算、关账 |
| 成本池归集 | 财务 | 录入/同步费用、查看动因量、生成费率 |
| 收费映射配置 | 技术/财务 | 配置 BOM/项目到收费项的规则 |
| 成本异常中心 | 财务/技术 | 查看异常、修复配置、触发补算 |
| 出库成本预览 | 仓管 | 出库前查看预估成本、收费、利润和缺口 |
| 切片成本明细 | 主任/财务 | 按 BOM/项目/病例下钻作业成本 |
| 盈利分析 | 主任/财务 | 项目、病例、BOM 三维切换 |
| 导出中心 | 财务 | 导出月度核算表、异常表、明细表 |

## 7. 目标流程设计

### 7.1 配置阶段

```text
1. 技术人员维护 BOM 和作业关联
2. 财务/技术维护收费标准和映射规则
3. 财务维护成本池来源和归集规则
4. 系统检查配置完整性
5. 缺失项进入异常中心
```

配置阶段的产品目标不是“能填表”，而是能判断“这个配置能否支撑核算”。

### 7.2 出库阶段

```text
1. 仓管选择项目、BOM、样本/切片数量
2. 系统预估材料成本、作业成本、收费、利润
3. 若配置缺失，提示但允许按权限继续
4. 出库成功后写入成本快照
5. 若 ABC 计算失败，写入成本异常台账
```

关键原则：库存动作和成本动作可解耦，但成本失败不能静默。

### 7.3 月末核算阶段

```text
1. 财务打开本月核算期间
2. 系统汇总本月出库、病例、BOM、异常
3. 财务归集作业中心成本池
4. 系统生成动因费率
5. 系统重算本月成本快照
6. 财务检查异常和差异
7. 主任/财务复核
8. 关账并生成报表
```

这是 ABC 产品化最关键的闭环。

### 7.4 关账后调整

```text
1. 发现收费标准、材料价格或作业成本错误
2. 创建调整单
3. 系统仅对允许范围补算
4. 记录调整前后差异
5. 导出补充报表
```

2026-06-16 首版已落地：已关账期间继续禁止直接重算，但可创建关账后调整单；审核通过后进入成本看板调整额、调整后利润和 ABC 导出，创建/审核均写审计日志。

## 8. 关键产品规则

### 8.1 成本失败规则

| 场景 | 出库是否成功 | 成本状态 |
|---|---:|---|
| 材料库存不足 | 否 | 不生成成本记录 |
| 材料出库成功，ABC 计算成功 | 是 | `costed` |
| 材料出库成功，缺收费映射 | 是 | `costed_with_warning` |
| 材料出库成功，缺作业费率 | 是 | `pending_cost` |
| 材料出库成功，计算异常 | 是 | `cost_exception` |

### 8.2 重算规则

| 期间状态 | 是否允许重算 | 说明 |
|---|---:|---|
| open | 是 | 业务仍在发生 |
| collecting | 是 | 成本池归集中 |
| calculated | 是 | 复核前可重算 |
| reviewed | 需权限 | 需记录原因 |
| closed | 否 | 必须走调整单 |

### 8.3 收费聚合规则

收费应优先按病例聚合，其次按项目/出库单聚合，不能简单按 BOM 单独计算后求和。

建议规则顺序：

1. 如果存在病例号，按病例聚合所有相关 BOM。
2. 如果无病例号但有关联项目和同批出库单，按项目出库批次聚合。
3. 如果仅有 BOM 出库，则按 BOM 作为临时成本对象，但标记为非最终收费口径。

## 9. 分期路线

### Phase 0：基线统一

目标：消除“文档说有、当前目录没有”的状态。

交付：

| 任务 | 验收 |
|---|---|
| 确认 ABC 实现来源 | 明确以主目录还是 worktree 为准 |

## 13. 总结

现有 ABC 方案不是完全推倒重来。它已经有好的方向：出库数据、BOM 用量、收费标准、作业中心、盈利分析入口都已经出现。

截至 2026-06-16，本诊断中的 Phase 0/1 核算底座已经完成首版落地，Phase 2 的 BOM 多收费项映射、规则预览、病例级阶梯/封顶聚合、映射完整性检查、前端配置入口、病例号前端采集、BOM 出库来源快照、BOM 版本表与版本差异 UI、生效范围选择、历史影响统计、未关账追溯自动重算、病例聚合取消回退、非最新历史单剩余 ABC 明细重排、对账差异异常化、对账页面触发入口、真实 CSV 导出、关账后调整单和 LIS 病例进入标准出库流程也已完成首版。

但要成为真正产品，下一步不能继续“补更多页面”，而要先补“核算底座”：

```text
期间 → 成本池 → 动因费率 → 成本快照 → 异常补算 → 关账 → 报表导出 → 审计追踪
```

只有这条链路成立，ABC 才能从“看起来像成本分析”变成“可以被财务和主任信任的成本核算产品”。
