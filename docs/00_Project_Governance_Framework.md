# COREONE 项目治理框架

> **版本**: 1.0.0
> **创建日期**: 2026-06-11
> **更新日期**: 2026-06-11
> **适用范围**: `docs/` 正式治理文档、PRD/FRS/测试场景、待办、Bug、变更、决策、发布门禁
> **维护者**: PM + Codex + VS Code Claude Code CLI/mimo

---

## 0. 框架目的

本框架是 COREONE 后续治理的总入口，解决三个问题：

1. 当前项目文档来源分散，容易把历史设计、AI 生成结论和当前代码事实混用。
2. 多 Agent/多工作台并行时，需要统一任务入口、证据格式和关闭标准。
3. 发布前需要把需求、代码、测试、Bug、权限和回滚方案放到同一套门禁里判断。

治理目标不是替代代码，也不是替代 PM 决策，而是让每次修改都能回答：

- 改了什么？
- 依据是什么？
- 谁复核过？
- 如何验收？
- 风险是否关闭？

## 1. 事实源优先级

当文档之间冲突时，按以下顺序判断可信度：

| 优先级 | 事实源 | 使用方式 |
|---:|---|---|
| 1 | 当前代码、路由注册、类型定义、数据库初始化、E2E/单测实际文件 | 判断系统当前真实行为 |
| 2 | 最新工作日志、CI 结果、手工验证记录、执行回执 | 判断最近变更和验证状态 |
| 3 | `docs/` 正式治理文档 | 判断已确认的产品范围、规则、验收和发布门禁 |
| 4 | `docs/FRS/`、`docs/TestScenarios/` | 作为模块规格和测试设计输入，需定期与代码核对 |
| 5 | `V1.1设计稿/`、归档目录、旧 PRD/TECH-SPEC/DATABASE-DESIGN | 仅作历史参考，不得直接当成当前事实 |

硬规则：

- 代码与文档冲突时，以当前代码为准，同时在对应文档记录差异。
- AI 生成内容必须保留“依据来源”和“待确认项”。
- 未运行测试的内容不能写成“已通过”。
- 未在当前工作树看到的文件、接口、页面，不能写成“已存在”。

## 3. 状态口径

所有治理文档统一使用以下状态，避免“已完成”含义不清：

| 状态 | 含义 | 关闭条件 |
|---|---|---|
| 已核验 | 已对照当前代码/测试/执行证据，且结论可复现 | 写明证据路径或命令 |
| 已实现待复核 | 当前代码中看到实现，但测试或人工验收未完成 | 补测试或补验收记录 |
| 设计存在 | 文档/设计中存在，但当前代码未确认 | 需要代码核对 |
| 缺口 | 当前代码或文档链路缺失 | 进入 `09_Task_Backlog.md` 或 `11_Bug_Log.md` |
| 待 PM 确认 | 需要产品/业务决策 | PM 给出明确结论 |
| 已归档 | 历史资料，仅作参考 | 不再作为当前事实源 |

禁止口径：

- “100% 完成”但没有测试报告或验收记录。
- “全部通过”但没有本次运行命令、日期、结果。
- “后端已有”但当前路由文件不存在或 `app.ts` 无法编译。

## 4. 双工作台分工

| 角色 | 主责 | 交付要求 |
|---|---|---|
| PM | 业务决策、优先级、验收、发布批准 | 对待确认项给出结论 |
| Codex | 治理、规划、复核、小范围定向修复 | 提供任务包、审查结论、文档更新和风险说明 |
| VS Code Claude Code CLI + mimo | 主要代码执行、批量修复、测试运行 | 回传改动文件、测试命令、结果、未完成项 |

任务关闭必须同时满足：

1. 代码或文档已合入目标分支。
2. 相关测试或人工验证完成。
3. `09/10/11/13` 中对应任务、变更、Bug 或决策已更新。
4. 如涉及发布，`12_Release_Checklist.md` 已同步。

---

## 5. 任务包格式

交给执行工作台的任务必须包含：

| 字段 | 要求 |
|---|---|
| 背景 | 为什么要做，关联哪个需求/缺陷/决策 |
| 允许修改文件 | 明确可改范围 |
| 禁止修改文件 | 明确不可碰范围，尤其是并行会话边界 |
| 目标行为 | PM 能理解的最终效果 |
| 技术约束 | API、数据、权限、测试、兼容要求 |
| 验收方式 | 自动测试命令 + PM 手工验收步骤 |
| 回传格式 | 修改文件、测试结果、风险、未完成项 |

## 6. 发布治理原则

发布前必须确认：

- P0 Bug 为 0，或 PM 明确批准带风险发布。
- 核心 E2E、后端测试、前端测试、构建至少完成本次发布范围内的验证。
- 权限矩阵与前端菜单、后端路由、角色测试没有未解释差异。
- 数据库变更有备份和回滚方案。
- `.env`、密钥、Token、生产凭证不进入 Git。
- 发布后有健康检查和回滚负责人。

## 7. 本轮文档审查发现

2026-06-11 审查 Claude Code CLI + mimo 生成的治理文档时，发现以下需要修正的共性问题：

| 问题 | 影响 | 处理原则 |
|---|---|---|
| 多份文档引用本文件，但本文件缺失 | 治理入口断链 | 补建 `00_Project_Governance_Framework.md` |
| 部分文档把未复核内容写成“已完成/100%” | 误导排期和发布判断 | 改为“已实现待复核/缺口/待确认” |
| 项目现状报告中的路由、E2E、CI 数量与当前工作树不一致 | 新 Agent 接管时会误判系统规模 | 用当前文件系统事实修正，并标注工作树异常 |
| FRS README 指向不存在的 FRS-00/FRS-13 | 文档导航断链 | 标注缺失并纳入待补 |
| 发布清单引用当前不存在的 supplier-returns spec | 发布命令不可执行 | 改为以当前存在的核心 spec 为准 |
