## 0. 一句话 + 贯穿原则

**贯穿原则（本轮全部决策的公约数）= 粗放起步 · 轻量引导 · 增量精细**。这是 [[adoption-first-design]] 最高否决权的落地：**实验室现在这块几乎零记录，系统的价值是引导他们「从 0 到有」，而不是「从粗到精」。功能一重、逼他们从粗放突跳精细，直接导致弃用。**

一句话结论：
1. 基础数据现在是**假种子**——讨论的是**结构和口径**，不是数字。
2. **收入侧（对账单里的真实收费代码）是唯一真锚**，成本侧向它靠。
3. 但收入码≠直接接 BOM——**桥接锚点在"物理工序单位（蜡块/免疫组化片/特染片）"**，不在"收费码"。老"制片+诊断"混合码要**先拆出制片份额**再接 BOM。
4. BOM 用量结构按**行业标准**建（可配参数 + 默认典型值），实验室只填自己的用量数字。
5. 消耗/对账/盘点/采购一律**轻量引导**，不做重功能；一年一盘、采购不做财务账、设备工时无数据只能估。

## 3. T1 落地 —— 单一真锚 + 老混合码桥接（回答 PM 的挑战）

### 3.1 决策与挑战
- **决策**：成本侧以"对账单真实收费码"为锚，收费码指向 BOM 物料耗材（最贴合真实场景）。
- **PM 挑战**：老收费码里有大量**"制片+诊断"混合码**（国标 2703 系列：手术标本 270300005 / 内镜活检 270300002 / 术中快速冰冻）——一个码同时含①实验室亲手做的制片工序（取材/脱水/包埋/切片/HE，**这部分才耗料、才有 BOM**）+②病理医生判读/诊断/报告（**零耗材、无 BOM**）。混合码怎么"指向 BOM"？

### 3.2 关键洞察：桥接锚点从"收费码"下移到"物理工序单位"（复核 CONFIRMED）
混合码不直接接 BOM。**中转键 = 病例的物理工序计量（LIS 的蜡块数 / 免疫组化片数 / 特染片数）**。

```
                              ┌─── 收入侧：拆分工作量 (workload) ───┐
   对账单混合收费码 ──拆分──►  蜡块数 / 免疫组化片 / 特染片  ◄──同一物理计量──  BOM (每单位耗材)
                              └─── 成本侧：ABC 动因 (cost driver) ──┘
                                    ↑ 天然公共键（已有代码实证：
                                      block_count 两侧共用）
```

为什么比"收费码当键"更稳（有代码实证）：收费码是旧码/院定制码、code-agnostic 不解析语义、前导零 14/15 位混用+后缀字母、且 LIS 与 COREONE 各自内嵌同套规则→调价要双系统同改会漂移。而物理工序单位是**既驱动成本又驱动收入**的同一个真实量。

### 3.3 老混合码"四步桥"（复用已落地的 G1 拆分，不重造）

G1 拆分公式（逐字，来源 `COREONE-G1收入口径-…-2026-06-30.md`）：
> 组织制片份额 = `36 × LIS蜡块数 / (36 × LIS蜡块数 + 105)`，**逐病例**（对账单 × LIS 按病理号 join）。费率：组织 ¥36/标本、细胞 ¥75/玻片、冰冻按 ¥36 近似；诊断费 ¥105/病例。

1. **逐病例拆混合码** → 剥出制片份额 = 实验室收入；`1−制片份额` = 诊断份额 → 诊断桶（不进实验室盈亏）。免疫组化/特染（2705）是"整条计入实验室"，判读已含在诊断费里、**不拆不重复计**。
2. **用同一批 LIS 物理计量做公共键接 BOM**：码→该病例蜡块/免疫组化/特染数→BOM 每单位耗材。BOM 挂在物理工序单位上，不挂在收费码上。
3. **只有"制片份额有值"的物理单位归集 BOM 成本**，诊断桶零 BOM（诊断环节无耗材）。毛利 = 制片+染色收入 − 物理单位×BOM单位成本，两侧口径都只含制片/染色，干净闭环。
4. **双路径 + 降级诚实标注**：①有 LIS→真蜡块/真片数（精算）；②无 LIS（本院 TCT/冰冻）→降级用账单数量、标 `partial_quantities`；③账单上线新拆分 81 码（处理费/诊断费已分开）→直接用，无需拆。**绝不用精算路径反推翻估算路径的覆盖广度。**

### 3.4 守恒红线 + golden
守恒：`labRevenue + diagnosisSettle + outSettle + unmatched + ambiguous == 对账单总结算`。
锚点（和睦家全月）：对账单总结算 **¥55,541** → 纯实验室 **¥27,870**（golden，占 50.2%）+ 诊断桶 ¥27,671。旧口径把诊断也当实验室收入 → **高估约 2 倍**。LIS 精算覆盖组织制片 ¥13,079；TCT ¥3,106、冰冻 ¥37 仍为账单数量估。

### 3.5 风险清单（复核确认，含承重墙）
- **⚠️ 承重墙风险（最脆）**：把国标 36/105 比例套到和睦家溢价单价（组织 ¥165–470 vs 国标 ¥141）**本质是政策分摊，不是"证明康湾真实制片价值就是这个比例"**。溢价若来自双语/外籍/服务→高估制片；若来自更复杂标本→低估。**必须配敏感性区间（诊断锚 105→本地协商值），不当事实售价证明。**
- 双计防护：每条账单行只归唯一去向；诊断份额落诊断桶不领 BOM 成本；2705 染色不拆。守恒 golden 强制。
- 漏计提醒：诊断桶**不是零成本**（现场冰冻仍有人工/差旅），只是"非实验室技术工序成本"；日后做服务利润分析需另设桶。
- join 键完整性：**"病理号 100% 匹配" ≠ "每条业务线工作量字段都可用"**（真实 LIS 导出仅 6 基础数量列，分子/冰冻/多重染色列不存在）；两者分开看，别误以为全线精算。

### 3.6 前置会未尽、写码前要定的口径（真数据裁决）
- 拿**几条真实老混合码 + 对应 LIS 病例**手核：制片份额 vs BOM 归集成本，两侧对不对得上。
- 敏感性区间的诊断锚 105 用不用本地协商值。

## 4. T2 落地 —— 标准耗材用量结构（调研 CONFIRMED）

### 4.1 核心发现：扁平"每样本用量"装不下真实工艺
调研三类检测，结论一致：一个只有 `usage_per_sample`（每样本一个标量）的 BOM，会**系统性算错**单片成本。

### 4.2 扁平标量漏掉的四类维度（复核建议整合为 4 类，避免为复杂而复杂）

| 维度 | 是什么 | 漏了会怎样 |
|---|---|---|
| **① 计量粒度层级** | 每例 / 每蜡块 / 每片 / 每步——同一检测跨粒度（固定液按例、包埋盒按块、载玻片按片、特染按步） | 成本无法正确归集到"片"；一例→多块→多片的 1 对多展开表达不了 |
| **② 批次摊销与损耗** | 一缸/一次 run 的试剂由 N 张片共用（1÷N）；开瓶后效期到期报废；刀片/液缸按寿命摊 | 批量小时严重低估、大时高估；到期整缸弃置的损耗全漏 |
| **③ 母液→工作液两级** | 浓缩试剂按稀释倍数配工作液（浓缩 vs 即用型 RTU 分叉）；单片实际原液=工作液量÷稀释倍数 | 同一抗体不同稀释倍数单片成本差数倍，扁平模型完全表达不了 |
| **④ 质控/对照/重做分摊** | 每 run 每抗体强制带阳性对照片（CAP 强制），与患者样本数**解耦** | 只算患者片，漏掉"为质控而消耗"，低估耗材与工时 |

### 4.3 特染的特殊性：项目 = 有序多步 recipe
特染不是"一行用量"，而是**一串按固定顺序、各带时间/温度的反应步**（六胺银 GMS ~12 步、网状纤维 ~7 步、Masson ~6 步、PAS ~4 步、抗酸 3 步）；银染类试剂"临用现配、当批用完即弃"。BOM 必须能挂"项目下第 N 步用什么试剂"。PAS-D 还需消化+未消化两片 → 一个检测项目=多张玻片。

### 4.4 建议的 BOM 结构升级方向（实验室只填数字，结构行业统一）
- 检测项目 → **有序步骤序列**（每步：试剂 + 计量口径 + 典型时间/温度）
- 每行耗材带**计量口径枚举**：`per_case / per_block / per_slide / per_step / per_run_batch / per_dilution`
- 试剂支持**母液→工作液两级**（携带稀释倍数系数；剂型分叉：浓缩/RTU）
- **批次维度**：单 run 片数（batch size）+ 现配工作液"一次配制服务几张片"
- **损耗参数**：开瓶效期/整缸更换周期/耐用件寿命（刀片、液缸）
- **质控计次**：按 run/按抗体独立计数，随批分摊进患者单片
- 现状唯一有分摊语义的是质控品表（`bom_quality_controls` 有 `usage_per_batch/covers_samples`）——**可作为升级的样板**推广到通用试剂。

### 4.5 落地策略（粗放起步，呼应贯穿原则）
**先把这些维度做成"可配参数 + 行业默认典型值"，不一上来全精细。** 调研数值都是"典型区间"非唯一标准（FISH 每片 5–30µL 因平台而异、每缸可染片数强依赖仪器 SOP）→ **按院/按平台可标定**，别写死常数。实验室先用默认值跑起来，再增量校准（接 [[coreone-incremental-correction-architecture]]）。

### 4.6 现状 gap（PARTIALLY_CONFIRMED，带 file:line）
- `bom_items` 是标量结构：仅 `usage_per_sample + unit + is_alternative + main_item_id + sort_order`（`DatabaseManager.ts:205`），无稀释/多步骤/批次字段。
- BOM 材料标准成本 = Σ(加权均价 × usage_per_sample)（`bom-v1.1.ts:746-762`），默认样本数=1。BOM 已版本化（`boms UNIQUE(code,version)` + `bom_versions` 快照）。
- ~~**⚠️ 潜在 bug**：`updateBomStandardCost` 无 `is_alternative` 过滤 → 主料/替代料 usage 都累加~~ **【已复核=误报】**：`is_alternative=1` 是**辅料（都要用）**非二选一替代料，成本累加**正确**；且该函数仅 codex 线有、master(178 行)无。见 [[coreone-bom-is-alternative-misnomer]]。

## 5. T3/T4/T5/T6 —— 进销存流程：轻量引导，粗放起步

### 5.1 设计铁律
**从 0 到有 > 从粗到精。功能越轻越好。** 突然逼实验室做精细记录 = 弃用。系统的角色是"温和地把他们从'什么都不记'带到'记个大概'"。

### 5.2 消耗记录（T3）
- 现状（核实）：两层——真实库存扣减在**仓库出库**（扣 `inventory.stock`+`batches.remaining`+写 `stock_logs`）；台面领用 `batch_usage_tracking` 是出库的**自动派生**台账（"领了一瓶用多久"的宏观进度，`expected_days` 写死 30），耗尽确认只改 tracking 状态、**不回写仓库批次**（守恒不变量，`depletion-v1.1.ts:176-186`）。
- 决策方向：**以仓库出库为主消耗记录**；台面领用/耗尽保持"宏观进度"这种轻量形态，**不逼逐次精确消耗**。引导技术员"领了→大致用完打个卡"即可。

### 5.3 对账（T4）
- 现状（核实）：理论用量=病例数×BOM `usage_per_sample`，实际=归属项目且已完成的出库汇总；**刻意排除无项目的出库**以对等口径；差异只出报表不自动扣账（阈值 warn=|diff|>20%、danger=diff>+50%）。
- 决策方向：**先做"消耗趋势 / 大差异提示"，不做天天精确对账标红**（标红没人信=弃用）。且对账可信度**依赖 T2 用量标准可信**——T2 不落地，对账差异都是噪声。接增量纠错架构，把差异当"纠错信号"而非"考核"。

### 5.4 盘点（T5）
- 现状（核实）：支持单条"一物一单"+批量（共享 `sheet_no`，上限 500 条），但**无周期性全盘自动铺单/快照编排**；系统量在确认时**服务端实时读取**（无需冻结快照）。
- 决策方向：**一年一盘，轻量**。做"轻引导全盘"（一次铺单 + 移动端逐项点确认差异），**不逼频繁循环盘点**。

### 5.5 采购 / 退款（T6）
- 现状（核实）：供应商退款无财务闭环（无应付台账）。
- 决策方向：**不做 AP 财务账实源**，只记实物收发；钱交公司级财务系统（同"边界移出、否则越做越大"原则）。

## 6. T7/T8 —— 权限与成本侧衔接

### 6.1 data_scope 行级隔离（T7）
- 事实澄清：**我们实际只有一个实验室**；产品里的"多院区"指的是**合作医院**（收入/客户维度，已在收入侧建模），**不是**内部部门行级隔离。
- 决策：**移除内部 `data_scope` 空壳**（现在 UI 配了但后端从不执行=既误导又埋越权错觉）；行级隔离记为**后续迭代**，真有多实验室时再做。

### 6.2 设备 / 工时成本（T8）
- 现状（核实）：设备可逐台建模 + BOM 出库自动登记折旧（`bom_equipment_templates`）+ 手工登记；工时**只有标准定义表**（`standard_labor_times`），无实际工时记录，人工成本按标准工时×费率推算。
- 事实：实验室**根本没有逐台分钟/逐次工时数据**。
- 决策：**粗放估算**，对齐 G2「耗材硬、工时间接估」。设备默认走**月度直线折旧估算**而非逼逐次登记；工时按标准推算（现状即如此，保留）。

## 7. 现状事实源（给实现用，经代码核实 · PARTIALLY_CONFIRMED）

真实库主数据规模（`coreone.db`，均为假种子）：物料 743 / 分类 641 / 检测项目 271 / BOM 538 / BOM 明细 2174 / 供应商 108 / 库位 241 / 设备 36 / LIS 病例 467。

**复核更正的 3 处**（写实现时按此为准）：
1. 盘点 `system_stock` 由**服务端确认时实时读**（`getScopedSystemStock`，`stocktaking-v1.1.ts:407/454`），不是调用方传入 → 无需冻结快照。
