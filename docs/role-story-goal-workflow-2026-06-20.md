# 用户角色故事目标模式工作流

> 创建日期: 2026-06-20
> 适用范围: COREONE 按用户角色和用户故事复核主流程
> 使用方式: 作为 Codex 目标模式的总入口说明

## 一、意图

上一轮 `main-flow-entry-*` 已按导航入口完成主流程收口。本轮不再重复按菜单逐项检查，而是从真实用户的工作视角出发，验证不同角色是否能完成自己的日常任务，并在跨角色交接后形成可信、可回看、可审计的业务结果。

本目标模式要回答的问题是:

1. 每个角色登录后，看到的菜单、权限和工作路径是否符合职责。
2. 用户能否按自然工作顺序完成一段完整故事，而不是只在单个页面内点通功能。
3. 角色之间的交接是否顺畅，例如采购员下单、仓管入库、技术员维护 BOM、财务核算、管理者查看结果。
4. 系统是否减少用户线下沟通、重复录入、人工查账和手工修正。
5. 库存、BOM、出库、成本、预警和审计事实链是否在用户故事中保持一致。

## 二、文档分工

目标模式执行时必须同时使用这三个文档:

| 文档 | 作用 |
|:---|:---|
| `docs/role-story-goal-workflow-2026-06-20.md` | 总原则: 为什么按角色故事做、故事如何定义、完成标准是什么 |
| `docs/role-story-execution-logic-2026-06-20.md` | 执行算法: 如何选故事、如何检索、如何生成问题清单、如何验证和切换 |

当文档冲突时，优先级为:

1. 用户最新明确指令。
2. `AGENTS.md`，尤其是 Playwright 禁止默认下载规则。
3. 本工作流总原则。
4. 执行逻辑文档。
5. 故事队列文档。

## 三、故事定义

本轮入口不是按钮、弹窗、API，也不是单一导航菜单，而是“某个角色为了完成一个业务目的所经历的完整工作故事”。

示例:

- “作为采购员，我要维护供应商并创建采购订单，以便仓库可以按订单入库。”
- “作为仓库管理员，我要把采购到货入库并在库存里回看批次，以便后续出库和成本核算可信。”
- “作为财务，我要完成消耗对账和成本核算，以便管理者能看见可解释的利润和趋势。”

一个故事可以跨多个页面，但必须有清晰边界。执行时只修当前故事中高频、主流程、事实链相关的问题；其他角色或相邻故事的问题记录为线索，不抢跑。

## 四、角色故事主流程定义

角色故事主流程必须同时满足:

1. 角色可进入: 使用该角色登录后，菜单、路由守卫和后端权限一致。
2. 用户可理解: 页面命名、字段、提示、状态和禁用原因符合该角色的业务语言。
3. 操作可完成: 用户能按自然顺序完成创建、维护、审核、导入、导出、核算或回看。
4. 交接可继续: 当前角色产出的数据能被下一个角色接住，不需要线下补口径。
5. 事实可追踪: 结果能在列表、详情、库存、BOM、成本、预警或审计日志中回看。
6. 权限可解释: 角色不能做的事要被合理隐藏、禁用或拒绝，且前后端口径一致。

## 五、用户故事质疑

每个故事必须主动质疑产品设计，而不是只查功能是否报错。

必须检查:

1. 角色职责是否准确: 该角色应该看到什么、能改什么、不能改什么。
2. 用户目标是否完整: 当前页面组合是否真的帮用户完成任务，还是只提供一堆孤立功能。
3. 信息是否够用: 用户做决定时是否缺供应商、批次、库存、BOM、成本、病例、状态、审计信息。
4. 交接关系是否正确: 上游数据是否能被下游角色识别、筛选、引用和解释。
5. 状态流转是否合理: 用户操作后的状态是否让下游知道能不能继续处理。
6. 用户工作量是否过高: 是否要求用户记住系统外规则、重复录入、线下确认或长期手工修正。
7. 成本与审计是否支撑: 故事结果是否最终能支撑成本分析和责任追溯。

如果设计会导致角色职责错位、交接断裂、事实链失真、成本结果不可解释或用户长期手工补救，即使按钮和接口正常，也必须作为问题进入清单。

## 六、高频核心问题口径

优先处理影响角色日常完成工作的事项:

- 角色登录后看不到应该处理的任务，或看到了不该处理的高风险操作。
- 前端允许操作但后端拒绝，或前端隐藏但后端可直接调用。
- 当前角色完成操作后，下游角色无法继续处理或无法理解状态。
- 页面成功提示与真实副作用不一致。
- 数据可写入但无法在角色常用页面回看。
- 用户故事跨库存、BOM、出库、成本、审计时事实链断裂。
- 用户需要线下表格、口头确认、重复录入或长期人工修正。

以下内容默认不主动扩展，除非阻断当前故事:

- 极低频非法参数组合。
- 纯粹分页、排序、导出格式边界。
- 与当前角色故事无关的隐藏路由。
- 只在开发者构造请求时出现、普通用户无法触发的问题。

## 七、完成标准

一个用户故事只有满足以下条件才算完成:

1. 本故事 P0/P1 已全部 fixed 并 verified。
2. 影响主流程和角色交接的 P2 已处理，或有清晰暂缓理由。
3. 已用对应角色真实登录或等价自动化验证页面、弹窗、菜单和权限。
4. 已验证后端副作用、数据库状态、数据回看和审计日志。
5. 已说明该故事对库存、BOM、出库、成本、预警和审计事实链的影响。
6. 已回答角色职责、信息字段、状态流转、交接关系和用户工作量是否合理。
7. 已更新 `docs/role-story-runs/` 中的故事工作记录。
8. 已把 `docs/role-story-list-2026-06-20.md` 中该故事更新为 `completed`，并填写验证摘要。

完成当前故事后，执行者应自动选择故事清单中的下一个 `pending` 故事继续执行。

## 八、目标模式一次性文案

创建目标时，直接使用下面这段:

```text
在 /Users/maxiaoyuan/Documents/进销存 中，按照以下三个文档执行 COREONE 用户角色故事复核目标:

1. docs/role-story-goal-workflow-2026-06-20.md
2. docs/role-story-execution-logic-2026-06-20.md
3. docs/role-story-list-2026-06-20.md

目标:
按 docs/role-story-list-2026-06-20.md 的故事队列，从下一个 pending 或 in_progress 的用户故事开始，逐个从真实角色视角完成职责权限检查、用户故事走查、产品建模质疑、问题整理、修复、验证和收口。每个故事完成后自动更新故事清单并切换下一个故事，直到故事清单全部 completed。

硬性要求:
1. 本轮按用户角色和用户故事粒度处理，不按按钮、弹窗、API 或单一菜单拆分。
2. 执行顺序必须遵循“系统与权限底座 -> 基础资料 -> 采购入库 -> 仓储作业 -> 技术建模 -> 财务成本 -> 医生/管理者查看 -> 跨角色全链路”。
3. 每个故事的问题清单由当次执行现场生成和持续补充，不使用预设问题池代替现场判断。
4. 不只检查功能异常，还要主动质疑角色职责、权限边界、信息字段、状态流转、交接关系、用户工作量和成本/审计支撑能力是否合理。
5. 如果设计会导致角色职责错位、交接断裂、业务事实错误、成本分析失真、用户长期手工修正，即使当前按钮和接口能正常运行，也必须作为问题进入清单。
6. 只处理当前故事的高频主流程核心问题；低频边界除非阻断当前故事，否则记录为 P3 或相邻线索。
7. 当前故事 P0/P1 和影响角色交接或事实链的 P2 未修复并验证前，不切换下一个故事。
8. 验证必须覆盖对应角色真实页面/弹窗、前后端权限一致性、后端副作用、数据回看、必要测试，以及库存/BOM/出库/成本/预警/审计等关键事实链影响。
9. 使用 Playwright 前必须先读 AGENTS.md 的 Playwright 强规则和其中指定的两条记忆文件；禁止默认运行 npx playwright install 或重复下载浏览器。
10. 旧审计报告、旧计划、上一轮入口记录和历史测试只能作为线索，不能作为当前故事完成证明。
11. 每完成一个故事，必须更新 docs/role-story-list-2026-06-20.md 的状态和对应 docs/role-story-runs/ 工作记录。
```

## 九、执行口径

本工作流是上一轮导航入口收口后的“角色验收层”。上一轮完成证明不能替代本轮角色故事验收；本轮必须站在用户工作现场重新判断。

最终判断标准只有一个: 每个角色是否能用系统顺畅完成自己的工作，并把结果交给下一个角色，最终形成可信、可解释、可重算、可审计的经营和成本事实。
