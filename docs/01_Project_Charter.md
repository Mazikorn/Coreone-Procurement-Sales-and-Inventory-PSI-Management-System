## 0. 一句话定义

COREONE 是一个面向病理免疫组化特染领域的 B 端实验室进销存与单张切片成本控制系统（PSI），支持多角色权限管理，核心目标是实现耗材从采购到使用的全链路追踪和精确成本核算。

---

## 1. 项目背景

### 1.1 业务背景

病理免疫组化特染实验室在日常运营中面临以下挑战：

- 耗材种类多、批次管理复杂，手工记录容易出错
- 成本核算依赖人工，无法精确到单张切片
- 多角色协同（仓库、技术员、病理医生、采购、财务）缺乏统一平台
- 缺少库存预警和消耗追踪，容易出现耗材断供或过期

### 1.2 系统定位

| 项目 | 定义 |
|------|------|
| 系统名称 | COREONE — 病理免疫组化特染进销存与单张切片成本控制系统 |
| 英文缩写 | PSI（Purchase-Sales-Inventory） |
| 目标用户 | 病理免疫组化特染实验室 |
| 系统类型 | B 端 SaaS / 私有化部署 |
| 核心价值 | 耗材全链路追踪 + 单张切片精确成本核算 |

---

## 2. 项目目标

### 2.1 核心目标

| 编号 | 目标 | 验证方式 |
|------|------|----------|
| G1 | 实现耗材从采购入库到出库使用的全链路追踪 | 每笔入库/出库/退库/报废均可追溯到批次和操作人 |
| G2 | 实现单张切片级别的成本核算 | 出库时自动计算材料成本、设备折旧、人工工时、间接成本分摊 |
| G3 | 支持 6 种角色的权限隔离 | 每个角色只能看到和操作其权限范围内的功能 |
| G4 | 提供多维度成本分析报表 | 按项目、物料、供应商、时间等维度生成成本报表 |
| G5 | 实现库存预警和消耗追踪 | 低库存、临期预警可生成与处理；自动通知、呆滞预警、消耗追踪需按当前代码复核 |

### 2.2 非目标（不做的事）

| 不做 | 原因 |
|------|------|
| 不做多租户 SaaS | 当前为单实验室部署 |
| 不做移动端 APP | 当前仅 Web 端 |
| 不做与 LIS 系统的实时对接 | 仅支持 LIS 病例数据导入，不对接 LIS 系统 |
| 不做财务核算（应收/应付/总账） | 仅做成本核算，不做财务系统 |
| 不做生产制造管理 | 仅做耗材进销存，不做生产排程 |
| 不做审批流引擎 | 仅做简单的确认/取消，不做复杂审批流 |

## 3. 目标用户与角色

### 3.1 角色定义

| 角色 | 业务职责 | 系统权限概要 |
|------|----------|-------------|
| `admin` | 系统管理员 | 全部功能 |
| `warehouse_manager` | 仓库管理员 | 入库、出库、盘点、退库、报废、调拨、库位管理 |
| `technician` | 技术员 | 查看项目、BOM、库存、出库记录、项目出库、退料 |
| `pathologist` | 病理医生 | 查看项目、成本报表、全成本分析 |
| `procurement` | 采购员 | 采购订单、供应商管理 |
| `finance` | 财务 | 成本报表、ABC 分析、间接成本、成本对账 |

### 3.2 用户规模

- 单实验室部署，预计 5-20 个并发用户
- 预计管理 100-1000 种耗材
- 预计每月处理 100-1000 笔出入库操作

## 7. 约束与限制

### 7.1 技术约束

| 约束 | 说明 |
|------|------|
| 数据库为 SQLite | 不支持高并发写入，单 worker E2E |
| 无 ORM | 原生 SQL，迁移通过 PRAGMA table_info 增量检查 |
| 单文件路由 | 业务逻辑直接在路由文件中，非控制器/服务分离 |
| 前后端分离 | 通过 API 通信，前端不直接访问数据库 |

### 7.2 业务约束

| 约束 | 说明 |
|------|------|
| 单实验室部署 | 不支持多租户 |
| 无实时 LIS 对接 | 仅支持批量导入 |
| 无审批流 | 简单确认/取消机制 |
| 密钥管理 | 通过 .env 文件，不支持密钥管理服务 |
