- 入口: 成本池

## 产品建模质疑

| 检查点 | 当前设计 | 质疑结论 | 影响 | 处理决定 |
|:---|:---|:---|:---|:---|
| 分类是否准确 | 成本池按期间、作业中心、来源归集直接/间接成本 | 分类符合 ABC 分摊入口，但来源值应受控 | 来源混乱会让财务无法解释费用从哪里来 | 本轮保持现状，记录来源枚举进一步收敛为 P3 |
| 字段是否够用 | 有期间、作业中心、直接成本、间接成本、总额、动因量、费率、来源、说明 | 字段覆盖 MVP 费率计算；缺少调整原因/附件/审批链 | 低频调整审计不足，但不阻断主流程 | 本轮先修主流程事实校验，调整审计留到成本调整/审计入口 |
| 关系是否正确 | 成本池引用作业中心并进入切片成本分摊 | 成本池必须只引用启用作业中心；此前 API 可写入空/停用/不存在作业中心 | 形成未关联作业中心或停用中心费率，后续成本分析失真 | 修复为写入时强校验启用作业中心 |
| 状态流转是否合理 | 成本期间可 open/collecting/calculated/closed，自动动作已有关账保护 | 手工写入此前绕过 `ensurePeriodOpen()` | 已关账期间仍可被改写，破坏关账可信度 | 修复 POST `/cost-pools` 关账保护 |
| 用户工作量是否合理 | 页面提供一键同步、自动归集、重算；无手工新增表单 | 高频流程不要求财务逐条建池；但 API 必须守住数据质量 | 自动归集可降低工作量；手工 API 宽松会产生长期修正 | 保持页面操作台形态，补 API 约束 |
| 是否支撑成本分析 | `totalCost / driverQuantity = driverRate` 进入后续 ABC 计算 | 动因数量必须为正，金额不能为负；此前零动因量会产生 0 费率 | 作业费率失真，切片成本和利润分析失真 | 修复金额非负、动因量大于 0 |
