# 物料分类 主流程收口记录

## 入口边界

- 入口: 物料分类
- 导航路径: 基础数据 -> 物料分类 (`/categories`)
- 主要用户: 管理员、仓管
- 本轮目标: 现场检查并修复物料分类高频主流程核心问题，确认分类建模能支撑物料、库存统计和成本归类。
- 包含范围: 分类列表、新建、编辑、删除/停用、状态展示、分类被物料引用后的保护、下游物料/库存/成本口径影响。
- 明确不做: 不扩展非导航入口；不修复旧版 `/cost-analysis`；不处理与物料分类无关的报表边界。

## 产品建模质疑

| 检查点 | 当前设计 | 质疑结论 | 影响 | 处理决定 |
|:---|:---|:---|:---|:---|
| 分类是否准确 | 三级树，允许没有子项的任意层级作为叶子 | 分类树本身可表达病理耗材大类/中类/末级，但物料必须绑定叶子而非汇总分类 | 避免汇总口径和实物物料混挂 | 已用后端校验和前端下拉过滤限制物料只能绑定无子分类节点 |
| 字段是否够用 | `code/name/parent_id/level/sort_order/status` | 基础字段可支撑主流程；树接口原先未返回 `parentId/status`，前端无法可靠编辑关系 | 编辑子分类可能误移到一级，状态/叶子过滤也缺少依据 | 已补树接口 `parentId/status/createdAt/updatedAt` |
| 关系是否正确 | 分类与物料通过 `materials.category_id` 绑定 | 原实现允许物料迁移/创建到仍有子分类的汇总节点 | 分类统计、库存统计、成本归类会混入汇总节点 | 已阻断非叶子分类绑定 |
| 状态流转是否合理 | 删除被子分类/物料引用时阻断；状态字段存在但分类页暂无独立停用入口 | 当前主流程删除保护可用；停用入口属于后续生命周期增强 | 不阻断本入口主流程，但后续可减少继续使用旧分类 | 记录为 P3 相邻线索，不在本轮扩展状态管理 |
| 用户工作量是否合理 | 分类详情可查看物料并迁移 | 迁移下拉原先可能让用户选错汇总分类，后续需要人工纠偏 | 增加分类核对和成本解释负担 | 已过滤为启用叶子分类，并在提示中明确“目标末级分类” |
| 是否支撑成本分析 | 分类进入物料、库存和成本材料维度 | 修复后分类事实更稳定，可支撑按末级/汇总层级解释成本 | 降低成本归类失真 | 已通过 API/数据库回看验证物料迁移后的分类事实 |
