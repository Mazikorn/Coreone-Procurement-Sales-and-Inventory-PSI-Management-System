# 退库管理 主流程收口记录

## 入口边界

- 入口: 退库管理
- 导航路径: 库存作业 -> 退库管理 (`/returns`)
- 主要用户: 仓管
- 本轮目标: 检查并修复退库创建、撤销、库存恢复、批次成本和审计流水的高频主流程核心问题。
- 包含范围: 退库列表、退库创建、退库撤销/删除、来源出库关联、批次库存恢复、库存流水、成本事实影响。
- 明确不做: 不提前修供应商退货、调拨、报废、盘点本体；若退库发现这些入口问题，先作为相邻线索记录。

## 当前检索记录

- 从 `015 出库管理` 自动切换而来。
- 已检索 `returns-v1.1.ts`、`Returns.tsx`、`returns.test.ts`、库存/报表规则和权限矩阵。
- 现场确认当前退库实现和页面文案把“退库”做成当前库存扣减；但业务规则和入口目标要求退库是“退回已出库物料”，应恢复库存并扣减净消耗成本。
- 现场确认类型中已有 `outboundItemId` 预留，但数据库、API 和页面未建立来源出库明细关系。

## 产品建模质疑

| 检查点 | 当前设计 | 质疑结论 | 影响 | 处理决定 |
|:---|:---|:---|:---|:---|
| 分类是否准确 | 退库入口被实现为库存扣减 | 与业务分类相反；这是“退回已出库物料”，不是再次出库 | 库存、批次、库位和净消耗成本方向全反 | P1 修复 |
| 字段是否够用 | 表单只选物料/批次/数量/原因 | 缺少来源出库明细，无法证明退回的是哪次出库 | 可凭空改库存，成本无法追溯原出库单价 | P1 修复：新增来源出库明细 |
| 关系是否正确 | `return_records` 未存 `outbound_item_id` | 退库与出库明细断链 | 后续成本报表只能按物料粗扣，审计不可解释 | P1 修复 |
| 状态流转是否合理 | 创建即 completed，删除即撤销 | 主流程可接受，但撤销必须反向扣回且防止已消耗库存被撤销 | 避免撤销后库存/批次为负 | 已修复 |
| 用户工作量是否合理 | 用户原来需要理解“退库却扣库存”的反直觉流程 | 改为选择已出库且仍可退明细，工作量更贴近业务事实 | 降低长期手工纠错 | 已修复 |
| 是否支撑成本分析 | 报表按退库成本扣减出库成本 | 报表方向可用，但前提是退库成本来自原出库明细 | 修复后使用原 `outbound_items.unit_cost` | 已验证 |
