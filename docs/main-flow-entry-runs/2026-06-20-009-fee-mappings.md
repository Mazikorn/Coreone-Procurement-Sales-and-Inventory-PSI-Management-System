# 收费映射 主流程收口记录

## 入口边界

- 入口: 收费映射
- 导航路径: 成本管理 -> 收费映射 (`/abc/fee-mappings`)
- 主要用户: 财务、技术员
- 本轮目标: 检查并修复 BOM/项目收费规则、收费项目映射、病例/样本口径、异常提示和成本收入链路的高频主流程核心问题。
- 包含范围: 收费映射列表、搜索筛选、新建、编辑、启用/停用、删除、BOM/项目/收费标准关联、收入口径、未映射异常、成本分析输入。
- 明确不做: 不直接修出库、成本看板或 ABC 计算本体；若发现相邻问题只记录线索。

## 产品建模质疑

| 检查点 | 当前设计 | 质疑结论 | 影响 | 处理决定 |
|:---|:---|:---|:---|:---|
| 分类是否准确 | 收费标准含 category/projectType，映射按 BOM 配置，可按出库单或病例聚合 | 入口能表达当前 MVP 的 BOM 收费规则；病例级阶梯已经能支撑一病例多 BOM | 分类错误会导致收入和利润解释失真 | 保留现有分类，重点修数据有效性 |
| 字段是否够用 | 映射字段为 feeStandardId、quantityMultiplier、aggregationScope、sortOrder/status | 覆盖主流程；但没有生效期/版本，费标停用后的历史解释需后续成本入口复核 | 收费标准变更可能影响重算解释 | 本轮记录为 P3，不阻断当前配置闭环 |
| 关系是否正确 | `bom_fee_mappings` 关联 BOM 和收费标准，但数据库无外键，后端保存前此前未硬校验 | 关系必须由接口守住，否则会出现“审计已配置但成本无收费”的错账 | 脏映射会让出库收费、利润、关账判断失真 | 修复保存/预览/成本读取的有效性校验 |
| 状态流转是否合理 | 映射本身 active；收费标准 active/inactive；审计按 active fee standard 判断 mapped/missing | 审计能识别停用收费标准为 missing，但成本计算此前仍可能读取停用 fee standard | 停用后仍收费会与审计、页面状态冲突 | 修复成本计算只读取启用收费标准 |
| 用户工作量是否合理 | 财务可在一个弹窗配置多收费项并预览；未映射可批量审计生成异常 | 主流程可用；页面已在 UI 层限制至少选择一个收费项，但后端需同等保护 | 仅靠页面限制会被 API 脏写绕过 | 后端补齐同等校验 |
| 是否支撑成本分析 | 出库、成本异常、关账和成本预览都读取收费映射 | 修复后主链路更可信；缺映射/停用 fee standard 会进入未映射和 0 收费状态 | 防止利润建立在无效收费项上 | P1 清零后收口 |
