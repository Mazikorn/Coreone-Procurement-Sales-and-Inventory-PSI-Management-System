# 库位管理 主流程收口记录

## 入口边界

- 入口: 库位管理
- 导航路径: 基础数据 -> 库位管理 (`/locations`)
- 主要用户: 仓管
- 本轮目标: 检查并修复库位结构维护、默认库位、停用/删除保护和库存引用链路的高频主流程核心问题。
- 包含范围: 库位列表、新建、编辑、删除、启用/停用、库区/库位层级、物料默认库位、库存/调拨引用保护。
- 明确不做: 不处理非库位入口的库存调拨细节；不扩展旧版成本分析；相邻库存问题只记录线索。

## 产品建模质疑

| 检查点 | 当前设计 | 质疑结论 | 影响 | 处理决定 |
|:---|:---|:---|:---|:---|
| 分类是否准确 | 库位类型有 shelf/fridge/cabinet/counter/other，层级标签可配置 | 类型本身可覆盖当前高频物理库位，但层级配置仍是前端本地配置，暂不作为跨用户事实源 | 若未来多人协作需要统一库位结构，前端本地配置会造成口径不一致 | 本轮记为 P3 相邻线索，不阻断主流程 |
| 字段是否够用 | `capacity/used/status/parent_id/zone/shelf/position` 支撑基础管理 | 容量是库存位置事实的一部分，0 容量不能被静默改成默认大容量；负容量必须拒绝 | 库位使用率、容量预警和后续库存位置解释会失真 | 已修复并补回归 |
| 关系是否正确 | 服务端已有父级存在/启用/防循环校验，删除/停用引用保护较完整 | 页面父级候选不能展示自己、下级或停用库位；表单候选也不能受当前列表搜索过滤影响 | 高频新建/编辑时用户容易选到提交后失败的关系，或搜索后找不到合法父级，增加长期手工修正 | 已修复并补回归 |
| 状态流转是否合理 | 停用前检查子库位、物料默认库位、库存总账、多库位库存 | 主流程保护基本合理；启用时校验父级可用 | 避免停用后仍被库存/物料引用 | 现有保护保留，测试覆盖 |
| 用户工作量是否合理 | 新建子库位会继承父级已填层级，剩余层级手动补齐 | 继承逻辑能减少重复输入；但当前点击树节点后新建不自动带入父级 | 不阻断主流程，但会增加高频建库位时点击/选择成本 | 记为 P3 相邻线索 |
| 是否支撑成本分析 | 库位影响物料默认库位、库存位置、调拨、入库事实 | 库位编码必须是稳定审计身份，不应被更新接口改写 | 若编码可被改写，虽然数据库外键用 id，用户导出/审计口径仍会出现同一库位身份漂移 | 已修复并补回归 |
