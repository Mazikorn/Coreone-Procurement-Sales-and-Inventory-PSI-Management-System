# 库存盘点 主流程收口记录

## 入口边界

- 入口: 库存盘点
- 导航路径: 库存作业 -> 库存盘点 (`/stocktaking`)
- 主要用户: 仓管
- 本轮目标: 检查并修复盘点创建、差异调整、批次/库位库存影响、原因记录、库存流水和审计的高频主流程核心问题。
- 包含范围: 盘点列表、盘点创建、盘点差异、库存调整、批次/库位影响、原因说明、流水和审计。
- 明确不做: 不提前修消耗对账、预警中心、成本分析本体；若盘点发现这些入口问题，先作为相邻线索记录。

## 产品建模质疑

| 检查点 | 当前设计 | 质疑结论 | 影响 | 处理决定 |
|:---|:---|:---|:---|:---|
| 分类是否准确 | 前端差异原因是 `normal`/`record`/`physical`/`other`，后端仅要求非空 | 前后端分类口径不一致，任意字符串会污染库存调整原因 | 审计、差异归因、成本异常解释会被手写原因打散 | P1 修复为后端受控枚举 |
| 字段是否够用 | 盘点单保存创建备注，确认原因和处理说明写入库存日志备注 | 当前可满足库存日志追溯；盘点详情不直接展示确认原因 | 详情回看原因需跳到日志，用户查找成本较高 | P3 记录，后续可扩 `adjust_reason`/`adjust_remark` 字段 |
| 关系是否正确 | 盘点按单物料总量调整，确认时同步库存、批次、库位、日志 | 对单物料总量盘点可闭环；不表达批次/库位级盘点范围 | 若未来做库位盘点或批次盘点，当前模型不够精确 | P3 相邻模型线索，不扩本轮范围 |
| 状态流转是否合理 | `completed` 创建后待确认，`confirmed` 后不可重复确认；撤销有库存变化保护 | 高频状态流转可接受 | 避免重复确认和旧盘点覆盖新库存 | 保留并回归 |
| 用户工作量是否合理 | 单物料选择、实盘录入、差异确认分两步 | 高频盘点工作量可接受 | 无需长期手工改库存 | 保留 |
| 是否支撑成本分析 | 盘亏扣批次，盘盈生成盘点批次并按物料价入账，日志记录原因 | 受控原因后可支撑成本异常解释和后续审计 | 无效原因会造成后续成本分析失真 | P1 已修复 |
