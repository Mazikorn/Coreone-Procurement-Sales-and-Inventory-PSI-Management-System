# BOM清单 主流程收口记录

## 入口边界

- 入口: BOM清单
- 导航路径: 基础数据 -> BOM清单 (`/bom`)
- 主要用户: 技术员
- 本轮目标: 检查并修复 BOM 创建、编辑、启用、删除、版本、项目绑定、物料组成、成本预览和出库支撑能力的高频主流程核心问题。
- 包含范围: BOM 列表、搜索筛选、新建、编辑、复制/版本、停用/启用、删除/批量操作、物料明细、项目/服务绑定、成本预览、可支撑样本数。
- 明确不做: 不处理项目、出库、采购/入库、成本报表入口自身流程；若发现相邻问题只记录线索。

## 产品建模质疑

| 检查点 | 当前设计 | 质疑结论 | 影响 | 处理决定 |
|:---|:---|:---|:---|:---|
| 分类是否准确 | BOM 类型含 `he/ihc/ss/mp/cyto/project`，服务绑定要求类型一致 | 类型能支撑当前检测服务/BOM 关系；列表标签仍偏业务简称，但不阻断主流程 | 类型错配会导致项目成本套错 BOM，后端已有 422 保护 | 本轮保留，后续可在文案层细化 |
| 字段是否够用 | BOM 编号、名称、版本、类型、服务、状态、物料组、质控、设备模板、收费类目 | 字段覆盖主流程；但 BOM 编号和可支撑样本数的人工编辑语义不合理 | 编号可改会破坏历史追溯；可支撑数人工维护会和库存实时计算冲突 | 修复为编号编辑态只读/API 拒绝改写；可支撑数只读且不入保存 payload |
| 关系是否正确 | BOM 可绑定一个启用检测服务，并同步 `projects.bom_id` | 主关系成立；同一服务只能绑定一个 BOM，类型必须一致 | 关系错误会影响出库选择和成本快照 | 既有后端保护保留并纳入验证 |
| 状态流转是否合理 | active/inactive，停用前检查项目引用，启用前检查依赖可用 | 状态保护合理；历史空 BOM 不能重新启用 | 防止出库和成本引用不可用 BOM | 既有保护保留并纳入测试 |
| 用户工作量是否合理 | 主用户为技术员，但前端和路由内写权限曾只允许 admin | 技术员能看到入口却不能创建 BOM，不符合主流程责任 | 技术员无法完成基础配置，出库/成本链断裂 | 修复前后端写权限与 `bom` 模块权限一致 |
| 是否支撑成本分析 | BOM 明细、设备模板、质控、服务绑定、成本预览和版本快照存在 | 主链路可支撑成本分析；本轮修复避免身份字段/实时库存字段污染成本事实 | 维护错误会让后续出库、成本、审计长期靠人工修正 | P1/P2 清零后收口 |
