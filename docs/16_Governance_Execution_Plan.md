## 1. 执行目标

本规划的目标不是重新开发 COREONE，而是把已有代码、历史设计、E2E、QA 报告、多 Agent 工作日志统一纳入可持续治理体系，让后续开发进入稳定闭环。

当前工作方式已经明确为双工作台：

| 工作台 | 定位 | 负责事项 |
|---|---|---|
| PM | 最终决策与验收 | 决定业务规则、优先级、任务范围、验收结论和发布决策 |
| Codex | 规划、治理、复核、定向修 bug | 生成治理/规划文档、拆任务、写任务包、做影响分析、复核 mimo 结果、生成 PM 验收脚本、处理小范围关键 bug |
| VS Code Claude Code CLI + mimo | 主要代码执行 | 按 Codex 任务包完成大部分前端、后端、测试、脚本和批量修复工作，并提供测试结果和执行证据 |

## 2. 执行范围

### 2.1 本次治理包含

| 范围 | 内容 | 默认主责 |
|---|---|---|
| 文档治理 | 建立正式 `docs/` 文档体系，迁移和索引历史资料 | Codex |

| PM 验收 | 把技术测试转成 PM 可执行操作脚本 | Codex |

| 发布治理 | 建立发布前检查清单和回滚要求 | Codex |

### 2.2 本次治理不包含

| 不包含 | 说明 |
|---|---|
| Codex 承担全部具体开发 | 大部分具体实现交给 VS Code Claude Code CLI + mimo |
| mimo 自行定义任务边界 | mimo 必须按 Codex 任务包或 PM 明确指令执行 |
| 自动部署 | 发布必须由 PM 明确批准 |
| 自动合并分支 | 合并动作必须由 PM 决策 |
| 读取密钥 | 不读取 `.env`、token、生产凭证；只允许读取 `.env.example` |
| 一次性重构全部文档 | 先建立事实源和核心闭环，再逐步归档 |

### 2.3 Codex 可直接修 bug 的边界

Codex 默认负责规划、文档和复核，但以下 bug 可以由 Codex 直接修：

| 可由 Codex 直接修 | 不建议 Codex 直接修 |
|---|---|
| 影响规划验证的明显文档/脚本问题 | 大范围页面重构 |
| 复核 mimo 产出时发现的单点缺陷 | 批量 UI 适配 |
| 小范围、低风险、可快速验证的 bug | 涉及多模块的数据迁移 |
| PM 明确指定 Codex 修复的 blocker | 需要长时间跑 E2E 的批量修复 |

## 3. 执行原则

| 原则 | 执行要求 |
|---|---|
| 先事实，后整理 | 每份文档先列依据来源，再给结论 |
| Codex 先出任务包，mimo 再执行 | 具体实现前先明确允许修改文件、禁止事项、验收标准和测试要求 |
| 先高风险，后美化 | 权限、库存、成本、测试门禁优先于格式优化 |
| 一次生成一组相关文档 | 避免一次生成全部文档导致事实错漏 |
| 每份文档有 PM 审核点 | AI 起草不等于 PM 确认 |
| 旧文档不删除，先归档 | 先标注历史参考，确认后再移动 |
| 代码与文档冲突时以代码为准 | 同时记录文档过期风险 |
| mimo 执行后必须回传证据 | Codex 复核需要实际修改文件、测试结果、风险和未完成项 |
| Codex 复核后再关闭任务 | mimo 完成不等于任务关闭 |

## 4. 总体路线图

建议按 6 个阶段执行。若每天投入 1-2 个集中工作块，预计 2-4 周可建立完整治理闭环；若只利用零散时间，可按阶段逐步推进。

| 阶段 | 名称 | 预计工作量 | Codex 主责 | mimo 主责 | 阶段门禁 |
|---|---|---:|---|---|---|
| Phase 0 | 启动与基线冻结 | 0.5-1 天 | 执行规划、协作边界、基线规则 | 提供当前执行状态 | PM 确认执行范围 |
| Phase 1 | 项目接管与事实报告 | 1-2 天 | `14_Project_Resume_Report.md`、资料可信度判断 | 补充环境/测试执行结果 | 历史资料和当前实现差异明确 |
| Phase 2 | 核心事实文档 | 3-5 天 | `01`、`04`、`05`、`06` 初稿与核对 | 按需补查代码和运行验证 | 权限、业务规则、数据对象可核对 |
| Phase 3 | PRD 与验收体系 | 3-5 天 | `02`、`07`、`08` 与 PM 验收脚本 | 补齐对应 E2E/测试证据 | 每个核心功能有验收脚本 |
| Phase 4 | 任务/缺陷/决策统一 | 2-3 天 | `09`、`10`、`11`、`13` 和任务包 | 按任务包执行修复并回传结果 | 后续任务只有一个入口 |
| Phase 5 | 发布门禁与归档 | 1-2 天 | `12`、归档策略、发布评审模板 | 执行归档、测试、发布前检查项 | 发布检查和历史归档规则可执行 |
| Phase 6 | 持续运行机制 | 持续 | 周复盘、复核、任务拆解、关键 bug 修复 | 日常开发执行、测试补充、回归修复 | 每周更新并可追溯 |

| MOD-PROC | 采购 | 采购订单、供应商退货 |
| MOD-INBOUND | 入库 | 入库登记、批次、采购订单关联 |
| MOD-INVENTORY | 库存 | 库存列表、批次、预警、盘点、调拨、报废 |

| MOD-AUDIT | 审计与日志 | 操作日志、变更追踪 |

### 8.4 验收标准格式

```markdown
### FEAT-INBOUND-001 入库登记

**关联模块**: MOD-INBOUND
**关联规则**: BR-INBOUND-001, BR-INBOUND-002
**关联对象**: InboundRecord, Batch, Inventory
**优先级**: P0
**状态**: 待验收

**前置条件**
- 已有有效供应商、物料、库位。

**操作**
1. 以 warehouse_manager 登录。
2. 打开入库页面。
3. 新增入库记录并提交。

**期望结果**
- 页面提示成功。
- 入库记录可查询。
- 批次记录生成。
- 库存数量增加。

**证据**
- 截图或 E2E 报告。
- 数据查询结果。
```

## 14. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| AI 一次性生成太多文档导致错误累积 | PM 审核困难 | 按批次生成，每批不超过 3 份核心文档 |
| Codex 和 mimo 职责重叠 | 重复修改、重复规划、互相覆盖 | Codex 默认输出任务包和复核；mimo 默认执行；例外必须在任务中写明 |
| mimo 自行扩大任务范围 | 引入新风险 | Codex 任务包必须列允许/禁止修改文件，mimo 发现遗漏先反馈 |
| Codex 直接做大范围实现 | 破坏分工 | Codex 只做定向 bug 修复或 PM 明确指派的实现 |
| 历史文档与当前代码冲突 | 错误治理结论 | 每份文档必须写依据来源和冲突说明 |
| PM 审核压力过大 | 文档无法长期维护 | PM 只审核业务判断，AI 负责格式和追溯 |
| 工作区已有大量未提交改动 | 容易误归因 | 只做文档任务，不回滚、不覆盖他人改动 |
| E2E 当前仍有失败 | 发布判断失真 | 在发布清单中区分核心 E2E、完整 E2E、已知失败 |
| 根目录历史资料太多 | AI 误读 | Phase 5 前不删除，先索引再归档 |
| 权限和菜单不一致 | 角色验收失败 | 权限矩阵必须同时核对后端路由、前端菜单、E2E |
| 成本规则描述不精确 | 后续实现偏差 | 成本相关规则必须引用具体公式和业务例子 |

## 16. PM 审核清单
