**总体结论**：基础功能的**主链路与事务底座是可靠的**——横向数据一致性审查确认六类库存写操作（入/出/调/盘/废/退）均包裹在 `BEGIN IMMEDIATE` 事务内、BEGIN↔COMMIT 间无 await（node:sqlite 同步 API + 单线程 → 单进程内无 TOCTOU 竞态），取消/驳回普遍走对称回滚，且有 `inventory-consistency.ts` 守恒不变量侦测层。RBAC 也无可越权写入的硬漏洞（端点级硬守卫兜底正确）。

但存在**一个真正的跨账守恒断裂（P0 critical）**，以及一批**"看起来配了、实际静默失效"的空壳/口径割裂**和**与业务规则/验收标准直接冲突的流程断裂**——这些大多不是崩溃，而是**侵蚀核心价值与采纳信任**的缺口（库存预警静默不触发、标准成本恒回退、对账永远不平、利用率永远 0%、过期→报废断链等）。这类问题恰好踩中项目既定的 [[adoption-first-design]] 最高否决原则。

## 三、P0 — 必须优先修（数据正确性 / 安全空窗 / 核心价值静默失效）

### AUDIT-P0-01 ⛔ 耗尽确认破坏库存四账守恒（唯一 critical）
- **位置**：`后端代码/server/src/routes/depletion-v1.1.ts:182-189`
- **问题**：`POST /depletion/tracking/:id/deplete` 在事务内执行 `UPDATE batches SET remaining = ?, status = 2 WHERE batch_no = ? AND material_id = ?`，用操作员上报的台面剩余量（remainQty）**绝对覆盖**仓库批次 `batches.remaining`，且**全程不联动** `inventory.stock` / `inventory_locations` / `batch_location_balances`，并按 `batch_no`（非主键 id）模糊匹配。
- **影响**：单条耗尽确认即让"启用批次剩余汇总"与 `inventory.stock`/库位余量/批次库位余量三账劈叉；`status=2` 使该批次被 `allocateBatches`（只取 status=1）与一致性求和排除 → 立即触发 `INVENTORY_BATCH_MISMATCH`，且 `inventory.stock` 仍虚高 → **永久幽灵库存**；同 `batch_no` 多行时会误伤其它行。出库时已扣减过 remaining，此处等于二次覆盖。
- **修复**：区分"领用台账 `batch_usage_tracking`"与"仓库批次台账 `batches`"两个语义层——耗尽确认只更新 tracking 的 remaining/status，**绝不直接覆盖 `batches.remaining`**；若确需把台面余量退回仓库，走与出库回退对称的**增量**逻辑（`restoreInventoryLocationStock` + `restoreBatchLocationStock` + `inventory.stock += ?` + `batches.remaining += ?` + 写 `stock_logs`），且按出库时记录的 `batch_id` 精确定位。

### AUDIT-P0-02 🔒 data_scope（本部门/本人）全链路从不强制执行 — 空壳越权
- **位置**：`middleware/auth.ts:82-152` + `routes/roles-v1.1.ts:83-89,191,252` + `routes/users-v1.1.ts:50-63`
- **问题**：角色/用户层完整维护并校验 `data_scope`（all/dept/self），dept 角色还强制绑定 department、列表回显 dataScope；但**任何中间件与数据路由从不读取它**，JWT 载荷连 department 都不含。
- **影响**：被授予 dept/self 范围的角色实际拿到**全量数据**（横向数据泄露）；管理员认真配置的数据范围被系统悄悄忽略 → 既误导又埋合规风险。
- **修复**（二选一，勿留半截）：(a) MVP 不做行级隔离则从 schema/UI 移除 data_scope；(b) 保留则把 department 写入 JWT，并在各数据路由 where 按 role.data_scope 实际过滤（dept→`AND department=?`，self→`AND created_by=?`），补 E2E。

### AUDIT-P0-03 🔒 改密/改角色/停用/删除后已签发 JWT 不失效 — 最长 8h 安全空窗
- **位置**：`routes/auth.ts:103-118` + `middleware/auth.ts:86-102` + `routes/users-v1.1.ts:204-371`
- **问题**：token 有效期 8h，`authenticateToken` 仅 `jwt.verify`，不回查 `users.status`/`is_deleted`、不比对 role、无 tokenVersion/黑名单。停用/降权/重置密码只改库，不触碰已发 token。
- **影响**：停用某员工 / 把其从 admin 降为 manager / 重置疑似泄露的密码后，旧 token 在到期前（≤8h）仍可调用全部原权限接口 → 应急响应失效。
- **修复**：`authenticateToken` 对写敏感路由轻量回查 `users`（status=1 且 is_deleted=0 且 role 与 token 一致），或引入 `token_version`/`password_changed_at`，停用/改密/改角色时自增使旧 token 立即失效（`/auth/refresh` 已有类似 status 检查可对齐）。

### AUDIT-P0-04 库存预警阈值双轨割裂 — 低库存告警在多数物料上静默失效
- **位置**：`MaterialFormModal.tsx:152-167` / `routes/materials.ts:28,112` / `MaterialTable.tsx:205` / `utils/alertChecker.ts:14-22` / `routes/alerts-v1.1.ts:385-397`
- **问题**：表单把标签"安全库存"绑到 `minStock`、"保险库存"绑到 `safetyStock`（语义错位）；列表 lowStock 过滤、表格标红、stats 都用 `min_stock`；但**真正生成低库存告警的引擎读 `safety_stock`**（默认 0）。
- **影响**：仓管按直觉在"安全库存"栏设阈值后，库存跌破时列表会标红但**预警中心不产生任何告警** → 库存预警这一核心价值在多数物料上静默失效，用户难自查。
- **修复**：统一单一阈值语义（建议预警引擎与列表都用 `min_stock`，`safety_stock` 仅作补货参考或废弃），表单标签改为与引擎一致（"库存预警阈值"），补 E2E：设阈值→出库跌破→断言预警出现。

### AUDIT-P0-05 BOM 标准成本加权均价取数失效 — 成本恒回退主数据价
- **位置**：`后端代码/server/src/routes/bom-v1.1.ts:734-762`
- **问题**：`updateBomStandardCost` 第 1 步 SQL `SELECT bi.usage_per_sample, m.price ...` **漏选 `bi.material_id`** → `materialIds` 恒空 → `weightedPrices` 恒为 `{}` → 落库材料成本恒按主数据 `price` 算，永远不反映批次加权价。
- **影响**：列表/详情动态展示的 `unitCost` 与落库 `standard_total_cost` 口径不一致；主数据 price 未维护/为 0 时标准成本被严重低估/归零 → 成本控制核心卖点失信。
- **修复**：第 1 步 SQL 补 `bi.material_id`，使 `weightedPrices` 生效；建议抽取与 GET /:id、cost-preview 共用的加权价函数消除三处口径漂移。

### AUDIT-P0-06 物料对账「按物料汇总」口径不对等 — 对账永远不平
- **位置**：`后端代码/server/src/routes/reconciliation-v1.1.ts:652-666`
- **问题**：`actualOutboundRows` 只 `WHERE o.status='completed' AND o.is_deleted=0`，**无 project_id 过滤**（SUM 全部出库）；而 theory 只累加项目派生量（已配 BOM 的项目）。两口径不对等（项目级 `getProjectMaterialReconciliation` 反而正确带 `o.project_id=?`）。
- **影响**：只要物料存在任何无项目归属出库，"按物料汇总"差异即失真，几乎所有物料标红/黄 → 财务无法判断哪个真异常 → 回退线下 Excel（adoption-first 反模式）。
- **修复**：actual 两分支补 `AND o.project_id IS NOT NULL AND o.project_id != ''`（且项目已配 BOM）使其与 theory 同口径；或在 UI 把"未关联项目出库"单列、不参与对账。

## 四、P1 — 重要（流程断裂 / 采纳杀手 / 空壳功能）

### 流程断裂（与业务规则/验收冲突）
- **AUDIT-P1-01｜BOM 出库辅料缺货应跳过却整单回滚**：`outbound-v1.1.ts:591-640` 对通用试剂/耗材/质控品逐项 `allocateBatches`，缺货时 throw → 整单 `ROLLBACK`，**直接违反 BR-OB-022/023/024（"不足时跳过、不阻断"）**。一个辅料缺货即挡住整张 BOM 出库 → 仓管改线下记账。修：对辅料 try/catch 跳过（记 cost_exception 提示），或修订 BR 文档对齐。
- **AUDIT-P1-02｜过期预警→报废流程断裂**：`AlertHandleModal.tsx:24-28` 对 expiry/expired 预警只让填文本"记录处理结论"，不触发也不跳转报废；`AlertPanel.tsx:46-56` 的"已过期"卡片是纯展示 div 点不动。仓管须关弹窗→手动导航 /scraps→999 物料下拉重搜→重选批次。修：弹窗加"去报废"复用深链 `/scraps?action=create&materialId=&batchId=&reason=expired`。
- **AUDIT-P1-03｜审计 before/after 渲染成 `[object Object]`**：`LogDetailModal.tsx:132-137` 用 `String(value)` 渲染，而 materials 等写入的 `requestData` 是 `{before:{…},after:{…}}` 嵌套对象 → 审计员看不到改前/改后值（前序评审专门补的最有价值证据在 UI 失效）。修：识别 before/after 做并排字段对比渲染。
- **AUDIT-P1-04｜盘点一物一单、无批量/导入**：`stocktaking_records` 每行单 material_id，创建弹窗恒单物料，POST 只收单 `materialId`。周期全仓盘点要逐 SKU 走 3 步弹窗 → 改用线下 Excel。修：引入盘点单表头 + 明细行，按库位/分类批量生成待盘明细 + Excel 导入实盘。
- **AUDIT-P1-05｜设备使用不录项目/出库归属**：`EquipmentDetailModal.tsx:41` usageForm 仅 date/minutes/count，后端字段 project_id/outbound_id 恒 NULL，**违反 REQ-24-004**。事后无法从折旧追溯哪个项目/出库消耗了设备。修：登记表单加可选项目/出库单。

### 空壳 / 目的未闭环（"看似接线、实则死值"）
- **AUDIT-P1-06｜库位 used/容量/利用率是装饰字段**：`locations.used` 全后端**无任何写入方**，`/stats` 的 avgUtilization 永远 0%、容量进度条永远空。仓管看一次发现数字假即弃用。修：用 `SUM(inventory_locations.stock) GROUP BY location_id` 派生利用率（零双写），或移除容量/利用率展示。
- **AUDIT-P1-07｜BOM 设备模板 UI 不可配置、只能脚本播种**：`BOMForm` 无 equipmentTemplates 字段、buildPayload 不构造，后端只在传该字段时才写 → 任何角色无法经界面增改设备模板，`BOMDetailModal:322` 只读渲染。新建 BOM 设备标准成本恒 0（设备成本是 BR-BM-014/018 五要素之一）。修：BOMFormModal 增设备模板 Tab + buildPayload/toForm 双向。
- **AUDIT-P1-08｜设备折旧只能手填、无 PSI 自动入口**：`equipment_usage` 唯一写入方是手工 `POST /:id/usage`，ABC 设备折旧池整额来自其 SUM → 技术员每月不逐台手填则设备成本为 0。修：出库消耗 BOM 设备模板时按 usage_minutes×样本量自动生成台账，或月度直线法自动折旧分录。
- **AUDIT-P1-09｜"消耗异常/呆滞(stagnant)"预警空壳**：种子有 RULE-003 stagnant、前端做成一等公民（标签/筛选/两个专用弹窗），但**全代码无任何写 type='stagnant' 的语句** → 该筛选永远 0 条、弹窗永不触发。修：接线消耗对账差异/呆滞天数生成 stagnant alert，或下线该类型与 RULE-003。

### 安全 / RBAC 边界
- **AUDIT-P1-10｜Token 自动续期未接线（AC-01-002 落空）**：后端发 refreshToken(7d) + 有 `/auth/refresh` 端点，但前端 `request.ts:35-43` 401 只清 token 跳登录，**从不调用 refresh** → 8h 一到任意角色被强制踢回登录、未存表单丢失。修：401 分支加续期（isRefreshing 锁 + 重放队列），refresh 也失败才登出。
- **AUDIT-P1-11｜登出/401 只清 token，残留 user/refreshToken/rememberUsername**（H→M）：违反 AC-01-003；`user` 是前端角色来源，共用终端身份残留。修：抽 `clearAuth()` 统一清除，三处（登出/401/refresh 失败）复用。
- **AUDIT-P1-12｜库位写权限(仅 admin)与权限矩阵(admin+wm)矛盾**：`locations-v1.1.ts:11` 用 `requireStrictRole('admin')` 守写，但矩阵三处声明库位写=admin+warehouse_manager（角色故事 002/005 把库位归仓管日常）。仓管建库位被 403。修：对齐实现与文档二选一（建议放开给 wm，与 inbound/outbound 一致）。
- **AUDIT-P1-13｜供应商 refundAmount 无上界、可注入虚高退款**（H→M）：`supplier-returns-v1.1.ts:302-307` 仅校验 ≥0，不与 `inbound_price×qty`（BR-SR-015）勾稽，且该值进入供应商对账 SUM。修：以来源单据成本为默认与上界，超出需审批 + 审计记录系统值 vs 人工值。
- **AUDIT-P1-14｜供应商退货退款财务闭环未达成**（H→M）：refundAmount 创建后不可改、'refunded' 状态无任何财务过账、finance 角色无权访问 → 财务拿不到可信退款应收、错填只能删单重建。修：加 refundAmount 可编辑端点 + finance 只读访问 + refunded 时写应付贷项/对账记录。

### 正确性
- **AUDIT-P1-15｜BOMFormModal 在 `if(!open) return null` 之后调用 useState**：`BOMFormModal.tsx:29-31` 违反 Rules of Hooks，反复开关存在 "Rendered more hooks" 运行期风险。修：把早退移到所有 Hook 之后，或抽子组件由父层条件渲染。
- **AUDIT-P1-16｜设备使用登记缺幂等**：`equipment-v1.1.ts:553-622` 无去重，双击/重试/同出库多次登记 → 双计折旧 → 设备净值低估、ABC 设备动因高估。修：对 (equipment_id, outbound_id) 加唯一约束或先查后插（outbound_id 为空的纯手工登记除外）。
- **AUDIT-P1-17｜耗尽确认按 batch_no 模糊匹配 + 绝对覆盖**（同 P0-01 根因，后端视角）：`depletion-v1.1.ts:182-189` 用领用残量覆盖批次全局 remaining 并 status=2，同 batch_no 多行全中。随 P0-01 一并修。

## 五、横向交叉结论

**① 库存守恒与事务（cross-data-integrity）— 整体可靠，唯一 critical 已列 P0-01。**
所有六类写操作均 `BEGIN IMMEDIATE`…COMMIT/ROLLBACK 包裹，BEGIN↔COMMIT 间无 await（同步 API + 单线程 → 单进程内无真正 TOCTOU）；取消/删除/驳回重放对称回滚，对"批次被后续业务调整""恢复超量"有 422/409 守门；存在 `inventory-consistency.ts` 守恒不变量侦测。**唯一断裂是耗尽确认（P0-01）。**

**② RBAC（cross-rbac）— 无可越权写漏洞，问题在一致性与文档漂移。**
7 角色在所有 PSI 路由上无可越权写入（端点级硬守卫兜底正确）。真实问题：(a) **strict vs loose 不一致**——outbound 用 `requireStrictRole` + 端点级写守卫双层硬隔离，而 transfers/scraps/returns/stocktaking 仅靠 app.ts 单层 `requireRole` 软校验（无端点级写守卫）→ 带模块权限的自定义角色可写这些库存台账，与 outbound 口径不一致；(b) **文档与代码不一致**——materials 写、locations 写、materials 读与矩阵 3.3/4 章对不上；(c) `categories` 路由层缺 `requireRole`（app.ts:201 仅 authenticateToken）。建议统一库存台账写守卫口径 + 同步权限矩阵文档。

**③ 设计规范（cross-design）— 主色/字体/h-10/text-black 合规，五处系统性违反：**
(1) 全站弹窗一律 `shadow-xl`（含共享 `Modal.tsx:27`，违反"modal 用 shadow-lg"）；(2) **React Query 强制规范被全量违反**——`useQuery/useMutation` 全仓 0 处，18 个页面 hook 一律手写 useEffect+命令式 Api，`QueryClientProvider` 成死依赖；(3) Login/TopBar 大量 hex 字面量（`text-[#111827]`/`bg-[#3b82f6]`）架空主题 token；(4) `border-gray-100` 分隔线 14 处（规范明令禁用）；(5) 弹窗圆角不一致（rounded-xl 为主，部分 rounded-lg）。
