# 主流程导航入口目标模式工作流

> 创建日期: 2026-06-20
> 适用范围: COREONE 主流程高频核心功能优化
> 使用方式: 作为 Codex 目标模式的总入口说明

## 一、意图

本工作流用于长期目标模式，不用于一次性人工挑任务。

目标模式只需要预设一次目标，然后由执行者按导航栏功能入口自主推进:

1. 从入口清单中选择下一个未完成的导航入口。
2. 围绕该入口做全面检索、现场问题整理、修复和验证。
3. 当前入口完成后，自动更新入口状态并切换到下一个入口。
4. 不依赖提前预设问题池，不让旧审计清单牵着走。
5. 按业务建模顺序推进: 先基础数据配置，再过程数据，最后成本分析。

这套机制要解决的偏移是: 近期修复容易集中到筛选、分页、导出、非法参数、极低频保护等外围边界，而不是把用户每天真正使用的主流程做顺、做实、做可信。

## 二、文档分工

目标模式执行时必须同时使用这三个文档:

| 文档 | 作用 |
|:---|:---|
| `docs/main-flow-entry-goal-workflow-2026-06-20.md` | 总原则: 为什么做、入口如何定义、完成标准是什么 |
| `docs/main-flow-entry-execution-logic-2026-06-20.md` | 执行算法: 如何选入口、如何检索、如何建问题清单、如何修复、如何自动切换 |
| `docs/main-flow-entry-list-2026-06-20.md` | 入口队列: 按导航栏功能入口列出处理顺序、范围和状态 |

当三个文档冲突时，优先级为:

1. 用户最新明确指令。
2. `AGENTS.md`。
3. 本工作流总原则。
4. 执行逻辑文档。
5. 入口清单文档。

## 三、入口定义

入口按左侧导航栏的功能菜单定义，而不是按单个按钮、弹窗或 API 定义。

示例:

- `入库管理` 是一个入口。
- `出库管理` 是一个入口。
- `BOM清单` 是一个入口。
- `成本看板` 是一个入口。

入口内部可以包含多个主流程动作，例如新建、编辑、删除、导入、导出、审核、取消、详情回看、下游联动等。执行时应在当前导航入口内统一处理，不要让用户手动把入口切成更小目标。

如果某个导航入口过大，执行者可以在入口工作记录中拆分内部检查段落，但这只是执行组织方式，不改变目标模式的入口粒度，也不能在入口未收口时切换到另一个导航入口。

## 四、主流程定义

主流程指普通用户在正常业务中高频使用、并会留下业务结果的路径。它必须同时满足:

1. 前端可操作: 页面、表单、弹窗、按钮、提示真实可用。
2. 后端有副作用: 数据确实创建、更新、删除、流转、重算或导出。
3. 数据可回看: 操作结果能在列表、详情、报表、日志或下游入口看见。
4. 事实链不断裂: 库存、批次、BOM、病例、收费、成本、关账、审计等关键链路保持一致。
5. 产品目标一致: 功能结果能减少用户手工修正，而不是制造新的核对负担。

## 五、产品建模质疑

执行者不能默认当前功能设计就是正确的。每个导航入口除了检查“有没有 bug”，还必须质疑“这个入口的业务建模是否合理”。

必须主动检查:

1. 分类是否准确: 例如物料分类、项目分类、BOM 类型、成本科目、作业中心、成本动因是否符合病理实验室真实管理口径。
2. 字段是否够用: 当前字段是否能支撑采购、入库、出库、对账、成本归集、关账和审计。
3. 关系是否正确: 物料、项目、BOM、收费、病例、批次、库位、设备、工时之间的绑定是否符合业务事实。
4. 状态流转是否合理: 启用、停用、删除、取消、审核、关账、重算等状态是否会破坏历史事实。
5. 用户工作量是否合理: 当前设计是否会让用户反复手工修正、线下核对、重复录入或无法解释结果。
6. 成本目标是否支撑: 该入口是否能为最终成本分析提供可信、可重算、可审计的数据。

如果发现产品建模问题，即使功能按钮可以点击、接口可以成功，也必须进入问题清单。建模问题按影响分级: 会导致后续事实链错误或成本分析失真的，至少按 P1 处理。

## 六、高频核心问题口径

优先处理会影响用户日常完成工作的事项:

- 用户无法完成创建、提交、审核、取消、重算、导入、导出等核心动作。
- 页面显示成功但后端没有真实副作用。
- 后端写入了数据，但页面或下游入口无法正确回看。
- 数据链影响库存、批次、BOM、病例、收费、成本、关账、审计可信度。
- 成功提示、错误提示、禁用状态、确认弹窗会误导用户做错事。
- 用户完成主流程后仍需要大量人工修正、人工查账或人工解释。
- 当前分类、字段、状态或关系设计会导致业务事实表达错误。
- 当前设计能运行，但无法支撑后续成本分析的可信、可重算、可审计。

以下内容默认不主动扩展，除非它阻断当前导航入口主流程:

- 极低频非法参数组合。
- 纯粹的分页、排序、筛选边界。
- 与当前入口无关的历史数据兼容。
- 不影响主流程完成和事实链可信度的导出格式细节。
- 只在开发者直接构造请求时出现、普通用户无法触发的问题。

## 七、完成标准

一个导航入口只有满足以下条件才算完成:

1. 本入口 P0/P1 已全部 fixed 并 verified。
2. 影响主流程的 P2 已处理，或有清晰暂缓理由。
3. 页面、弹窗、按钮和提示经过真实操作或等价自动化验证。
4. 后端副作用经过 API、数据库、页面回看或测试验证。
5. 对库存、批次、BOM、病例、收费、成本、关账、审计的影响已经说明。
6. 已明确回答本入口的分类、字段、状态、关系设计是否合理；如不合理，已修复或记录阻塞原因。
7. 本入口工作记录已更新问题清单、修复记录、验证证据和收口报告。
8. 入口清单中的状态已更新为 `completed`，并记录工作记录路径。

完成当前入口后，执行者应自动选择入口清单中的下一个 `pending` 入口继续执行。

## 八、目标模式一次性文案

创建目标时，直接使用下面这段，不需要每次手动指定入口:

```text
按照以下三个文档执行 COREONE 主流程导航入口优化目标:

1. docs/main-flow-entry-goal-workflow-2026-06-20.md
2. docs/main-flow-entry-execution-logic-2026-06-20.md
3. docs/main-flow-entry-list-2026-06-20.md

目标:
按 docs/main-flow-entry-list-2026-06-20.md 的入口队列，从下一个 pending 或 in_progress 的导航栏功能入口开始，逐个完成主流程高频核心功能检查、问题整理、修复、验证和收口。每个入口完成后自动更新入口清单并切换下一个入口，直到入口清单全部 completed。

硬性要求:
1. 每个入口按导航栏功能菜单粒度处理，不要要求用户手动拆成按钮级或弹窗级入口。
2. 入口顺序遵循“基础数据配置 -> 过程数据 -> 成本分析”，不要跳过底座直接修报表。
3. 每个入口的问题清单由当次执行现场生成和持续补充，不使用预设问题池代替现场判断。
4. 不只检查功能异常，还要质疑当前产品设计是否合理，包括分类、字段、关系、状态流转和成本分析支撑能力。
5. 只处理当前入口的高频主流程核心问题；低频边界除非阻断主流程，否则记录为 P3 或相邻线索。
6. 当前入口 P0/P1 和影响主流程的 P2 未修复并验证前，不切换下一个入口。
7. 验证必须覆盖真实页面/弹窗、后端副作用、数据回看、必要测试，以及库存/BOM/出库/成本/审计等关键事实链影响。
8. 旧审计报告、旧计划和历史测试只能作为线索，不能作为完成证明。
9. 每完成一个入口，必须更新 docs/main-flow-entry-list-2026-06-20.md 的状态和对应 docs/main-flow-entry-runs/ 工作记录。
```

## 九、执行口径

本工作流优先级高于旧审计批次和历史计划中的任务惯性。旧文档只能作为线索，不能作为当前入口完成证明。

最终判断标准只有一个: 每个导航入口的高频主流程，用户是否能顺畅完成，并在系统里留下可信、可回看、可审计的结果。
