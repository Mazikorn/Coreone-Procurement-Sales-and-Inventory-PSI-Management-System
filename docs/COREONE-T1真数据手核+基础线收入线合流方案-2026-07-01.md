## 1. T1 真数据手核结论（CONFIRMED）

**问题**：老"制片+诊断"混合码怎么接 BOM？**答案（真数据验证）**：桥接锚点落在**物理工序单位（蜡块/免疫组化片/特染片）**——混合码先按 `36×LIS蜡块/(36×LIS蜡块+105)` 逐病例拆出制片份额接 BOM，诊断份额进诊断桶零 BOM；同一 LIS 计量既当收入拆分量、又当成本动因。

**和睦家 26.2 全月 165 病例（LIS 100% 匹配）**：

| 业务线 | 纯实验室收入 | 物理单位 | G2 成本[低~中~高] | 毛利率(中) | 信号 |
|---|---:|---:|---:|---:|---|
| 组织学制片 | ¥13,079 | 260 蜡块 | ¥7,800~10,400~13,000 | **20%** | 人力主导、薄，高成本档≈盈亏平衡 |
| 免疫组化(染色) | ¥11,648 | 80 抗体 | ¥1,440~2,640~3,840 | **77%** | 利润中心 |
| 细胞 TCT | ¥3,106 | 71 玻片 | ¥1,988~2,840~3,692 | 9% | 薄 |
| 院内冰冻 | ¥37 | 1 例 | ¥40~65~90 | 负 | 1 例、LIS 未充分记，噪声 |
| **合计** | **¥27,870** | — | ¥11,288~**15,985**~20,682 | **43%** | 毛利 ¥7,188~**11,885**~16,582 |

**逐例（尖锐信号）**：`S26-00472`（18 块纯 HE、无免疫组化）收入 ¥647 / 成本 ¥540~900 → **可能亏本**；`S26-00460`（1 块 HE-only）≈盈亏平衡；`S26-00783`（2 块 + 14 免疫组化）毛利率 58~83%。**这正是产品要给的洞察**——纯常规组织学薄利甚至亏、免疫组化撑利润。

**独立复核 verdict = CONFIRMED**：从原始脱敏数据独立重算，¥27,870 / 诊断桶 ¥27,671 / 守恒 ¥55,541 逐项吻合；**无双计、无符号错、join 正确**（物理单位正确限定在 165 张已结算病例，未把全 LIS 1005 行/1567 蜡块计成本）。

**4 条诚实 caveat（复核提出，写码前须知）**：
1. **中值成本靠 band 中点假设**（免疫组化 ¥33=原液18/即用48 中点）；真硬边界是 band 上下界 ¥11,288~20,682。精度瓶颈=原液/即用混用比例未知（同 G2 承重墙）。
2. **染色 77% 毛利是"口径不对称"**：染色收入是整条结算净额（含诊断解读价值），而组织侧做了制片/诊断拆分。两条线口径不对称，横向比毛利率要小心 → **引出关键开放问题：诊断桶是否零成本（§4）**。
3. 840 例 LIS-only 未结算、未计成本（本月口径正确，但若跨月/他渠道结算则真实产能成本被低估）——数据完整度问题，非算错。
4. 冰冻 1 例 / 特染 2 次，样本太小，不能据此单线定价。

## 3. 合流方案（PARTIALLY_CONFIRMED，已并入 5 处必改）

### 3.1 目标：加"目录层"，不合表
四套词汇各自绑着生产引擎（projects 绑 ABC、charge_codes 绑收费引擎），直接收敛成一套=改主键=撞两个引擎红线（DatabaseManager 注释明确 fee_standards↔charge_codes "完全独立互不读写"）。故**加一层映射，不合表**（adoption-first：粗放起步先映高频，增量精细）：

| 新表(全 `CREATE TABLE IF NOT EXISTS` + ensureColumn 兼容旧库) | 作用 |
|---|---|
| `catalog_items` | 统一检测项目主目录（一个"检测项目概念"一行，带 business_line） |
| `catalog_process_units` | **物理工序单位定义**（蜡块/免疫组化片/特染片/诊断桶 + 每例数量 + 零BOM标记）=成本↔收入唯一硬锚 |
| `catalog_mappings` | 四套词汇 → 目录项的软/硬映射（match_type=exact_code/keyword/prefix/manual + confidence + 留痕） |
| `mixed_code_split_rules` | 老混合码拆分规则（承载 36×蜡块/(36×蜡块+105) 制片份额/诊断桶） |

物理工序单位贯穿：LIS 已有蜡块/片计量 → 经 `catalog_process_units` 同时喂 (a) 收入侧拆分 (b) 成本侧 ABC 动因。**已被 T1 手核验证可行（165 例、逐线毛利闭合、无双计）。**

### 3.2 策略：master 为主干，手工移植 codex 基础功能
表 diff 实证——两线互不为超集：codex 独有 7 表（多库位库存 5 + login_attempts + audit_log_archives）；master 独有 11 表（收入侧 8 + idempotency_keys/user_roles/app_settings）。**选 master 为主干**（收入侧+ABC+跨院串账防护+幂等键更贵重），把 codex 的基础功能表当增量搬进去。无共同历史 → **一模块一 PR 手工移植 + 每步 e2e 守零回归 + 黄金复校**。

### 3.3 合流顺序（复核已修正：phase2 提前为硬前置）

| 步 | 动作 | 守护 |

### 3.4 复核挑出的 5 处必改（已并入上表）
1. **split/混合码拆分是 phase2 独有，不是 master 现状** → phase2 从"可选尾步"升为**步骤 3 硬前置**（否则步骤 5 落空）。
2. 步骤 5 回归须**同时纳入 phase2 的 `hemujia-purelab-golden.test.ts`（¥27,870）**，不只 `partner-revenue-golden`（¥13,152）→ **双黄金**。
3. `scope='split'/splitWorkload='lis_blk'` 是 **partner-config.ts 的 JSON 配置字段**，不是 `partner_configs` 表列（盘点按"表列"找会漏）。
4. lis_cases 唯一键是**匿名 `UNIQUE(partner_id, case_no, service_month)`**（含 service_month），不是具名 `uq_lis_cases_partner_case`（grep 找不到）。
5. `projects` 表 **codex 与 master 两侧都有**（共享基础表），不是 codex 独有；表 diff 以 11/7 精确清单为准。

## 4. 需 PM 裁决的开放口径（阻塞实现）

| 开放问题 | 为什么重要 | 怎么定 |
|---|---|---|
| **catalog_items 粒度**：ER/PR/HER2 是 3 个目录项，还是 1 个免疫组化项 + 抗体维度？ | 决定映射维护量与成本动因精度 | PM+病理业务；建议按对账单/收费习惯的项目粒度起步，别过细 |
| **诊断桶零成本 vs 回填诊断成本** | 直接影响免疫组化 77% 毛利是否高估（染色收入含诊断解读价值，成本侧却没算诊断） | 真数据 + PM；接 ABC 间接成本中心 or 认定零成本 |
| **36/105 通用 vs 逐院/逐码** | 若逐院不同，`mixed_code_split_rules` 需加 partner_id 维度 | 真数据（多院混合码样本）手核 |
| **映射置信门禁**：低置信(关键词/前缀)映射能否进正式毛利报表？ | 报表可信度 | PM；建议先展示但标注置信度（adoption-first） |
| **国标 81 码是否入库为字典** | 现状系统内无国标对照表；涉外部数据入库合规 | PM+数据合规 |
| **phase2 本轮是否并入** | 决定是否含步骤 3（我方案已判定：接 T1 桥接必须并） | 已倾向"必并"，待 PM 确认优先级 |
