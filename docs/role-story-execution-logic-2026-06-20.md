# 用户角色故事执行逻辑

> 创建日期: 2026-06-20
> 配套文档: `docs/role-story-goal-workflow-2026-06-20.md`
> 故事队列: `docs/role-story-list-2026-06-20.md`

## 一、执行模式

这是一个顺序执行工作流:

1. 从故事清单选择当前故事。
2. 锁定当前角色和故事边界。
3. 用该角色视角检索菜单、页面、API、权限、数据和下游交接链路。
4. 现场建立并维护问题清单。
5. 修复当前故事的高频主流程、权限一致性、交接和事实链问题。
6. 验证真实页面、角色权限、后端副作用、数据回看和审计证据。
7. 收口当前故事。
8. 自动切换到下一个故事。

执行者不得在当前故事未收口时跳到另一个故事做修复。相邻故事问题只能记录为线索。

## 二、启动步骤

每次目标模式启动或恢复时，先执行:

```bash
git status --short --branch
git log --oneline --decorate -n 40
```

然后读取:

1. `AGENTS.md`
2. `docs/role-story-goal-workflow-2026-06-20.md`
3. `docs/role-story-execution-logic-2026-06-20.md`
4. `docs/role-story-list-2026-06-20.md`

如果本故事需要 Playwright 或真实浏览器页面验证，必须先按 `AGENTS.md` 的 Playwright 强规则读取指定记忆文件，并确认本地缓存或备份恢复路径，禁止默认下载。

如果存在未提交改动，先判断是否属于当前故事或历史遗留。无关改动不得回退、覆盖或顺手整理。

## 三、故事选择算法

按以下顺序选择当前故事:

1. 如果故事清单中有 `in_progress`，继续这个故事。
2. 否则选择排序最靠前的 `pending`。
3. 如果某故事状态是 `blocked`，只有阻塞条件已消失才恢复，否则跳过到下一个 `pending`。
4. 如果所有故事均为 `completed`，目标完成。

选择故事后，必须在故事清单中把状态更新为 `in_progress`，并创建或继续对应工作记录。

## 五、全面检索要求

围绕当前角色故事，从用户目标反向检索完整链路。至少覆盖:

1. 该角色的菜单可见性、路由守卫、默认首页和不可访问页面处理。
2. 该故事涉及的前端页面、弹窗、组件、hook、状态管理和 API 调用。
3. 后端路由、端点级权限、事务、副作用、错误处理和审计日志。
4. 数据表、关键字段、状态枚举、引用关系和下游可回看位置。
5. 已有单测、集成测试、页面测试、E2E 记录。
6. `docs/05_Role_Permission_Matrix.md`、`docs/02_PRD.md`、FRS、上一轮入口记录中与故事有关的线索。
7. 该故事对库存、BOM、出库、成本、预警或审计事实链的影响。

建议命令:

```bash
rg -n "<角色代码|页面路径|接口名|表名|业务关键词>" 前端代码/src 后端代码/server/src docs V1.1设计稿
rg -n "ROLE_MENU_MAP|requireRole|authenticateToken|operation_logs" 前端代码/src 后端代码/server/src
```

检索完成后，先把故事链路写入工作记录，再开始改代码。

## 六、用户故事评估

每个故事必须先完成一轮用户故事评估，并写入工作记录。

必须质疑的问题包括:

1. 角色是否应该执行这个故事，是否存在职责错位或权限过宽。
2. 用户要完成的目标是否被系统完整覆盖，是否仍要线下补流程。
3. 表单、列表、详情、筛选和提示是否提供该角色完成判断所需信息。
4. 上游角色生成的数据是否可被当前角色识别、筛选、引用和解释。
5. 当前角色操作后，下游角色能否继续处理并理解状态。
6. 前端菜单、路由守卫、后端 API 权限和审计日志是否一致。
7. 故事结果是否能降低库存、BOM、出库、成本或审计的后续修正成本。

建模问题不是建议项。如果它会导致角色职责错位、事实链错误、成本分析失真、用户长期手工修正，必须进入现场问题清单。

## 七、问题分级

| 优先级 | 判定标准 | 当前故事处理要求 |
|:---:|:---|:---|
| P0 | 当前角色无法完成核心故事，或会写错库存、成本、审计关键事实 | 必须修复并验证 |
| P1 | 故事能完成但结果不可信、不可回看、权限错位、下游无法交接，或建模导致事实链/成本分析失真 | 必须修复并验证 |
| P2 | 高频体验、提示、效率或信息缺口，会增加日常操作成本 | 影响主流程或交接则修复，否则说明暂缓 |
| P3 | 低频边界、细节优化、与当前故事关系弱 | 默认记录，不展开 |

旧审计报告、上一轮入口记录和历史测试只能作为线索，不能自动成为本故事的完成证明。

## 八、修复顺序

按用户故事闭环修复，而不是按文件类型修复:

1. 先确认角色职责、权限边界和可见菜单是否正确。
2. 再确认用户目标是否完整，字段和状态是否足够支撑判断。
3. 再修复当前角色的主动作。
4. 再保证后端副作用正确。
5. 再保证上游输入和下游交接可回看、可解释。
6. 再补齐错误提示、禁用状态、确认弹窗和审计日志。
7. 最后补测试和回归。

涉及 bug 修复或新能力时，优先补能证明业务结果和角色边界的测试。测试不要只断言按钮存在，要断言真实副作用、权限拒绝或数据回看。

## 九、验证要求

每个故事至少选择与风险匹配的验证组合:

- 对应角色真实登录后的菜单、路由、页面和弹窗验证。
- 前端可见权限与后端 API 权限一致性验证。
- API 请求和响应验证。
- 数据库或后端状态验证。
- 列表、详情、报表、预警、日志或下游故事回看。
- 单测、集成测试、构建或针对性回归。
- 如果影响库存、BOM、出库、成本或 ABC 输入，补相关回归。
- 如果使用 Playwright，必须证明使用的是本地缓存或本地备份恢复后的浏览器，且未下载。

不能只因为上一轮菜单入口完成就判定故事完成；也不能只因为测试通过就跳过真实角色页面和交接链路。

## 十、自动切换规则

当前故事满足完成标准后:

1. 更新工作记录的收口报告。
2. 更新故事清单中该故事状态为 `completed`。
3. 填写完成日期、工作记录路径、验证摘要。
4. 重新读取故事清单。
5. 选择下一个 `pending` 故事继续执行。

如果上下文或时间不足，必须至少确保故事清单和工作记录能让下一次恢复时继续，不要只在最终回复中口头说明。

## 十一、阻塞规则

只有以下情况可以把故事标记为 `blocked`:

1. 缺少必须由用户决定的角色职责或业务口径，且无法做保守假设。
2. 当前故事需要外部服务、账号、真实业务数据或权限，且本地无法替代验证。
3. 继续执行会覆盖用户未确认的重要改动。

标记 `blocked` 时必须写明:

- 阻塞原因。
- 已完成的检查。
- 需要用户决定的问题。
- 暂停前的当前证据。

阻塞解除后，从同一个故事继续，不重新开新故事。
