## 二、Critical 问题（必须在开发前解决）

### 问题 C-1: bom_equipment_templates 迁移缺少数据完整性保障

**来源**: 后端视角  
**描述**: 废弃分析 §二 的迁移方案中，Step 2 通过 `JOIN equipment e ON bet.equipment_id = e.id WHERE e.type_id IS NOT NULL` 迁移数据。但未说明：当 `equipment.type_id` 为 NULL 时，这些 BOM 模板记录如何处理。  
**影响**: 迁移后部分 BOM 可能丢失设备模板，导致成本计算返回 0。  
**建议**: 
1. 迁移前必须运行数据审计脚本，统计无 type_id 的设备数量
2. 提供人工映射工具（管理员为无类型设备指定类型）
3. 迁移脚本增加事务保护，失败时回滚

**解决状态**: ✅ 已修订（废弃分析 v2+）
- 新增 Step 0 数据审计：SQL 统计无 type_id 设备数量及关联 BOM 模板数量
- 新增 Step 2b 人工映射：migration-helper API + 前端临时页面
- 整个迁移包裹在 `BEGIN IMMEDIATE / COMMIT / ROLLBACK` 事务中
- 新增 Step 1 备份步骤

## 三、High 问题（建议在开发前解决）

### 问题 H-1: 设备类型默认参数与实际设备参数不一致

**来源**: 产品视角  
**描述**: `equipment_types` 表存储默认折旧参数（default_purchase_price 等），用于 BOM 成本预览。但实际设备的参数可能与默认值差异很大（如同类型设备价格从 3 万到 50 万不等）。  
**影响**: 成本预览值与实际出库成本偏差大，用户信任度降低。  
**建议**: 
1. 成本预览增加两种模式：使用类型默认值 / 使用该类型下所有设备的加权平均值
2. 在预览结果中标注数据来源（"基于类型默认参数" vs "基于该类型下 N 台设备的平均值"）

**解决状态**: ✅ 已修订（废弃分析 v3 + 详细实施计划 Step 3.3）
- 成本预览 API 增加 `costMode` 查询参数：`type_default` / `equipment_average`（默认）
- `equipment_average` 模式使用该类型下所有设备的加权平均参数（按 total_capacity 权重）
- 响应新增 `priceSource` 字段（type_default / equipment_average / actual）
- 响应新增 `equipmentCount` 和 `note` 字段说明数据来源

### 问题 H-2: reports-v1.1.ts 内联代码重复未纳入迁移范围

**来源**: 后端视角  
**描述**: 废弃分析 §三 指出 `reports-v1.1.ts:310-324` 有内联设备成本计算，但代码更新计划的 Phase 1 步骤中未包含此项重构。  
**影响**: 设备类型迁移后，`/full-cost-by-project` 端点的设备成本计算会失败。  
**建议**: 在 Phase 1 Step 1.3 中明确包含此项重构，或在迁移脚本执行前先完成重构。

**解决状态**: ✅ 已修订（详细实施计划 Step 1.3）
- Step 1.3 新增第 6 项：重构 reports-v1.1.ts:310-324 内联设备成本计算为调用 `calculateEquipmentCost`
- 明确执行顺序约束：重构必须在 Step 1.1 数据库迁移之前完成
- 新增验证条件：设备成本计算结果不变（回归测试）+ 无内联代码残留

### 问题 H-3: 季度调整审核缺少权限矩阵

**来源**: 后端视角  
**描述**: VibeContract 提到"审核人 ≠ 调整人"，但未定义完整的权限矩阵。哪些角色可以调整？哪些可以审核？审核是否需要多人？  
**影响**: 实现时权限逻辑不清晰。  
**建议**: 补充权限矩阵：
| 操作 | admin | finance | 其他 |
|------|:-----:|:-------:|:----:|
| 生成调整建议 | ✅ | ✅ | ❌ |
| 手动创建调整 | ✅ | ✅ | ❌ |
| 审核调整 | ✅ | ✅（不能审自己） | ❌ |

**解决状态**: ✅ 已修订（废弃分析 §六 + 详细实施计划 Phase 4）
- 废弃分析新增 §六：完整权限矩阵（5 个角色 x 5 个操作）+ 自我审核约束实现逻辑
- 详细实施计划 Phase 4 VibeContract：新增权限矩阵表 + 自我审核约束说明
- Step 4.2：明确路由注册使用 `requireRole('admin', 'finance')` + 审核时检查 `submitted_by !== req.user.id`
- 异常契约新增权限不足场景（403）

### 问题 H-4: 成本趋势季度聚合的年份边界

**来源**: 测试视角  
**描述**: 对抗性提示中的"跨年季度"测试用例未覆盖：查询时间范围跨越年份（如 2025-10 到 2026-03）时，Q4 和 Q1 是否正确分组。  
**影响**: 年末/年初的季度数据可能被错误聚合。  
**建议**: 补充测试用例：
```typescript
it('跨年时间范围的季度聚合应正确分组', async () => {
  const trend = await getCostTrend({
    dimension: 'quarterly',
    startDate: '2025-10',
    endDate: '2026-03'
  })
  expect(trend.map(t => t.period)).toEqual(['2025-Q4', '2026-Q1'])
})
```

**解决状态**: ✅ 已修订（废弃分析 §七 + 详细实施计划 Phase 3）
- 废弃分析新增 §七：季度计算 SQL 公式（CASE WHEN month BETWEEN ...）+ 跨年边界验证说明
- 详细实施计划 Phase 3 对抗性提示：新增"边界 3: 跨年季度聚合"完整测试用例（含数据断言）
- Step 3.1：新增季度计算公式 SQL + 验证条件（12 月归入 Q4，1 月归入 Q1）
- 两个测试用例覆盖：跨 6 个月范围 + 单月跨年边界

### 问题 H-5: 设备折旧统计页复用 EquipmentEfficiency 路由

**来源**: 前端视角  
**描述**: 废弃分析建议将 `/abc/equipment-efficiency` 路由复用于折旧统计页。但这两个功能的用户角色不同（设备效率是运营视角，折旧统计是财务视角），复用同一路由可能导致权限混乱。  
**影响**: 财务人员可能看到不需要的运营指标，或被错误的权限控制。  
**建议**: 使用独立路由 `/equipment/depreciation-stats`，保持 `/abc/equipment-efficiency` 作为未来设备效率分析的预留。

**解决状态**: ✅ 已修订（废弃分析 F1 + 详细实施计划 Step 1.6）
- 废弃分析 F1：改为"保留路由 + 新建页面"，保留 `/abc/equipment-efficiency` 预留未来
- 详细实施计划 Step 1.6：新建 `DepreciationStats.tsx`，路由 `/equipment/depreciation-stats`
- 新增路由对照表：`/equipment/depreciation-stats`（财务视角，admin/finance）vs `/abc/equipment-efficiency`（运营视角，admin/warehouse_manager）
- 更新 barrel export（cost/index.ts）

### 问题 H-6: 成本预览缓存策略未定义

**来源**: 后端视角  
**描述**: VibeContract 提到"成本预览 API 响应时间 > 2s 时返回缓存结果"，但未定义缓存策略（TTL、失效条件、存储位置）。  
**影响**: 实现时缓存逻辑不清晰，可能导致数据过期。  
**建议**: 
1. 初期不缓存（实时计算），监控响应时间
2. 如确实需要缓存，使用内存缓存（Map），TTL = 5 分钟
3. BOM 更新时自动失效缓存

**解决状态**: ✅ 已修订（废弃分析 §五 + 详细实施计划 Step 3.3）
- 废弃分析新增 §五：完整缓存策略定义（初期不缓存 → 优化期 Map 缓存 TTL=300s）
- 缓存实现代码：getCachedCostPreview / setCachedCostPreview / invalidateBomCostCache
- 缓存失效条件：BOM 物料/设备/工时变更、间接成本中心变更、设备类型参数变更
- 详细实施计划 Step 3.3：缓存策略说明 + 验证条件
- 监控指标：首次计算响应时间、缓存命中率、内存占用

## 四、Medium 问题（可在开发中解决）

### 问题 M-1: 产品 — 月度环比"数据不完整"的展示方式

**描述**: 当月数据不完整时，如何在 UI 上标注？  
**建议**: 使用虚线边框卡片 + tooltip 说明"截至 X 月 X 日的数据"。

### 问题 M-2: 产品 — 设备类型种子数据的来源

**描述**: 计划提到"预置 5 个设备类型"，但未列出具体类型。  
**建议**: 基于病理实验室常见设备：切片机、染色机、数字扫描仪、PCR 仪、其他。

### 问题 M-3: 产品 — 参考值来源标签的分类标准

**描述**: reference_source 的可选值（supplier/industry/system）定义模糊。  
**建议**: 明确定义：
- `supplier` — 设备/试剂供应商提供的推荐参数
- `industry` — 行业标准或文献参考值
- `system` — 系统预设或历史数据推算

### 问题 M-4: 设计 — 成本结构饼图的交互

**描述**: 按活动中心和按成本类型的切换交互方式未定义。  
**建议**: 使用 Toggle 按钮组（两个选项），切换时饼图平滑过渡动画。

### 问题 M-5: 设计 — 季度调整页面的信息层级

**描述**: 调整建议表格 + 调整记录表格的布局和信息层级。  
**建议**: 上方为当前季度调整建议（可操作），下方为历史调整记录（只读），用 Tab 切换。

### 问题 M-6: 前端 — BOMFormModal 设备模板 Tab 的表单验证

**描述**: 新增"设备模板"Tab 后，设备类型选择 + 使用时长的验证规则。  
**建议**: 设备类型必填，使用时长 > 0 且 ≤ 480 分钟（8 小时）。

### 问题 M-7: 前端 — 折旧统计图表的数据刷新

**描述**: 筛选条件变化后图表如何刷新？  
**建议**: 使用 React Query，筛选变化时自动 refetch，显示 loading 骨架屏。

### 问题 M-8: 前端 — 成本预览弹窗的加载状态

**描述**: 成本预览计算可能需要 1-2 秒，加载状态如何展示？  
**建议**: 弹窗内显示 skeleton 加载动画，汇总卡片区域先显示。

### 问题 M-9: 后端 — equipment_types 表的描述字段长度

**描述**: VibeContract 定义 description ≤ 200 字符，但未说明超长截断策略。  
**建议**: 后端 express-validator 校验长度，返回 400 错误而非静默截断。

### 问题 M-10: 后端 — 成本差异分析的时间范围限制

**描述**: VibeContract 提到"时间范围不超过 12 个月"，但未说明是否包含当月。  
**建议**: 包含当月，即 startDate 到 endDate 最多 13 个月数据。

### 问题 M-11: 测试 — 季度调整并发审核的乐观锁实现

**描述**: 对抗性提示要求并发审核只有一个成功，但未指定乐观锁实现方式。  
**建议**: 使用 `review_status` + `reviewed_at` 作为乐观锁条件：
```sql
UPDATE cost_adjustments SET review_status = ?, reviewed_by = ?, reviewed_at = CURRENT_TIMESTAMP
WHERE id = ? AND review_status = 'pending' AND reviewed_at IS NULL
```
检查 `changes === 0` 则表示已被他人审核。

### 问题 M-12: 测试 — 设备类型删除的级联策略

**描述**: 删除设备类型时，关联设备的 type_id 如何处理？  
**建议**: 两种策略（管理员选择）：
1. 阻止删除（默认）
2. 将关联设备的 type_id 设为 NULL
