## 2. 方案一：渐进式改造

### 2.1 核心思路

在不改动现有出库流程的前提下，通过"后处理分摊 + 成本看板"模式，将已有的 ABC 基础设施从业务死代码激活为可用的成本分析补充层。

### 2.2 数据流

```
配置层：作业中心 → 成本动因 → 成本池 → BOM作业关联

触发层：手动/定时触发 → 遍历出库记录 → 调用 calculateSlideCost() → 写入 case_cost_records

分析层：成本看板（汇总卡片 + 瀑布图 + 饼图 + 排行榜）
```

### 2.3 核心页面

1. **ABC 成本看板** (`/abc/dashboard`) — 一屏掌握"花了多少、收了多少、赚了多少"
2. **病例成本明细** (`/abc/case-detail`) — 下钻到单张切片级别
3. **BOM 作业配置** (`/bom` 增加 Tab) — 配置 BOM 与作业中心关联
4. **月度成本池管理** (增强现有) — 从间接成本中心导入
5. **传统 vs ABC 对比** (`/abc/comparison`) — 双模型对比

### 2.4 API 设计

|------|------|------|
| POST | `/api/v1/abc/calculate` | 执行月度 ABC 分摊计算 |

| GET | `/api/v1/abc/case-costs` | 病例成本明细列表 |
| GET | `/api/v1/abc/comparison` | 传统 vs ABC 对比数据 |
| GET | `/api/v1/abc/profitability` | 盈利性分析 |

### 2.5 优势

- **零侵入**：不修改现有出库流程，风险最低
- **复用率高**：约 70% 的 ABC 基础设施可直接使用
- **渐进式**：可分阶段交付，每阶段独立可用
- **双模型并存**：传统成本模型与 ABC 模型可同时运行，便于对比验证

- **非实时**：需要手动或定时触发计算，无法在出库时即时获得成本
- **数据延迟**：成本数据存在滞后性，无法支持实时决策
- **无出库时校验**：出库时无法校验成本合理性

## 3. 方案二：深度集成

### 3.1 核心思路

将 ABC 作业成本计算引擎嵌入出库事务流，以"单张切片成本"为核心指标替代传统五维成本模型，实现出库即算成本、成本即出报表的端到端闭环。

### 3.2 数据流

```
配置层：
  间接成本中心 → 成本池（月度同步）
  → 作业中心 → 成本动因
  → BOM作业关联

计算层：
  出库请求 → BOM查找
  → 材料成本 + abc_bom_activity_links
  → 逐个作业中心计算
  → totalCost = materialCost + SUM(activityCosts)
  → 写入 outbound_records.total_cost + case_cost_records

分析层：
  项目维度汇总 → 切片维度下钻 → 物料维度下钻 → 盈利性分析
```

### 3.3 核心页面

1. **单张切片成本看板** (`/abc/slide-cost`) — 核心页面，按项目分组的单张切片成本
2. **成本池管理** (改造现有) — 从间接成本中心同步 + 自动归集
3. **BOM 作业配置** (改造 BOM 详情页) — 新增"作业关联" Tab
4. **盈利性分析** (改造现有) — 成本 vs 收费对比
5. **出库成本预览** (嵌入出库弹窗) — 出库前预览全成本

### 3.4 API 设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/outbound/bom` (改造) | 响应增加 `abcCostBreakdown` |

| GET | `/api/v1/abc/outbound-preview` | 出库成本预览 |

### 3.5 优势

- **成本精确度最高**：单张切片级别精确计算，逐作业中心归集
- **出库即算成本**：实时计算，无数据延迟
- **复用现有基础设施**：约 85% 的 ABC 基础设施可直接使用
- **下钻能力强**：支持项目 → 切片 → 物料多维度下钻
- **与 BOM 深度绑定**：成本计算与 BOM 结构紧密关联，数据一致性高

### 3.6 劣势

- **配置前置工作量大**：需要完整的作业中心、成本动因、BOM 作业关联配置
- **成本池数据质量依赖**：计算结果依赖成本池数据的准确性和完整性
- **替代传统模型有风险**：全面替代需要充分验证，存在业务连续性风险
- **性能考量**：出库时实时计算可能影响出库响应速度
- **学习成本**：ABC 概念较复杂，需要培训用户理解成本结构

## 4. 方案三：收费驱动

### 4.1 核心思路

### 4.2 数据流

```

  fee_standards + projects → project_fee_mappings

运行阶段：
  出库时自动触发

  → 全成本计算(现有) + 收费金额计算(新增)
  → profitability_reports

分析阶段：
  前端盈利性分析页面
```

### 4.3 核心页面

1. **盈利性分析仪表盘** (`/abc/profitability`) — 哪些项目赚钱哪些亏损
2. **项目盈利明细** (`/abc/profitability/:projectId`) — 收入构成 vs 成本构成
3. **收费标准管理** (`/abc/fee-standards`) — 查看和管理收费标准
4. **项目收费映射配置** (`/abc/fee-mapping`) — 将检测项目与收费标准关联
5. **收费驱动的成本分析** (增强 CostAnalysis) — 叠加收费对比维度

### 4.4 数据库变更

| 新建视图 | `v_profitability_summary` | 盈利性汇总视图 |

### 4.5 优势

- **低侵入性**：不修改现有成本计算流程，仅叠加收费维度
- **数据资产激活**：激活已导入的 82 条收费标准数据
- **业务语义清晰**：以收费为锚点，业务人员易理解
- **阶梯定价完整支持**：支持复杂收费规则（如阶梯定价、组合收费）
- **增量可交付**：可分阶段交付，每阶段独立可用

- **诊断费聚合难题**：部分诊断费用难以精确归属到单个检测项目

## 6. 最终选择：方案二（深度集成）

### 6.1 选择理由

1. **成本精确度最高**：单张切片级别精确计算，满足病理实验室精细化管理需求
2. **端到端闭环**：出库即算成本，数据无延迟，支持实时决策
3. **基础设施复用率高**：约 85% 的 ABC 基础设施可直接使用，减少重复开发
4. **下钻能力强**：支持多维度下钻分析，满足管理层深度分析需求
5. **与 BOM 深度绑定**：成本计算与 BOM 结构紧密关联，数据一致性强

### 6.2 实施策略

1. **双模型并行期**：新旧成本模型并行运行 1-2 个月，对比验证准确性
2. **分阶段交付**：优先实现核心计算逻辑，再逐步完善看板和分析功能
3. **用户培训**：提前进行 ABC 概念培训，降低学习成本
4. **性能优化**：对出库时的成本计算进行性能优化，确保响应速度

### 6.3 后续演进

- 短期（1-3 个月）：完成方案二核心功能，双模型并行验证
- 中期（3-6 个月）：根据验证结果，逐步停用传统成本模型
