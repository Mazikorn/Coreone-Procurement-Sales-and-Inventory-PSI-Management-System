# 2026-06-26 — DatabaseManager 冗余 is_deleted 迁移块清理

## 背景（报告的缺陷）
报告称：`purchase_orders.is_deleted` 的内联 ALTER 迁移块运行在该表 `CREATE TABLE` **之前**，
全新库上 `PRAGMA table_info(purchase_orders)` 返回空 → `ALTER` 抛 `no such table` 被 try/catch 吞掉，
导致全新/内存库 `purchase_orders` 缺 `is_deleted` 列；任何 `... AND po.is_deleted = 0` 的 JOIN 运行期
报 `no such column: po.is_deleted`（HTTP 500，经 supplier-returns GET 端点在内存测试库上观察到）。

## 实跑核查（先验证再动手）
用 `DATABASE_PATH=':memory:'` 跑真实 `initializeDatabase()` 后 `PRAGMA table_info`：
- **当前代码已不复现该 500**。原因：
  - `CREATE TABLE purchase_orders`（约 line 365）**已直接包含** `is_deleted INTEGER NOT NULL DEFAULT 0`；
  - 全部 CREATE TABLE 之后有统一幂等兜底 `ensureColumn('purchase_orders','is_deleted',...)`（约 line 662）。
- 真正残留的是**死代码**：line 188–222 共四个内联 `is_deleted` 迁移块
  （purchase_orders / return_records / scrap_records / stocktaking_records），全部位于各自 CREATE TABLE
  之前，在全新库上每次 init 都抛 `no such table` 被吞，且与「CREATE 含该列 + 末尾 ensureColumn」完全重复。
  四表 CREATE 均已含 `is_deleted`（307/321/346/365），ensureColumn 均在 659–676 兜底。

## 改动
`后端代码/server/src/database/DatabaseManager.ts`：删除 line 188–222 的四个冗余/错序/吞错的内联
`is_deleted` 迁移块，替换为一段说明注释（指明列已在 CREATE TABLE + 末尾 ensureColumn 处理，避免日后
再引入错序迁移）。正确顺序的 `inbound_records`（purchase_order_id/no，line 177–186）保持不动。

## 验证（零回归）
- **全新内存库**：四表 `is_deleted` 均 present；`po.is_deleted = 0` 及 supplier-returns 形态 JOIN 全部 OK。
- **旧库路径**：预建无 `is_deleted` 的 legacy `purchase_orders`（含 1 行数据）→ 跑真实 `initializeDatabase()`
  → ensureColumn 幂等补列，旧行 `is_deleted` 默认 0；再次 init 不报错（幂等）。
- **测试**：`returns`(10)/`stocktaking`(21)/`purchase-order-inbound`(23)/`scraps`(21) 单文件隔离全绿。
  - `scraps.test.ts` SC-004 在「多文件同跑」时偶发 404≠400 → 经对照（stash 原文件）确认是**既有跨文件
    SQLite 共享状态 flakiness**，与本改动无关（单文件跑原/改两版均 21/21 绿）。
  - `supplier-returns.test.ts` / `purchase-orders.test.ts` 被 `vitest.config.ts` exclude（既有配置，无法本地跑）；
    但已用内存库直接复跑其 JOIN 形态，证实通过。
