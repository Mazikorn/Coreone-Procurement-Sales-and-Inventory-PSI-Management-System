# COREONE E2E 测试路径矩阵

## 核心理念："然后呢"三层追问

### 1. 场景端的"然后呢" — 树状扩散延伸场景

每个业务场景都不是孤立的，而是不断追问"然后呢"，形成场景树：

```
技术员出库
├── 项目领用出库
│   ├── 正常出库成功
│   │   └── 然后呢？→ 查看出库记录 → 查看详情 → 验证库存 → 验证成本
│   ├── 出库失败（库存不足）
│   │   └── 然后呢？→ 提示用户 → 等待补货 → 重新出库
│   ├── 出库失败（网络错误）
│   │   └── 然后呢？→ 重试 → 成功/失败 → 记录日志
│   └── 出库后发现错误
│       ├── 然后呢？→ 退库 → 库存回退 → 成本调整
│       └── 然后呢？→ 修改出库单 → 重新提交
├── 调拨出库
│   ├── 正常调拨成功
│   │   └── 然后呢？→ 查看调拨记录 → 验证库存变更
│   └── 调拨失败（库存不足）
│       └── 然后呢？→ 提示用户 → 等待补货 → 重新调拨
└── 报废出库
    ├── 正常报废成功
    │   └── 然后呢？→ 查看报废记录 → 验证库存减少
    └── 报废失败（库存不足）
        └── 然后呢？→ 提示用户 → 等待补货 → 重新报废
```

### 2. 操作端的"然后呢" — 不断指向延伸操作

每个用户操作都不是孤立的，而是不断追问"然后呢"，形成操作链：

```
项目领用出库操作链
├── 1. 登录
│   ├── 成功 → 然后呢？→ 进入首页
│   └── 失败 → 然后呢？→ 显示错误 → 重试
├── 2. 进入出库页面
│   ├── 成功 → 然后呢？→ 显示出库列表
│   └── 失败 → 然后呢？→ 显示错误 → 刷新页面
├── 3. 点击"新增出库"按钮
│   ├── 成功 → 然后呢？→ 显示出库表单
│   └── 失败 → 然后呢？→ 按钮不可见 → 检查权限
├── 4. 选择出库类型
│   ├── 项目领用 → 然后呢？→ 选择项目
│   ├── 调拨出库 → 然后呢？→ 选择目标库位
│   └── 报废出库 → 然后呢？→ 填写报废原因
├── 5. 选择项目
│   ├── 成功 → 然后呢？→ 显示项目详情
│   └── 失败 → 然后呢？→ 项目不存在 → 重新选择
├── 6. 选择BOM
│   ├── 成功 → 然后呢？→ 显示BOM详情
│   └── 失败 → 然后呢？→ BOM不存在 → 重新选择
├── 7. 填写出库数量
│   ├── 成功 → 然后呢？→ 验证数量
│   └── 失败 → 然后呢？→ 数量无效 → 重新填写
├── 8. 提交出库
│   ├── 成功 → 然后呢？→ 显示成功提示
│   ├── 失败 → 库存不足 → 然后呢？→ 提示用户
│   └── 失败 → 网络错误 → 然后呢？→ 重试
├── 9. 验证出库成功
│   ├── 查看出库记录 → 然后呢？→ 记录存在
│   ├── 查看详情 → 然后呢？→ 详情正确
│   ├── 查看库存 → 然后呢？→ 库存扣减
│   └── 查看项目成本 → 然后呢？→ 成本归集
└── 10. 后续操作
    ├── 继续出库 → 然后呢？→ 回到步骤3
    ├── 退库 → 然后呢？→ 进入退库流程
    └── 查看报表 → 然后呢？→ 进入报表页面
```

### 3. 测试路径矩阵 — 场景树 × 操作链

基于场景树和操作链，生成一个庞大的测试路径矩阵：

```
测试路径 = 场景节点 × 操作步骤 × 分支条件

示例：
路径1: 技术员出库 → 项目领用出库 → 正常出库成功 → 登录成功 → 进入出库页面成功 → 点击新增出库成功 → 选择项目领用 → 选择项目成功 → 选择BOM成功 → 填写数量成功 → 提交成功 → 验证出库记录 → 验证详情 → 验证库存 → 验证成本

路径2: 技术员出库 → 项目领用出库 → 出库失败（库存不足）→ 登录成功 → 进入出库页面成功 → 点击新增出库成功 → 选择项目领用 → 选择项目成功 → 选择BOM成功 → 填写数量成功 → 提交失败（库存不足）→ 提示用户 → 等待补货

路径3: 技术员出库 → 项目领用出库 → 出库后发现错误 → 登录成功 → 进入出库页面成功 → 点击新增出库成功 → 选择项目领用 → 选择项目成功 → 选择BOM成功 → 填写数量成功 → 提交成功 → 验证出库记录 → 发现错误 → 退库 → 验证库存回退 → 验证成本调整
```

## 测试用例生成规则

### 1. 场景树遍历规则
- 从根节点开始，遍历所有叶子节点
- 每个叶子节点都是一个完整的业务场景
- 每个场景都要追问"然后呢"，直到业务流程结束

### 2. 操作链遍历规则
- 从第一个操作开始，遍历所有操作步骤
- 每个操作步骤都要追问"然后呢"，直到操作链结束
- 每个操作步骤都要考虑成功和失败两种情况

### 3. 测试路径生成规则
- 测试路径 = 场景节点 × 操作步骤 × 分支条件
- 每个测试路径都是一个完整的测试用例
- 每个测试用例都要验证业务结果

### 4. 测试用例命名规则
```
场景-角色-操作-预期结果
```

**示例：**
```
技术员-项目领用出库-正常出库成功
仓管员-采购入库-入库失败（物料不存在）
财务-成本分析-查看项目成本成功
```

## 测试用例执行策略

### 1. 优先级排序
- P0: 核心业务流程（技术员出库、仓管员入库、仓管员盘点）
- P1: 次要业务流程（采购员采购、财务成本分析）
- P2: 边界情况（失败路径、异常路径）

### 2. 执行顺序
- 先执行P0测试用例
- 再执行P1测试用例
- 最后执行P2测试用例

### 3. 失败处理
- P0测试用例失败 → 阻塞发布
- P1测试用例失败 → 需要修复
- P2测试用例失败 → 记录问题

## 总结

通过"然后呢"三层追问，我们生成了一个庞大的测试路径矩阵：

1. **场景端的然后呢**：生成场景树，覆盖所有业务场景
2. **操作端的然后呢**：生成操作链，覆盖所有操作步骤
3. **测试路径矩阵**：场景树 × 操作链 × 分支条件，生成测试路径

每个测试路径都是一个完整的测试用例，每个测试用例都验证业务结果，形成线上（代码路径）和线下（用户操作）的匹配。
